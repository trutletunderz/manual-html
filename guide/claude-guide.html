02 · โมเดล

เลือกโมเดล

กดที่ชื่อโมเดล (เช่น "Sonnet 4.6 ⌄") จะเห็นรายการโมเดลให้เลือก แต่ละตัวเหมาะกับงานคนละแบบ และคิด rate limit ไม่เท่ากัน

โมเดลคำอธิบายในแอปเหมาะกับหมายเหตุ
Fable 5 For your toughest challenges โจทย์ยากที่สุด งานที่ต้องการความสามารถสูงสุด Included until June 22 — โมเดลตระกูล Claude 5 ระดับสูงสุด เปิดให้ใช้ช่วงโปรโมชัน
Opus 4.8 For complex tasks งานซับซ้อน วางแผน/ออกแบบสถาปัตยกรรม การวิเคราะห์เชิงลึก โมเดลทรงพลังสำหรับงานหนัก
Sonnet 4.6 Most efficient for everyday tasks งานประจำวัน เขียนโค้ด implement งาน tool-heavy เลือกอยู่ในรูป ✓ สมดุลคุณภาพ/ความเร็วดีที่สุดสำหรับงานทั่วไป
Haiku 4.5 Fastest for quick answers งานเร็วๆ ปริมาณมาก ตอบสั้น lookup เร็วที่สุด ประหยัด limit

ด้านล่างของเมนูยังมี Effort (เข้าเมนูย่อยตั้งระดับการคิด — หัวข้อ 03) และ More models สำหรับเข้าถึงโมเดลอื่นๆ เพิ่มเติม

กลยุทธ์เลือกโมเดลแบบประหยัดสำหรับงานพัฒนา

Opus 4.8 สำหรับวางแผน/ออกแบบ architecture · Sonnet 4.6 สำหรับ implement งานทั่วไป (default ที่ดี) · Haiku 4.5 สำหรับงานซ้ำๆ ปริมาณมากเช่น lookup spec หรือจัด format · เก็บ Fable 5 / Opus ไว้ใช้กับโจทย์ที่ยากจริงเพื่อไม่ให้เปลือง limit เร็ว

03 · Effort

Effort & Thinking

Effort คุมว่า Claude "คิดหนัก" แค่ไหนก่อนตอบ — ยิ่งสูง ยิ่งละเอียด แต่ช้าลงและใช้ limit เร็วขึ้น ส่วน Thinking เป็นสวิตช์แยกต่างหาก

หลักการ (ตามคำอธิบายในแอป)

"Effort ที่สูงขึ้น = คำตอบที่ละเอียดรอบคอบกว่า แต่ใช้เวลานานกว่าและกิน limit เร็วขึ้น" — เป็นการแลกระหว่าง คุณภาพ ↔ ความเร็ว/ค่าใช้จ่าย

ระดับคิดมากแค่ไหนเหมาะกับ
Lowคิดน้อยที่สุด ตอบเร็ว อิงความรู้/pattern เป็นหลักงานง่าย ปริมาณมาก เน้นประหยัด limit · มีป้าย Default
Mediumคิดพอประมาณ สมดุล (เลือกอยู่ในรูป ✓)งานประจำวันส่วนใหญ่ เขียนโค้ดทั่วไป ยืดการใช้ limit
Highคิดเยอะ จัดสรร budget การคิดมากขึ้นสมดุลคุณภาพ/ความเร็วโดยรวมดีที่สุด · งาน multi-step reasoning
Maxคิดลึกที่สุด ใช้เวลานานสุด กิน limit เร็วสุดโจทย์ยากจริง งานยาวที่รันต่อเนื่อง · อาจต้องแพลนสูงกว่า

Thinking (สวิตช์แยก)

Thinking ในเมนูเป็นสวิตช์เปิด/ปิดแยกจาก Effort เมื่อเปิด Claude จะใช้เวลาแตกปัญหา วางแผน และสำรวจหลายแนวทางก่อนตอบ และ แสดงกระบวนการคิดให้เห็น เข้ากันดีกับ Effort ระดับ High/Max สำหรับงานที่อยากเห็นว่า Claude คิดมายังไง — ใช้ร่วมกับ Effort ระดับใดก็ได้ ทั้งสองเป็นการตั้งค่าอิสระต่อกัน

จับคู่ Effort กับงานพัฒนา

สรุป action items / lookup → Low–Medium · เขียน/แก้โค้ดทั่วไป → Medium–High · refactor module ใหญ่, แก้บั๊กที่ต้องไล่หลายชั้น, ออกแบบ architecture → High–Max + เปิด Thinking เพื่อเห็น reasoning

06 · Use Style

สไตล์การตอบทั้ง 5 แบบ

แต่ละแบบ "เน้น" คนละอย่าง เลือกให้ตรงกับงานจะได้คำตอบที่ใช้ได้จริง

สไตล์เน้นอะไรโทน & ความยาวเหมาะกับไม่เหมาะกับ
Normal สมดุลทุกด้าน ค่าเริ่มต้น เป็นกันเอง · ปานกลาง งานทั่วไป ถามตอบหลากหลาย งานที่ต้องการ output เฉพาะทาง
Learning สอนแบบ Socratic ถามกลับให้คิดเอง เหมือนติวเตอร์ · ยาว เรียนเทคโนโลยีใหม่ อยากเข้าใจลึก งานเร่ง debug เฉพาะหน้า
Concise สั้น ตรงประเด็น ไม่มีน้ำ ตรงไปตรงมา · สั้นมาก รีบ รู้พื้นฐานแล้ว ขอ snippet งานที่ต้อง context หรือสอนคนใหม่
Explanatory ลึก มีเหตุผล context และ trade-off เหมือน senior review · ยาว ตัดสินใจ architecture เลือก library งานเล็กๆ ถามไวตอบไว
Formal ภาษาทางการ ศัพท์เฉพาะครบ สุภาพเป็นมืออาชีพ · กลาง-ยาว รายงาน อีเมล PR description brainstorm คุยเล่น prototype

Create & edit styles — สร้างสไตล์ของตัวเอง

5 สไตล์มาตรฐานเป็นของกลาง มักไม่ตรงงานเฉพาะทาง 100% ปุ่ม "Create & edit styles" ให้คุณเขียนคำสั่งกำหนด tone, ความยาว, ภาษา และกฎเฉพาะของตัวเองได้ — สำหรับงานพัฒนาจริงนี่คือทางเลือกที่คุ้มที่สุด (ดู template พร้อมใช้ในหัวข้อ 09)

07 · ลองเทียบ

คำถามเดียวกัน 5 สไตล์

เห็นภาพชัดที่สุดคือดูคำตอบของคำถามเดียวกันในแต่ละสไตล์ — กดปุ่มเพื่อสลับดู

คำถามตัวอย่าง
"ใน Vue 3 — ref กับ reactive ต่างกันยังไง?"
เน้น: สมดุลยาว: ปานกลางโค้ด: 1–2 ชุด
Claude · Normal

ref และ reactive ต่างกันที่ชนิดข้อมูลที่รองรับและวิธีเข้าถึง

ref ใช้ได้ทุก type ทั้ง primitive และ object แต่ต้องใช้ .value ใน script ส่วน reactive ใช้ได้เฉพาะ object และไม่ต้อง .value แต่ถ้า destructure จะเสีย reactivity

ส่วนใหญ่แนะนำใช้ ref เพราะ consistent กว่า

เน้น: ถามกลับให้คิดยาว: ยาวโทน: ติวเตอร์
Claude · Learning

ก่อนตอบ ลองคิดดูก่อนนะ: ถ้าคุณมีตัวแปร let count = 0 แล้วเปลี่ยนเป็น count = 1 คุณคิดว่า Vue จะ "รู้" ได้ยังไงว่าค่ามันเปลี่ยน?

JavaScript ปกติไม่มีกลไกบอกว่า "ค่าฉันเปลี่ยนแล้ว" Vue เลยต้อง wrap ค่าไว้ใน object/proxy เพื่อดักจับ — นี่คือเหตุผลที่ ref ต้องมี .value

ทีนี้ลองตอบดู: ถ้า reactive ต้องครอบ object ทั้งก้อน คุณคิดว่าทำไมมันถึงใช้กับ number เดี่ยวๆ ไม่ได้?

กลไกเบื้องหลัง — ref คือ object wrapper รูปแบบ { value: T } ส่วน reactive ใช้ Proxy ครอบ object ต้นทางโดยตรง

ผลที่ตามมา

  1. Type coverage: ref รองรับทุก type, reactive ต้องเป็น object
  2. Access: ref ต้อง .value ใน script (template auto-unwrap), reactive เข้าตรง
  3. Destructure: reactive เสีย reactivity ต้องใช้ toRefs()

คำแนะนำ — ใช้ ref เป็นหลักเพื่อความ consistent โดยเฉพาะใน composables ที่ต้อง return ค่าออกไป ส่วน Pinia store ใช้ reactive ภายในอยู่แล้ว ไม่ต้องกังวล

เน้น: ภาษาเอกสารยาว: กลาง-ยาวemoji: ไม่มี
Claude · Formal

Vue 3 Composition API จัดเตรียมกลไก reactivity สองรูปแบบหลัก ได้แก่ ref และ reactive

ref คือ reactive reference ที่ห่อหุ้มค่าด้วย object wrapper รองรับข้อมูลทุกประเภท การเข้าถึงค่าภายใน script context จำเป็นต้องอ้างอิงผ่าน property .value

reactive คือ reactive proxy ที่สร้างบน object ต้นทาง รองรับเฉพาะข้อมูลประเภท object การ destructure จะส่งผลให้สูญเสียคุณสมบัติ reactivity ซึ่งแก้ไขได้ด้วยฟังก์ชัน toRefs()

ตามแนวปฏิบัติที่แนะนำ ควรเลือกใช้ ref เป็นหลักเพื่อความสม่ำเสมอของโค้ดเบส

Hallucination

อาจ "เดา" field, API, ฟังก์ชัน หรือ citation ที่ไม่มีจริง โดยเฉพาะเมื่อไม่มีข้อมูลอ้างอิง

แก้: แนบ source of truth (Bruno spec), สั่ง "อ่านก่อนตอบ ห้ามเดา", ขอ citation/quote

Knowledge cutoff

ความรู้มีวันตัดรอบ ข้อมูลเวอร์ชันใหม่/ราคา/ข่าวอาจล้าสมัย

มั่นใจแต่ผิด

น้ำเสียงมั่นใจไม่ได้แปลว่าถูก — โค้ดที่ดู "เป๊ะ" อาจมีบั๊กซ่อน

แก้: ตรวจทานทุกครั้ง, ขอให้ flag จุดที่ไม่แน่ใจ, ขอ counterargument

API เก่า / รันไม่ได้

อาจใช้ syntax หรือ API ที่ deprecated แล้ว โดยเฉพาะ framework ที่เปลี่ยนเร็ว

แก้: ลองรันจริง/type-check, ระบุเวอร์ชันที่ใช้ (Vue 3.5, Nuxt 4) ใน prompt

Overengineering

โมเดลใหม่บางครั้งเพิ่มไฟล์/abstraction/ความยืดหยุ่นที่ไม่ได้ขอ

แก้: สั่ง "แก้เฉพาะที่ขอ ไม่เพิ่ม feature/refactor เกิน keep minimal"

เออออตามง่าย

Prefill (เติมคำตอบฝั่ง assistant) ถูกยกเลิกบนโมเดล 4.6+ แล้ว และคำสั่งกดดันแบบ "CRITICAL: MUST" อาจทำให้ overtrigger

แก้: ใช้ instruction ปกติ "Use this tool when…" แทนคำสั่งดุๆ

E · เพิ่มประสิทธิภาพ

จับคู่ model + effort กับงาน ไม่ใช้ของแรงกับงานง่าย
เก็บ context ให้สะอาด /clear ระหว่างงานที่ไม่เกี่ยวกัน

F · ตรวจทานให้ถูกต้อง (ตรวจสอบได้)

ผลลัพธ์จาก AI ต้องตรวจทานเสมอก่อนใช้จริง — เช็คตามรายการนี้

  • โค้ดรันได้จริง — มี import ครบ ผ่าน type-check/lint ลองรันก่อน merge
  • ตรงกับ source of truth — field/endpoint ตรง Bruno spec ไม่มีอะไรที่ Claude เดาขึ้นเอง
  • ไม่ใช้ API deprecated — เช็คว่าตรงกับเวอร์ชันที่โปรเจกต์ใช้จริง
  • ให้ Claude self-check แล้ว — สั่งตรวจคำตอบกับเกณฑ์/test ก่อนปิดงาน
  • ข้อมูล factual ตรวจสอบได้ — ผ่าน web search หรือมี citation ที่เปิดดูได้จริง
  • ผ่าน test / CI — มี test ครอบ logic สำคัญ ไม่ใช่แค่ผ่าน case เดียว
  • review diff ก่อน commit — อ่านทุกบรรทัดที่เปลี่ยน อย่า commit แบบไม่ดู (บทบาท decision-maker)
  • ถามจุดที่ไม่แน่ใจ — "มีตรงไหนที่ไม่มั่นใจหรือ assume ไว้บ้าง" ก่อนนำไปใช้
prompt ตรวจทาน (วางท้ายงานสำคัญ)
ก่อนสรุป ให้ทำสิ่งต่อไปนี้:
1. ตรวจโค้ดว่ามี import ครบและรันได้จริง
   พร้อมระบุวิธีรัน/test
2. ชี้จุดที่คุณ assume หรือไม่แน่ใจ และจุดที่อาจ break
3. ยืนยันว่า field/endpoint ทุกตัวตรงกับสเปกที่ผมให้
   ไม่มีตัวที่เดาเอง
4. ถ้าส่วนใดใช้ API ที่อาจ deprecated ให้บอก
ห้ามรับรองว่า 'ถูกต้องแน่นอน' ถ้ายังไม่ได้ตรวจ

หลักคิดสุดท้าย

AI ช่วยให้คุณทำงานเร็วขึ้น แต่ ความรับผิดชอบต่อความถูกต้องยังเป็นของคุณ ในบทบาท manager/decision-maker — ใช้ Claude สร้าง draft และ explore ทางเลือก แล้วคุณตรวจทาน อนุมัติ และตัดสินใจ นี่คือ workflow ที่ปลอดภัยและได้ผลที่สุด

17 · ฟีเจอร์เชิงลึก

Projects · Memory · Artifacts

สามฟีเจอร์ที่ยกระดับการใช้งานจริง แต่มักถูกมองข้าม — ทำความเข้าใจให้ใช้ได้เต็มที่

Projects — พื้นที่งานที่มี context ร่วมกัน

Project คือพื้นที่รวมหลายแชทที่แชร์ knowledge (ไฟล์/เอกสารที่อัปโหลด) และ custom instructions เดียวกัน ทุกแชทใน project จะเห็นข้อมูลพื้นหลังนั้นทันทีโดยไม่ต้องอธิบายซ้ำ

ส่วนของ Projectใช้ทำอะไร
Project knowledgeอัปโหลดไฟล์อ้างอิงถาวร เช่น CLAUDE.md, coding convention, Bruno spec, design system
Project instructionsกำหนดพฤติกรรม/บทบาท/กฎที่ใช้กับทุกแชทใน project นี้
Chats ภายในแต่ละแชทแยกเรื่องกันได้ แต่ดึง knowledge ชุดเดียวกัน

สำหรับ Pedpro

สร้าง Project "Pedpro" ใส่ CLAUDE.md + convention + Bruno spec ไว้ใน knowledge แล้วทุกแชทเรื่อง Pedpro จะรู้บริบทโปรเจกต์ทันที — เหมาะกับงานที่ต่อเนื่องและต้องยึด context เดียวกัน

Memory — Claude จำข้ามแชทได้

เมื่อเปิด memory ไว้ Claude จะดึงข้อมูลที่เกี่ยวข้องจากแชทก่อนๆ มาทำให้บทสนทนาต่อเนื่องและเป็นส่วนตัวขึ้น โดยไม่ต้องเล่าซ้ำ จุดที่ควรรู้:

  • ควบคุมได้ — สั่งให้ "จำว่า..." หรือ "ลืมเรื่อง..." และแก้ไขสิ่งที่จำได้ในตั้งค่า
  • อัปเดตเป็นช่วง — แชทล่าสุดมากๆ อาจยังไม่ถูกบันทึก และจะอัปเดตเป็นรอบ
  • ปิดได้ใน Incognito — แชทแบบ Incognito จะไม่ใช้และไม่สร้าง memory
  • ค้นแชทเก่าได้ — ขอให้ Claude อ้างอิงสิ่งที่คุยไว้ก่อนหน้า ("ที่เราคุยเรื่อง... เมื่อวาน")

Artifacts — output ที่แก้และใช้ต่อได้

Artifact คือผลงานที่แยกออกมาเป็นหน้าต่างของตัวเอง (เช่นโค้ด เอกสาร เว็บ React) ที่ดู แก้ และนำไปใช้ต่อได้ ต่างจากข้อความตอบทั่วไป — คู่มือที่คุณอ่านอยู่นี้ก็คือ artifact หนึ่ง

เรื่องรายละเอียด
เด้งมาเมื่อไรเมื่อ output เป็นชิ้นงานยาว/นำไปใช้ต่อ เช่น โค้ด เอกสาร รายงาน เว็บเพจ
แก้สดได้สั่งแก้เป็นรอบ "เปลี่ยนสีปุ่มเป็น teal" Claude อัปเดต artifact ให้
React/HTMLสร้าง component หรือ mini app โต้ตอบได้จริงในหน้าต่าง artifact
เก็บ/แชร์ดาวน์โหลดหรือเอาโค้ดไปใช้ต่อในโปรเจกต์จริงได้

  • ระวังให้สิทธิ์ connector เท่าที่จำเป็น (least privilege) โดยเฉพาะ repo/ไฟล์งาน

Claude เดาข้อมูลผิด?

แนบ source of truth, สั่ง "อ่านก่อนตอบ ห้ามเดา", เปิด web search สำหรับข้อมูลปัจจุบัน (ดูหัวข้อ 15-C)

ใช้ limit หมดเร็ว?

ลดโมเดล (Sonnet/Haiku) + ลด effort, /clear แชทที่ไม่เกี่ยว, อย่าแนบไฟล์ใหญ่เกินจำเป็น

ลืม context กลางทาง?

บทสนทนายาวเกินไป — เริ่มแชทใหม่, ย้าย context สำคัญไป Project knowledge, วางข้อมูลหลักไว้ต้น prompt

Glossary

คำศัพท์สำคัญที่ใช้ตลอดคู่มือนี้ รวมไว้ที่เดียว

คำความหมาย
Tokenหน่วยย่อยของข้อความที่โมเดลประมวลผล (ประมาณ 1 คำ = 1–2 tokens) ใช้วัดปริมาณ input/output
Context windowปริมาณข้อมูล (tokens) ที่ Claude "เห็น" ได้ในคราวเดียว ถ้าเกินจะเริ่มลืมส่วนต้น

Thinkingโหมดให้ Claude คิด/วางแผนก่อนตอบและแสดงกระบวนการคิด
Artifactผลงานที่แยกหน้าต่าง แก้และนำไปใช้ต่อได้ เช่นโค้ด เอกสาร เว็บ
Skillโฟลเดอร์คำสั่ง/สคริปต์ที่ Claude โหลดมาใช้เมื่อเจองานที่ตรง
