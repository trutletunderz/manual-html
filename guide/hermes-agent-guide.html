00

Hermes Agent คืออะไร

what it is

ไม่ใช่ coding copilot ที่ผูกกับ IDE และไม่ใช่ chatbot ครอบ API ตัวเดียว — แต่เป็น autonomous agent ที่ยิ่งรันนานยิ่งเก่งขึ้น เพราะมี closed learning loop ในตัว

แกนหลัก 4 อย่าง core pillars

ส่วนประกอบทำอะไร
Learning loopสร้าง skill จากงานจริง, ปรับปรุง skill ระหว่างใช้, curate memory เป็นระยะ, ค้นข้ามเซสชันด้วย FTS5 + LLM summarization
Runs anywhere6 terminal backend: local, docker, ssh, daytona, singularity, modal — Daytona/Modal เป็น serverless ที่ hibernate ตอน idle ค่าใช้จ่ายเกือบ 0
Lives where you doคุยได้ 20+ แพลตฟอร์ม: CLI, Telegram, Discord, Slack, WhatsApp, Signal, Matrix, Email, SMS, Teams ฯลฯ จาก gateway เดียว
Provider-agnosticใช้ได้กับ Nous Portal, OpenRouter, Anthropic, OpenAI, Qwen, DeepSeek, Kimi หรือ endpoint อะไรก็ได้ที่ OpenAI-compatible
บริบทของนาย · context นายเคยวางท่อ agentic pipeline ด้วย Hermes มาแล้ว รอบนี้ผมเรียง flow ให้เน้นจุดที่ต่อกับงาน xFish ได้ทันที: YouTrack-style task board → Kanban dispatcher, Telegram async gate approval, และ Docker/SSH sandbox สำหรับ serialize human judgment ตามหลัก "parallelize agents, serialize human judgment" ที่นายใช้อยู่
01

เตรียมเครื่อง

prerequisites

installer จัดการ dependency หลักให้เองทั้งหมด (Python 3.11 ผ่าน uv, Node.js v22, ripgrep, ffmpeg, Playwright/Chromium) — นายไม่ต้องลงเองทีละตัว สิ่งที่ต้องมีก่อนมีแค่ไม่กี่อย่าง

Linux / Ubuntu / WSL2

ต้องมี git + curl + xz-utils (installer ดาวน์โหลด Node.js เป็น .tar.xz) ถ้าจะใช้ Desktop app เพิ่ม build-essential

Windows (native)

ใช้ PowerShell ลงสคริปต์ได้เลย หรือใช้ Hermes Desktop installer ที่จัดการ dependency ให้ครบ (แนะนำสำหรับมือใหม่บน Windows)

ติดตั้ง prerequisite บน Ubuntu/Debian apt

bashUbuntu
# git + เครื่องมือที่ installer ต้องใช้
sudo apt update
sudo apt install -y git curl xz-utils
# เพิ่ม build-essential ถ้าจะใช้ Hermes Desktop (คอมไพล์ native module)
sudo apt install -y build-essential
# ตรวจว่ามี git แล้ว
git --version
ข้อกำหนดโมเดล · model requirement Hermes ต้องการโมเดลที่มี context อย่างน้อย 64,000 tokens โมเดล cloud ส่วนใหญ่ (Claude, GPT, Gemini, Qwen, DeepSeek) ผ่านสบาย ถ้ารัน local ต้องตั้ง ctx ≥ 64K เช่น --ctx-size 65536 (llama.cpp) หรือ -c 65536 (Ollama)
02

ติดตั้งบน Ubuntu

install · linux / wsl2

วิธีเร็วสุด — สคริปต์เดียวจบ installer จะ clone repo, สร้าง venv, ตั้งคำสั่ง hermes ให้เป็น global แล้วลง dependency ทั้งหมด

ติดตั้งแบบ command-line curl one-liner

bashLinux / macOS / WSL2 / Termux
curl -fsSL https://hermes-agent.nousresearch.com/install.sh | bash

เสร็จแล้ว reload shell แล้วเริ่มแชทได้เลย:

bashUbuntu
source ~/.bashrc   # หรือ source ~/.zshrc
hermes             # เริ่มแชท!
ทางลัดที่เร็วที่สุด · fastest path หลังลงเสร็จ สั่ง hermes setup --portal — OAuth ครั้งเดียวได้ทั้งโมเดล + Tool Gateway 4 ตัว (web search, image gen, TTS, cloud browser) ไม่ต้องไล่ใส่ API key ทีละตัว

ที่เก็บไฟล์หลังติดตั้ง install layout

วิธีลงโค้ดอยู่ที่binarydata dir
git installer (per-user)~/.hermes/hermes-agent/~/.local/bin/hermes~/.hermes/
pip installsite-packages~/.local/bin/hermes~/.hermes/
root mode (sudo)/usr/local/lib/hermes-agent//usr/local/bin/hermes/root/.hermes/

ลงบน server แบบ non-sudo / service user systemd account

กรณีรันเป็น user เฉพาะ (เช่น account hermes สำหรับ systemd) สิ่งเดียวที่ต้อง root คือ Playwright --with-deps ที่ apt-install shared lib ของ Chromium — แยกขั้นได้ตามนี้:

bashadmin (มี sudo) ทำครั้งเดียว
# 1) admin ลง system lib ที่ Chromium ต้องใช้
sudo npx playwright install-deps chromium
bashservice user (ไม่มี sudo)
# 2) รัน installer ปกติ — มันจะข้าม --with-deps ให้เอง
curl -fsSL https://hermes-agent.nousresearch.com/install.sh | bash

# ถ้าไม่ต้องใช้ browser automation เลย ข้าม Playwright ทั้งก้อน
curl -fsSL https://hermes-agent.nousresearch.com/install.sh | bash -s -- --skip-browser

# 3) ทำให้ shell ของ service user เห็นคำสั่ง hermes
echo 'export PATH="$HOME/.local/bin:$PATH"' >> ~/.bashrc

# 4) ตรวจสอบ
hermes doctor
WSL2 · ข้อควรระวัง บน WSL ให้ใช้ hermes gateway run (foreground) แทน gateway start เพราะ systemd บน WSL ไม่เสถียร — ห่อด้วย tmux เพื่อให้รันค้างไว้: tmux new -s hermes 'hermes gateway run'

ติดตั้งบน Windows

install · native / desktop

บน Windows มี 3 ทาง: (A) Desktop installer ครบจบในตัว, (B) PowerShell one-liner สำหรับ CLI ล้วน, (C) WSL2 แล้วใช้วิธี Ubuntu ด้านบน (เหมาะกับ dev ที่คุ้น Linux อยู่แล้ว)

A · Hermes Desktop installer แนะนำสำหรับมือใหม่

โหลด installer จาก hermes-agent.nousresearch.com แล้วรัน — ได้ทั้ง command-line + desktop app และจัดการ dependency ให้ครบ

B · PowerShell (CLI ล้วน) native

powershellWindows native
iex (irm https://hermes-agent.nousresearch.com/install.ps1)

ถ้าลง CLI ไปแล้วอยากเปิด Desktop app ทีหลัง สั่ง:

powershellWindows
hermes desktop
โค้ดอยู่ที่ไหนบน Windows · install path curl/PowerShell installer วาง repo ไว้ที่ %LOCALAPPDATA%\hermes\hermes-agent (ต่างจาก Linux ที่อยู่ ~/.hermes/hermes-agent) — เก็บไว้ใช้ตอนต้อง cd เข้าไปลง extras เช่น voice

C · ผ่าน WSL2 สาย Linux

เปิด Ubuntu บน WSL2 แล้วทำตามหัวข้อ 02 · ติดตั้งบน Ubuntu ได้เลย จุดเดียวที่ต่างคือเรื่อง gateway (ใช้ gateway run + tmux ตามที่เตือนไว้ข้างบน) — วิธีนี้นายจะได้สภาพแวดล้อม Linux เต็มซึ่งเข้ากับ pipeline ที่นายมีอยู่มากกว่า

เริ่มแชทหลังลงเสร็จ (ทุกแพลตฟอร์ม) first run

shellany OS
hermes            # classic CLI
hermes --tui      # TUI โหมดใหม่ (แนะนำ — mouse select, modal overlay)
04

ตั้งค่า & เลือก Provider

setup · provider · model

ขั้นที่สำคัญที่สุดของการตั้งค่า — เลือก provider ให้ใช้งานได้จริงก่อน ค่อยต่อ feature อื่น กฎเหล็ก: ถ้ายังแชทธรรมดาไม่ผ่าน อย่าเพิ่งเติม gateway/cron/skill

โหมดของ hermes setup 3 modes

โหมดเหมาะกับ
Quick Setup (Nous Portal)OAuth ฟรี ไม่ต้องใส่ API key — ได้โมเดล + Tool Gateway ครบ ทางที่เร็วที่สุด
Full Setupไล่ตั้งเองทุก provider/tool (เอา API key ตัวเองมาใส่)
Blank Slateเปิดเฉพาะขั้นต่ำ: provider+model, File Operations, Terminal เท่านั้น ที่เหลือปิดหมด — เหมาะตอนต้องการ agent ที่คุมทุกอย่างเอง แล้วค่อยเปิดเฉพาะที่ใช้

คำสั่งตั้งค่า setup commands

bashsetup
hermes setup --portal   # ทางลัด: OAuth Nous Portal + Tool Gateway จบในคำสั่งเดียว
hermes setup           # wizard เต็ม (ลงครั้งแรก) / reconfigure (ลงแล้ว)
hermes model           # เลือก provider + model (เพิ่ม provider ใหม่, OAuth, ใส่ key)
hermes tools           # เปิด/ปิด tool ต่อแพลตฟอร์ม
hermes gateway setup   # ตั้งค่าแพลตฟอร์ม messaging
hermes model ≠ /model hermes model (รันจาก terminal นอกเซสชัน) = wizard เต็ม เพิ่ม provider ใหม่/OAuth/ใส่ key ได้ ส่วน /model (พิมพ์ในเซสชัน) = สลับได้เฉพาะ provider/model ที่ตั้งไว้แล้วเท่านั้น ถ้าจะเพิ่ม provider ใหม่ต้องออกจากเซสชันก่อน

Provider ยอดนิยม (สำหรับงานนาย) good defaults

Providerคือตั้งค่ายังไง
Nous Portalsubscription zero-config, 300+ โมเดล + Tool GatewayOAuth ผ่าน hermes model
AnthropicClaude ตรงๆ (Max plan OAuth หรือ API key)hermes model → OAuth / ใส่ key
OpenRouterrouting ข้ามหลายโมเดลใส่ OPENROUTER_API_KEY
Qwen / DeepSeek / Kimiโมเดลจีน (นายเคยประเมินไว้)DASHSCOPE_API_KEY / DEEPSEEK_API_KEY / KIMI_API_KEY
Custom EndpointvLLM, SGLang, Ollama, OpenAI-compatibleตั้ง base URL + API key

secrets แยกจาก config ยังไง .env vs config.yaml

bashconfig set
# คำสั่งเดียว มันแยกไฟล์ให้เอง: key → .env, ที่เหลือ → config.yaml
hermes config set model anthropic/claude-opus-4.6
hermes config set terminal.backend docker
hermes config set OPENROUTER_API_KEY sk-or-...

แชทครั้งแรก + Sessions

first chat · resume

เริ่มด้วย prompt ที่เฉพาะเจาะจงและตรวจผลง่าย เพื่อยืนยันว่า provider + tool ทำงานครบ

1
เปิดเซสชัน
สั่ง hermes หรือ hermes --tui — banner ต้องโชว์ model/provider ที่เลือก
2
ทดสอบ tool
ลองสั่งงานที่ต้องใช้ terminal เช่น "What's my disk usage? Show top 5 largest dirs." — agent จะรันคำสั่งจริงให้
3
ยืนยัน multi-turn
คุยต่อมากกว่า 1 รอบโดยไม่ error = ผ่านจุดที่ยากที่สุดแล้ว
4
เช็ก resume
ออกแล้วสั่ง hermes -c ต้องกลับเข้าเซสชันเดิมได้

resume / slash commands sessions

bashsessions
hermes --continue    # ต่อเซสชันล่าสุด
hermes -c            # รูปแบบสั้น
hermes --resume <id> # ต่อเซสชันตาม id หรือ title
Slash command (ในเซสชัน)ทำอะไร
/helpโชว์คำสั่งทั้งหมด
/toolsดู tool ที่ใช้ได้
/model claude-sonnet-4สลับโมเดล (ตรวจ provider อัตโนมัติ) เติม --global เพื่อ save ถาวร
/reasoning highปรับระดับ thinking: none / minimal / low / medium / high / xhigh
/saveเซฟบทสนทนา
tip multi-line input: กด Alt+Enter / Ctrl+J / Shift+Enter · ขัดจังหวะ agent: พิมพ์ข้อความใหม่แล้ว Enter หรือ Ctrl+C

คำสั่ง CLI สำคัญ

essential cli

คำสั่งที่ใช้บ่อยที่สุด แยกเป็นกลุ่ม — ตัวเต็มดูที่ CLI Commands Reference

one-shot & scripting สำหรับ cron / CI / orchestrator

bashscripted
# -z = one-shot บริสุทธิ์ที่สุด: prompt เข้า, คำตอบสุดท้ายออก ไม่มีอย่างอื่น
hermes -z "What's the capital of France?"
answer=$(hermes -z "summarize this" < /path/to/file.txt)

# chat -q = one-shot แต่ยังเห็น tool output ใน transcript
hermes chat -q "Summarize the latest PRs"
hermes chat --provider openrouter --model anthropic/claude-sonnet-4.6

# worktree isolation สำหรับ parallel agent (เข้ากับ pattern ของนาย)
hermes chat --worktree -q "Review this repo and open a PR"

# safe-mode: ปิด customization ทั้งหมด เพื่อ isolate ว่าบั๊กมาจาก setup เราหรือ Hermes
hermes chat --safe-mode -q "Is this bug mine or Hermes'?"

global flags ที่ควรรู้ global options

Flagทำอะไร
-p <name>เลือก profile (หลาย agent แยกขาดกัน)
-w / --worktreeเริ่มใน git worktree แยก สำหรับ parallel-agent workflow
--yoloข้าม approval prompt ของคำสั่งอันตราย (ใช้ระวัง!)
--max-turns <N>จำนวน tool-calling สูงสุดต่อ turn (default 90)
--ignore-rulesไม่ inject AGENTS.md, SOUL.md, memory, preloaded skill

คำสั่งหน้าที่

07

config.yaml & .env

configuration

ทุกอย่างเก็บใน ~/.hermes/ — secret ไป .env, ที่เหลือไป config.yaml ลำดับความสำคัญ: CLI args → config.yaml → .env → built-in default

โครงสร้างไดเรกทอรี ~/.hermes/

textlayout
~/.hermes/
├── config.yaml     # settings: model, terminal, TTS, compression ฯลฯ
├── .env            # API keys + secrets
├── auth.json       # OAuth credentials (Nous Portal ฯลฯ)
├── SOUL.md         # identity ของ agent (slot #1 ใน system prompt)
├── memories/       # MEMORY.md, USER.md (จำข้ามเซสชัน)
├── skills/         # skill ที่ agent สร้างเอง
├── cron/           # scheduled jobs
├── sessions/       # gateway sessions
└── logs/           # errors.log, gateway.log (redact secret อัตโนมัติ)

จัดการ config manage

bashconfig
hermes config              # ดู config ปัจจุบัน
hermes config edit         # เปิด config.yaml ใน editor
hermes config set KEY VAL  # ตั้งค่าทีละตัว (แยกไฟล์ให้เอง)
hermes config check        # เช็ก option ที่หาย (หลัง update)
hermes config migrate      # เติม option ใหม่แบบ interactive

ตัวอย่าง config.yaml ที่ใช้บ่อย common knobs

yamlconfig.yaml
timezone: "Asia/Bangkok"          # IANA timezone (ของนาย)

agent:
  max_turns: 90                   # tool-calling สูงสุดต่อ turn
  reasoning_effort: ""            # none|minimal|low|medium|high|xhigh

compression:
  enabled: true
  threshold: 0.50               # compress ที่ % ของ context limit

memory:
  memory_enabled: true
  write_approval: false          # true = ขออนุมัติก่อนเขียน memory ทุกครั้ง

env substitution: # อ้าง env var ด้วย ${VAR} ได้
# auxiliary.vision.api_key: ${GOOGLE_API_KEY}
หลักการที่ตรงกับสไตล์นาย · factual context files Hermes inject context file พวก SOUL.md, AGENTS.md, CLAUDE.md, .cursorrules เข้า system prompt อัตโนมัติ (truncate ที่ context_file_max_chars, default 20000) — ตรงกับ pattern ที่นายเขียน CLAUDE.md เป็น factual statement เพื่อเลี่ยง prompt-injection guard ได้พอดี
08

Terminal Backend / Sandbox

isolation

agent มีสิทธิ์ filesystem เท่า user ที่รัน — ถ้าต้องการความปลอดภัย ใช้ Docker หรือ SSH backend นี่คือจุด "serialize human judgment" ที่นายเน้น เพราะ sandbox จำกัด blast radius ของ agent ได้

Backendคำสั่งรันที่ไหนเหมาะกับ
localเครื่องนายตรงๆ (ไม่มี isolation)dev ส่วนตัว
dockercontainer เดียวที่ persist ข้ามเซสชัน (cap-drop, no-new-privileges, pids-limit)sandbox ปลอดภัย, CI/CD
sshremote server ผ่าน SSH (ControlMaster reuse)remote dev, ฮาร์ดแวร์แรง
modal / daytonacloud sandbox (serverless, hibernate ตอน idle)ephemeral compute, eval
singularityApptainer container (--containall)HPC cluster, เครื่องที่ใช้ร่วม

สลับเป็น Docker sandbox recommended

bashswitch backend
hermes config set terminal.backend docker
docker version   # ยืนยันว่า Docker ทำงาน
yamlconfig.yaml — docker
terminal:
  backend: docker
  docker_image: "nikolaik/python-nodejs:python3.11-nodejs20"
  docker_run_as_host_user: true     # ไฟล์ที่สร้างเป็นของ host user ไม่ใช่ root
  docker_mount_cwd_to_workspace: true # mount โฟลเดอร์ที่เปิด Hermes เข้า /workspace
  docker_forward_env:                  # forward secret จาก host เข้า container
    - "GITHUB_TOKEN"
  container_cpu: 1
  container_memory: 5120             # MB

TERMINAL_SSH_PORT=22          # optional (default 22)
TERMINAL_SSH_KEY=~/.ssh/id_ed25519  # optional
คำเตือนความปลอดภัย · security บน local backend agent เข้าถึง filesystem ได้เท่า account นาย ถ้างานมีความเสี่ยง ใช้ hermes tools ปิด tool ที่ไม่ต้องการ หรือสลับไป Docker เพื่อ sandbox — และอย่าลืม --yolo คือการ ปิด approval prompt ใช้เฉพาะตอนที่รู้แน่ว่าปลอดภัย
09

Gateway (Telegram ฯลฯ)

messaging gateway

ต่อ agent เข้า 20+ แพลตฟอร์มจาก gateway เดียว — นี่คือชั้นที่ทำให้ "คุยกับ agent บน cloud VM ผ่าน Telegram โดยไม่ต้อง SSH" เป็นจริง และเป็นที่ที่นายจะวาง async gate approval ได้

ตั้งค่า + รัน gateway setup & run

bashgateway
hermes gateway setup    # wizard ตั้งค่าแพลตฟอร์ม (Telegram, Discord, Slack ฯลฯ)
hermes gateway run      # รัน foreground (แนะนำสำหรับ WSL, Docker, Termux)
hermes gateway install  # ลงเป็น systemd (Linux) / launchd (macOS) service
hermes gateway start    # สตาร์ท background service
hermes gateway status   # ดูสถานะ

รันค้างบน VPS ด้วย systemd always-on

bashUbuntu server
# ลงเป็น service แล้ว enable ให้ start ตอน boot
hermes gateway install
hermes gateway start
hermes gateway status
# บน WSL ใช้ tmux แทน เพราะ systemd ไม่เสถียร:
tmux new -s hermes 'hermes gateway run'

ส่งข้อความ/ไฟล์แบบ one-shot (ops script) hermes send

bashsend
# ส่งข้อความ — ไม่ต้องปลุก agent loop, ไม่กิน token
hermes send --to telegram "deploy finished"
echo "RAM 92%" | hermes send --to telegram:-1001234567890

# ส่งไฟล์/ภาพเป็น attachment ด้วย MEDIA: directive
hermes send --to telegram "Build chart MEDIA:/tmp/chart.png"
hermes send --to discord:#ops --file /tmp/report.md
เชื่อมกับ two-gate architecture ของนาย hermes send + hermes webhook subscribe --deliver telegram คือชิ้นส่วนที่ใช้ทำ async gate approval ผ่าน Telegram ได้เลย — agent ทำงานไป, ส่งสรุปมา Telegram, นายกดอนุมัติ/ปฏิเสธจากมือถือ ตรงกับหลัก "serialize human judgment" ที่นายออกแบบ pipeline ไว้

curator · self-maintenance Hermes มี curator ที่ทบทวน skill ที่ agent สร้างเอง, prune ตัวที่ stale, archive ตัวที่ล้าสมัย (skill bundled/hub ไม่แตะ) — hermes curator status / hermes curator run --dry-run เพื่อ preview ก่อน

Cron / Automations / Kanban

scheduled · orchestration

cron ในตัวส่งผลไปแพลตฟอร์มไหนก็ได้ ส่วน Kanban เป็น board หลาย-profile ที่ dispatcher กวาดงานให้ agent worker ทำ — ใกล้เคียง YouTrack-as-canonical-state ที่นายออกแบบไว้

cron jobs built-in scheduler

bashcron
hermes cron list                    # ดู job ที่ตั้งไว้
hermes cron create                  # สร้าง job จาก prompt (แนบ skill ด้วย --skill ได้)
hermes cron run <id>               # trigger job รอบ tick ถัดไป
hermes cron status                  # scheduler ทำงานอยู่ไหม

Kanban board multi-agent task queue

bashkanban
hermes kanban init                         # สร้าง board (SQLite — durable state ที่นายชอบ)
hermes kanban create "Fix login bug" --assignee dev
hermes kanban list                         # ดู task
hermes kanban specify <id>                 # ขยาย one-liner เป็น spec แล้วเลื่อนไป todo
hermes kanban decompose <id>               # แตก task เป็น sub-task ส่งให้ specialist profile
hermes kanban dispatch                     # dispatcher กวาด 1 รอบ
map กับ pipeline ของนาย · architecture fit Kanban + SQLite durable state + dispatcher + kanban specify/decompose ผ่าน auxiliary model = โครงสร้างเดียวกับ "parallelize agents, serialize human judgment" ที่นายวางไว้ — task เป็น canonical state, worker claim แบบ atomic (race guard), kanban block = human gate, kanban complete = ปิดงาน นายเอา YouTrack เป็น source of truth แล้ว mirror สถานะมา Kanban หรือใช้ Kanban ตรงๆ ก็ได้

แก้ปัญหา (Doctor)

troubleshooting

ลำดับการกู้สถานะเมื่อรู้สึก "พังๆ": doctor → model → setup → sessions list → --continue → gateway status

อาการสาเหตุที่น่าจะเป็นวิธีแก้
hermes: command not foundshell ยังไม่ reload / PATH ไม่ครบsource ~/.bashrc หรือเพิ่ม ~/.local/bin ใน PATH
API key not setยังไม่ตั้ง providerhermes model หรือ hermes config set OPENROUTER_API_KEY ...
ตอบว่างเปล่า/พังprovider auth หรือเลือก model ผิดhermes model ยืนยัน provider/model/auth
gateway สตาร์ทแต่ไม่มีใครคุยได้bot token / allowlist ยังไม่ครบhermes gateway setup แล้วเช็ก gateway status
--continue หา session เดิมไม่เจอสลับ profile / session ไม่ได้เซฟhermes sessions list ยืนยันว่าอยู่ profile เดิม
config มีปัญหาหลัง updateoption หาย/เก่าhermes config check แล้ว hermes config migrate

เครื่องมือวินิจฉัย diagnostics

bashdoctor
hermes doctor          # บอกว่าอะไรหาย + วิธีแก้
hermes doctor --fix    # พยายามซ่อมอัตโนมัติ
hermes dump            # สรุป setup copy-paste ไปถาม Discord/GitHub
hermes logs -f          # tail agent.log แบบ real-time
hermes logs gateway -n 100           # 100 บรรทัดล่าสุดของ gateway.log
hermes logs --level WARNING --since 1h  # เฉพาะ warning+ ใน 1 ชม.ที่ผ่านมา

ความปลอดภัย supply-chain security audit

bashsecurity

หมายเหตุสำหรับนาย · ongoing CVE awareness นายติดตามเรื่อง CVE/supply-chain ของ OpenClaw อยู่แล้ว — hermes security audit ทำหน้าที่เดียวกันฝั่ง Hermes ควรใส่ใน CI/cron เป็นประจำ และระวังว่า hermes skills install จาก third-party แม้สแกนแล้ว ก็ยังควรรีวิว SKILL.md เองก่อนเปิดใช้บน server บริษัท
