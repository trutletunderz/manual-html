xFish Orchestrator — Build Spec

สเปกไฟล์เดียวสำหรับสร้าง "ตัวร้อย" บางๆ ที่ขับ pipeline 7 ขั้นแบบ human-gated, อ่าน/เขียน state ลง SQLite, แสดงผลผ่าน Kanban, และเปิดช่องให้ Hermes เรียนสไตล์โดยไม่แตะ judgment ของมนุษย์

A spec for Claude Code to build a thin orchestrator that drives a 7-stage human-gated pipeline on a local Mac. No inbound server — outbound-only, runs 24/7. State in SQLite. Hermes plugs in as a style layer that never touches the gates.

00ภาพรวม

ระบบนี้ทำให้นายเป็น "ผู้จัดการ" เหนือ agent — ตัดสินที่ 2 จุดเท่านั้น ที่เหลือเป็น auto

ส่วนประกอบเกือบทั้งหมดนายมีอยู่แล้ว สิ่งเดียวที่ต้องสร้างใหม่คือ ตัวร้อย — สคริปต์ Python บางๆ ที่อ่าน state แล้ว "กดปุ่ม" slash command ตามลำดับ ไม่ใช่ framework

ส่วนทำหน้าที่สถานะ
ตัวร้อย (Python)poll → อ่าน state → dispatch command → เขียน stateสร้างใหม่
State storeSQLite ไฟล์เดียว · จำว่า task ไหนอยู่ขั้นไหน (durable)สร้างใหม่
Kanbanอ่าน state มาแสดงคอลัมน์ตาม "ใครถือลูก"สร้างใหม่ (เล็ก)
Claude Code (Max ×5)มือเขียน · build + self-healมีแล้ว
Codex (ChatGPT Plus)reviewer อิสระ · คนละหัวกับคนเขียนมีแล้ว
.claude/commandsนิยามแต่ละ stage (/intake, /implement)มีแล้ว
CLAUDE.mdstyle layer แบบ static (convention)มีแล้ว
Playwrightfunctional E2E + visual regressionมีแล้ว
Plan Review Gatesource of truth ของการอนุมัติ (TS/Firestore)มีแล้ว
Hermesเรียนสไตล์ตั้งแต่แรก → ป้อน plan ทีหลังlearn-first

01หลักการ + litmus test

"parallelize agents, serialize human judgment" — ขนาน agent ได้ แต่ judgment ต้องเรียงผ่านมนุษย์ทีละอัน

litmus test · ใช้เช็คทุกไอเดียในอนาคต
ใช้ AI/Hermes เพื่อ จำ / เสนอ / ร่าง ได้
เพื่อ ตัดสิน / ลงมือแทนนาย ไม่ได้

กฎที่ห้ามแตะ:

  • gate ทั้งสองเป็นของนายคนเดียว — ไม่มี agent ใด auto-approve, auto-merge, หรือสร้างแล้วรัน task เอง
  • operational state อยู่ใน SQLite (deterministic, แม่นยำ) — ไม่ใช่ความจำของ LLM
  • self-heal loop ต้องมีเพดาน — max_attempts แล้ว escalate ไม่วนไม่จบ
  • blind_approve_rate คือ metric เตือนภัย — ถ้า gate ไหนกดผ่าน 100% ติดกันนานๆ ให้สงสัยว่า gate นั้นกำลังตาย

02state schema (SQLite)

หัวใจของทั้งระบบ — ตารางเดียวที่ทุกส่วนอ้างอิง ปิด/sleep เครื่องแล้วเปิดมาทำต่อจากตรงนี้

schema.sql
CREATE TABLE tasks (
  id            TEXT PRIMARY KEY,        -- youtrack id เช่น xFish-1320
  title         TEXT NOT NULL,
  stage         TEXT NOT NULL,           -- ดู state machine ขั้น 03
  risk_tier     TEXT NOT NULL DEFAULT 'medium', -- low | medium | high
  attempts      INTEGER NOT NULL DEFAULT 0,  -- นับรอบ self-heal
  worktree      TEXT,                    -- path ของ git worktree
  plan_path     TEXT,                    -- artifact ของ plan
  preview_url   TEXT,                    -- link ให้นายดูที่ gate 2
  pr_url        TEXT,
  last_error    TEXT,                    -- เก็บไว้ตอน escalate
  updated_at    TEXT NOT NULL            -- ISO8601
);

-- log ทุก transition ไว้ตรวจย้อนหลัง + feed metric
CREATE TABLE transitions (
  id         INTEGER PRIMARY KEY AUTOINCREMENT,
  task_id    TEXT NOT NULL,
  from_stage TEXT, to_stage TEXT NOT NULL,
  actor      TEXT NOT NULL,             -- system | user
  note       TEXT,
  at         TEXT NOT NULL
);

-- ฟีดให้ Hermes เรียน (ขั้น 10) — เก็บการแก้/ตัดสินของนายที่ gate
CREATE TABLE gate_feedback (
  id        INTEGER PRIMARY KEY AUTOINCREMENT,
  task_id   TEXT NOT NULL,
  gate      TEXT NOT NULL,              -- gate1 | gate2
  verdict   TEXT NOT NULL,              -- approved | edited | rejected
  diff      TEXT,                       -- สิ่งที่นายแก้ (training signal)
  at        TEXT NOT NULL
);

Single source of truth. The orchestrator is the only writer; Kanban and metrics are readers. gate_feedback is the data Hermes learns from — generated automatically by your normal gate decisions.

03state machine 7 ขั้น

แต่ละ task เดินผ่าน stage เหล่านี้ ตัวร้อยอ่าน stage แล้วรู้ว่าต้องทำอะไรต่อ

stageใครทำอะไร→ ไปไหนต่อ
triageautopoll YouTrack, จัด risk, แจ้งนายนายเลือก → planning
planningauto/intake → ร่าง plan (+ Hermes style)gate1
gate1นายอนุมัติ plan (risk ต่ำ auto-pass)approve → building
buildingauto/implement + lint/test/Playwrightเขียว → review · fail → self-heal
reviewautoCodex ตรวจ diff + visual diffgate2
gate2นายดู preview · UX/logic/สวยงามapprove → ship · แก้ → planning
shipautoเปิด PR, ปิด YouTrack, แจ้งนายdone → task ถัดไป
loop guard

ที่ building: ถ้า test fail → attempts++ แล้ว self-heal วนใน stage เดิม จนถึง max_attempts (เริ่มที่ 3) เกินนั้น → set last_error, escalate หานายผ่าน Telegram, หยุดวน

04ตัวร้อย (orchestrator)

แก่นทั้งหมด: อ่าน state → เลือก stage ถัดไป → dispatch → เขียน state กลับ (next-step sequencing policy)

runner.py — โครงหลัก (pseudocode)
def tick():                       # เรียกทุก 1 ชม. ตอนเครื่องตื่น
    poll_youtrack_into_triage()   # stage 1: เพิ่ม task ใหม่
    for task in load_actionable():  # task ที่ stage เป็น auto
        match task.stage:
            case "planning":
                plan = dispatch(f"/intake {task.id}", model="opus")
                advance(task, "gate1"); notify_gate(task, "gate1")
            case "building":
                res = dispatch(f"/implement {task.id}", model="sonnet",
                               worktree=task.worktree)
                if res.green:
                    advance(task, "review")
                elif task.attempts < MAX_ATTEMPTS:
                    bump_attempt(task)         # self-heal วนต่อ
                else:
                    escalate(task, res.error)   # เกินเพดาน → หานาย
            case "review":
                verdict = codex_review(task)    # reviewer อิสระ
                advance(task, "gate2"); notify_gate(task, "gate2")
            case "ship":
                open_pr(task); close_youtrack(task); advance(task, "done")

# gate ไม่ได้อยู่ใน tick — มันถูก trigger ด้วย Telegram callback (ขั้น 05)
# dispatch = subprocess: claude -p "<cmd>" --output-format json --permission-mode ...
dispatch ผ่าน headless bridge
claude -p "/intake xFish-1320" \
  --output-format json \
  --permission-mode plan \      # intake = read-only
  --model opus

claude -p "/implement xFish-1320" \
  --output-format json \
  --allowedTools "Edit,Bash(npm test),Bash(npm run lint)" \  # scope แคบ
  --model sonnet
หมายเหตุ

ตัวร้อยเป็น stateless ต่อ tick — ทุกอย่างที่ต้องจำอยู่ใน SQLite ไม่ใช่ในตัวแปร process เพราะงั้นปิดเครื่องกลางคันแล้วเปิดมา tick ใหม่ ทุก task เดินต่อจากเดิมได้

05gates + Telegram

2 จุดที่นายตัดสิน — async ผ่านมือถือ โค้ดวิ่งจริงตอนเครื่องตื่นรอบถัดไป

Telegram = push (เด้งมาหานายตอนถึง gate) · Kanban = pull (นายเปิดดูภาพรวมเมื่อต้องการ) สองช่องเสริมกัน

gate callback contract
# gate 1 — อนุมัติ plan
# Telegram ส่งปุ่ม: [ดู diff] [/approve] [/reject] [แก้: พิมพ์ตอบ]
# เงื่อนไขกันการกดมั่ว: ต้องแตะ "ดู diff" ก่อน /approve จึงจะรับ
on_callback("approve", task_id, gate="gate1"):
    record_feedback(task_id, "gate1", "approved")
    advance(task_id, "building")
    create_worktree(task_id)

on_callback("reject" | edit_text, task_id, gate="gate1"):
    record_feedback(task_id, "gate1", "edited", diff=edit_text)
    advance(task_id, "planning")   # ร่างใหม่ตาม feedback

# gate 2 — ตรวจผลที่เขียวแล้ว (UX / logic / ความสวยงาม)
on_callback("approve", task_id, gate="gate2"):
    record_feedback(task_id, "gate2", "approved")
    advance(task_id, "ship")

on_callback(edit_text, task_id, gate="gate2"):
    record_feedback(task_id, "gate2", "edited", diff=edit_text)
    advance(task_id, "planning")   # human-feedback loop
สำคัญ

ทุก callback เขียน gate_feedback เสมอ — นี่คือจุดที่การตัดสินของนายกลายเป็น training data ให้ Hermes โดยอัตโนมัติ (ดูขั้น 10)

06risk tiers

ไม่ใช่ทุก task ต้องผ่าน gate 1 เต็มรูปแบบ — risk ต่ำ auto-pass เพื่อให้นาย "ดูจริง" เฉพาะที่ควรดู

tierตัวอย่างgate 1
lowcopy, docs, component เดี่ยวที่ไม่มี mutationauto-pass (log ไว้)
mediumUI หลายส่วน, state ในหน้าเดียวนายดู (เด้ง Telegram)
highmutation, auth, payment, migration, shared stateบังคับดู · ห้าม auto

ใช้ risk_policy.py / risk-tiers.json ที่มีอยู่แล้ว · gate 2 ไม่ขึ้นกับ risk — ตรวจ "สวยงาม" เป็นมนุษย์เสมอ (Playwright ตรวจไม่ได้)

verify ก่อนรบกวนนาย — นายเห็นแค่งานที่ "เขียวแล้ว" เท่านั้น

check แยก 3 ชั้น ไปคนละปลายทาง:

checkเครื่องตรวจได้?ปลายทาง

visual baseline ของ Playwright ต้องมาจาก mockup ที่นายอนุมัติ ไม่ใช่ auto-baseline จาก output ของ agent — ไม่งั้นนายให้ agent ตรวจการบ้านตัวเอง

08Codex review (อิสระ)

คนเขียน ≠ คนตรวจ — ทำให้คำว่า "เขียว" เชื่อถือได้

Claude Code (Sonnet) เป็นคนเขียนที่ building → Codex เป็นคนตรวจที่ review ตรวจ diff เทียบ Bruno spec, รัน scrutinize, เช็ก visual diff ก่อนส่งถึง gate 2

ทำไมสำคัญ

ถ้า agent ตัวเดียวเขียนทั้งโค้ดและ test "ผ่าน" แทบไม่มีความหมาย — มันเขียน test ให้ตัวเองผ่านได้ test ต้องผูกกับ Bruno spec / mockup (source of truth) ไม่ใช่ผูกกับ implementation

09Kanban board

หน้าต่างอ่าน state — ไม่ใช่ระบบใหม่ แค่ SELECT จาก SQLite มา render

จัดคอลัมน์ตาม "ใครถือลูก" ไม่ใช่ตาม 7 ขั้น (รอเลือก / ระบบกำลังทำ / รอนายตัดสิน / เสร็จ) การ์ดบอกขั้นปัจจุบัน + progress · คอลัมน์ "รอนายตัดสิน" ไฮไลต์

kanban query — จัดกลุ่มตาม actor lane
SELECT id, title, stage, risk_tier, attempts, preview_url,
  CASE
    WHEN stage = 'triage'                THEN 'รอเลือก'
    WHEN stage IN ('gate1','gate2')      THEN 'รอนายตัดสิน'
    WHEN stage = 'done'                  THEN 'เสร็จ'
    ELSE                                  'ระบบกำลังทำ'
  END AS lane
FROM tasks WHERE stage != 'archived'
ORDER BY updated_at DESC;

render ด้วย pattern Vue 3.5 + Naive UI + Pinia ที่มีอยู่ (จาก Plan Review Gate dashboard) · รันเป็นหน้า localhost บน Mac เปิดใน browser ตอนทำงาน · orchestrator ขยับ stage = การ์ดขยับคอลัมน์เอง

10Hermes — phase 1 (style → plan)

Hermes อยู่คนละท่อกับ Kanban โดยสิ้นเชิง — มันเรียน "สไตล์" ส่วน SQLite จำ "สถานะ"

วงเรียนรู้ปิด 2 ทิศ — เสียบที่ planning เท่านั้น ไม่แตะ gate:

  • ป้อนออก (เส้นเขียวทึบ): ตอน /intake เรียก Hermes ขอ style hint → plan ร่างมา "เป็นนาย" ตั้งแต่แรก
  • เรียนกลับ (เส้นเขียวประ): ทุก row ใน gate_feedback (นายแก้/อนุมัติ/ปฏิเสธ) ไหลกลับสอน Hermes ว่า "ที่ถูกควรเป็นแบบนี้"
hermes hook — interface เท่านั้น (phase 1)
# ป้อน style เข้า plan — เปิดเมื่อ Hermes มีข้อมูลพอ (ทีหลัง)
if HERMES_SUGGEST:
    style = hermes.suggest_style(task)   # คืน hint, ไม่ตัดสินใจ
    intake_prompt += style.as_context()

# เรียนจาก gate — เปิดตั้งแต่ loop นิ่ง (เก็บ data ทันที, ไม่ป้อนกลับ)
def on_gate_feedback(row):
    if HERMES_LEARN:
        hermes.learn(row.task_id, row.verdict, row.diff)
เส้นแบ่งที่ห้ามข้าม

Hermes เรียน จาก การตัดสินของนายได้ — แต่ห้ามเรียนแล้ว แทน การตัดสิน · ห้ามมีเส้นจาก Hermes ไปยัง advance(task, "building") หรือ "ship" เด็ดขาด gate ยังเป็นของนายเสมอ

เริ่มด้วยกัน — เรียนก่อน ป้อนทีหลัง

เสียบ Hermes ตั้งแต่แรกได้ แต่เปิดทีละเส้น: HERMES_LEARN=true ตั้งแต่ loop นิ่ง (เก็บการตัดสินของนายสะสมทันที) · HERMES_SUGGEST=false ไว้ก่อน เพราะวันแรก Hermes ยัง data-starved + CLAUDE.md ครบอยู่แล้ว · พอ feedback สะสมพอ (plan เริ่มมีจุดที่ CLAUDE.md เขียนเป็นกฎไม่ได้) ค่อยพลิก HERMES_SUGGEST=true — ตอนนั้น Hermes มีข้อมูลหลายสัปดาห์รออยู่แล้ว ไม่เริ่มจากศูนย์

11Hermes — future hooks

จุดต่อที่เว้นไว้ — ทุกอันผ่าน litmus test (จำ/เสนอ/ร่าง) เปิดทีละอันเมื่อ data แน่น

P1project memory ให้ intake

Hermes สะสมความรู้ codebase ข้ามเวลา (pattern เคยใช้, Bruno convention, เคยแก้คล้ายนี้ยังไง) → /intake query "เรื่องนี้เคยทำไหม" ต่อยอด vault-search ที่มีอยู่

P2เสนอลำดับ task ที่ triage

การ ดึง task ยังเป็นงาน deterministic ของตัวร้อย (poll → เขียน state) — Hermes ไม่ดึงเอง เพราะนั่นคือ "ลงมือ" และทำให้ Hermes กลายเป็น orchestrator ซ้อน · แต่หลังดึงมาแล้ว Hermes เรียนได้ว่านายมักเลือกอะไรก่อน (bug ก่อน feature, task เล็กตอนเช้า) → เสนอลำดับ/ไฮไลต์ "น่าทำตัวนี้ก่อน" นายยังเป็นคนเลือก ผ่าน litmus (เสนอ)

P2rationale log → CLAUDE.md promotion

บันทึก "ทำไม" นายแก้ที่ gate · ถ้ากฎเดิมโผล่ซ้ำหลายครั้ง → เตือนนายว่า "อันนี้ควรเลื่อนจาก soft memory เป็นกฎตายตัวใน CLAUDE.md" Hermes เป็นตัวค้นว่าอะไรควรกลายเป็น convention

P2ร่างงานในสำเนียงนาย

PR description, YouTrack comment, commit message, changelog — Hermes ร่างก่อน นายอ่านแล้วกดส่ง (เสนอ/ร่าง ไม่ใช่ส่งเอง)

P3advisory digest เชิงรุก

เฝ้า codebase เป็นสัปดาห์ → "pattern นี้ซ้ำ 6 ที่ น่า refactor" หรือ "component นี้โดน reject บ่อย" → สรุปเป็น draft task ส่ง Telegram · เสนอเท่านั้น ไม่ลงมือ

P3จำ spec ข้าม session

ถือ HTML-as-spec ไว้ระหว่าง session (workflow "เริ่มหน้าใหม่") ไม่ต้อง re-feed context เองทุกครั้ง

ห้าม (ตก litmus)
  • ให้ Hermes เฝ้าซ่อม orchestrator เอง — overkill, log + alert พอ
  • ให้ Hermes ดึง/route task เอง — เป็นงาน deterministic ของตัวร้อย (ตก "ลงมือ")
  • auto-approve / auto-merge / สร้างแล้วรัน task เอง — ตก "ตัดสิน/ลงมือแทนนาย"

รัน 24/7 บน MacBook Air M4 (เปิดทิ้งไว้) — ปลอดภัยได้เพราะ outbound-only + มี guard กั้น runaway

ทำไม 24/7 ยังปลอดภัย

security win เดิม (vs OpenClaw) คือ "ไม่มี inbound server เปิดออกเน็ต" — ไม่ใช่ "uptime ต่ำ" · ตัวร้อยเป็น outbound-only (poll YouTrack, long-poll Telegram, เรียก claude -p) ไม่เปิด port รับเข้า เปิดทิ้ง 24 ชม. หรือ 1 ชม. attack surface เท่ากัน

  • 24/7 ช่วยเฉพาะ auto-stage — planning/build/review เดินตอนนายหลับได้ แต่ gate ยังหยุดรอนายอยู่ดี · ประโยชน์จริง = "อนุมัติ plan ก่อนนอน → ตื่นมาเจอผลที่ build+review เสร็จรอที่ gate 2"
  • runaway ถูกกั้น — เปิดทิ้งไว้ไม่เผา token มั่ว เพราะ self-heal cap ที่ max_attempts และทุกอย่างหยุดที่ gate
  • กัน sleep — ใช้ caffeinate ตอนรัน หรือ pmset · เสียบสายไฟไว้ (งานเบา M4 fanless รับได้สบาย เพราะ inference อยู่บน cloud ไม่รันโมเดล local)
  • รันผ่าน launchd LaunchAgent — survive restart, รัน unattended · เริ่มจากรันมือเห็น log ก่อน แล้วค่อยอัปเป็น LaunchAgent
  • คง outbound-only (เส้นห้ามข้าม) — Telegram ใช้ getUpdates (long-poll) ไม่ใช่ webhook · Kanban bind localhost เท่านั้น ห้าม 0.0.0.0 · ข้ามเมื่อไหร่ security win หายทันที
  • secret (YouTrack/Telegram token) เก็บใน .env นอก git · SQLite ไฟล์ local ไม่ sync ออก · 16GB ไม่ใช่คอขวด

14วิธีป้อนให้ Claude Code

ไฟล์นี้คือ spec — โยนเข้า Claude Code แล้วให้มันสร้างตามนี้ทีละ phase

prompt เริ่มต้น (phase 0)
อ่าน spec ใน xfish-orchestrator-spec.html

สร้าง phase 0 ก่อน:
1. schema.sql ตามขั้น 02
2. runner.py ตามขั้น 04 (tick loop, dispatch ผ่าน claude -p)
3. gate handler ตามขั้น 05 (Telegram callback + gate_feedback)
4. max_attempts + escalate ตามขั้น 03

ยังไม่ต้องทำ: Kanban, Codex, Hermes (เปิด flag ไว้ปิด)
ใช้ convention ใน CLAUDE.md · ทุก state เขียนลง SQLite ไม่ใช่ตัวแปร process
ทำ plan มาก่อน ฉันจะอนุมัติเอง

สังเกตบรรทัดสุดท้าย — แม้แต่ตอนสร้างระบบ นายก็ยัง serialize judgment ของตัวเอง: ให้มันทำ plan มาให้นายอนุมัติก่อน ลงมือ หลักการเดียวกับระบบที่กำลังสร้าง
