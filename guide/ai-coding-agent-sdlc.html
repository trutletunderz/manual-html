SDLC

ทำงานร่วมกับ AI Agent ตลอด lifecycle ตั้งแต่ Planning ถึง Operate — ไม่ใช่แค่ให้มันเขียนโค้ดแทน แต่คือการออกแบบ workflow คุมคุณภาพ และ verify ผลลัพธ์อย่างมืออาชีพ

Throughline — เส้นเรื่องที่ร้อยทั้ง 8 module
คนพิมพ์ prompt→ คนออกแบบระบบ + คุม context + verify ผลลัพธ์

จุด leverage จริงของ agent ไม่ได้อยู่ที่มัน "เขียนเก่งแค่ไหน" แต่อยู่ที่สองอย่าง — ทุก module ด้านล่างวนอยู่รอบสองเรื่องนี้:

01 · Context Engineering

ป้อนของถูกให้มันคิด — context ขาดหรือเกินคือต้นตอของผลลัพธ์ที่ผิดแบบดูเนียน

02 · Verification Loop

จับมันได้ตอนคิดผิด — เพราะ agent มั่นใจเท่ากันทั้งตอนถูกและตอนผิด

01

AI Coding Agent คืออะไร

ต้องรู้ก่อน: ไม่มี — รากฐานของทั้งหมด ขยายเต็ม
1.1

Agentic Loop

  1. กลไกเต็มของ loop — วงจร รวบรวม context → วางแผน → เลือก tool → execute → สังเกตผล → ป้อนกลับ → วนใหม่ จนชน stopping condition สืบสายจาก ReAct (สลับ "คิด" กับ "ลงมือ" ทีละ step)
    ตัวอย่าง"แก้ bug ปุ่ม login กดไม่ติด" → grep หาไฟล์ → อ่าน LoginForm.vue → เห็น @click ผูกผิด method → แก้ → รัน test → เขียว → จบ · หนึ่งงาน = หลายรอบ loop เล็ก ๆ ต่อกัน
  2. Stopping condition มี 4 แบบ ที่ต้องแยกออก: เสร็จจริง / ชน max turns–token / เจอ error ไปต่อไม่ได้ / คุณ interrupt เอง
    ตัวอย่างมันบอก "เสร็จแล้ว" ฟังดูดี แต่จริง ๆ หยุดเพราะ ชน token limit กลางคัน → งานเสร็จแค่ครึ่งเดียว ต้องดูออกว่าหยุดแบบไหน
  3. Failure mode #1 — context เน่า: อ่านผิดไฟล์ / อ่านเยอะเกินจนของสำคัญถูกกลบ / observation เก่าสะสมจน signal จม noise
    ตัวอย่างให้อ่านทั้ง repo 40 ไฟล์ก่อนแก้ utils/date.ts → ตอนแก้มันอ้าง date helper ของอีกไฟล์ที่ไม่เกี่ยว เพราะ context ปนกันหมด
  4. Failure mode #2 — เลือก tool ผิด: ควร grep แต่ไล่อ่านทีละไฟล์ / ควรรัน test แต่เดาผลเอา
    ตัวอย่างหา "ที่ไหนเรียก API /wallet" ที่ถูกคือ grep "wallet" ได้ใน 1 step แต่มันเปิดอ่านทีละไฟล์ 20 รอบ เผา turn ทิ้งเปล่า
  5. Failure mode #3 — fix-the-symptom loop: วนแก้ให้ error หายทีละอันโดยไม่แตะ root cause
    ตัวอย่างTypeError ที่ user.name → ใส่ user?.name → เด้ง error ใหม่ที่ user.email → ใส่ ?. อีก → วนไม่จบ ทั้งที่ root cause คือ API ไม่ได้ return user
  6. ทักษะที่ต้องฝึก — "อ่าน loop ออก": มองทุก turn แล้วบอกได้ว่าอยู่ขั้นไหน (gather? execute? วน?) และพลาดขั้นไหน เพื่อกลับไปแก้ต้นเหตุ ไม่ใช่เดาสุ่ม
Step การลงมือจริง · 1.1
  1. ดูทุก turn ว่า agent อยู่ขั้นไหน — gather / plan / execute / observe
  2. เมื่อผลเพี้ยน ถามว่า พลาดขั้นไหน: context เน่า / เลือก tool ผิด / วน symptom
  3. ถ้า context เน่า → /clear แล้วป้อนเฉพาะที่เกี่ยว
  4. ถ้าเลือก tool ผิด → ชี้ tool ที่ควรใช้ตรง ๆ (เช่น "ใช้ grep")
  5. ถ้าวน symptom → หยุด กลับไปหา root cause
  6. เช็ค stopping condition: หยุดเพราะ เสร็จจริง หรือ ชน limit กลางคัน

1.2

Permission Model

  1. 3 ระดับ: allow / ask / deny — ตั้งเป็น pattern ต่อ tool (Read = allow, Bash(rm:*) = deny)
    ตัวอย่าง config"allow": ["Read","Grep","Bash(npm test)"] · "deny": ["Bash(rm:*)","Bash(git push:*)"]
  2. 2 ชั้น: global vs project — project = .claude/settings.json ใน repo commit ได้ → ทั้งทีมใช้ permission ชุดเดียวกัน ของที่เป็นนโยบายทีมใส่ที่นี่
    ตัวอย่างทีม xFish commit deny: git push origin main ไว้ใน project settings → ไม่มีใคร (หรือ agent ของใคร) push เข้า main ตรง ๆ ได้ ทุกคนถูกบังคับเท่ากัน
  3. หลักทอง — auto-approve เฉพาะ idempotent + read-only: Read, Grep, Glob, รัน test, git status/diff รันซ้ำกี่ครั้งก็ไม่เปลี่ยนสถานะระบบ
    ตัวอย่างgit diff รัน 100 ครั้งระบบเหมือนเดิม → allow ปลอดภัย เร่งความเร็วได้เต็มที่
  4. gate ทุกอย่างที่เปลี่ยนสถานะ — เขียน/ลบไฟล์, push, migration, deploy เส้นแบ่งคือ blast radius
    ตัวอย่าง blast radiusแก้ README.md พลาด = กระทบ 1 ไฟล์ (เล็ก) · npm run deploy:prod พลาด = ลูกค้าทั้งระบบเจอ (กว้าง) → อันหลังต้อง ask เสมอ
  5. กับดัก — over-permissioning เพราะขี้เกียจ: เปิด allow Bash ทั้งหมดเพราะเบื่อกด approve
    ตัวอย่างหายนะallow Bash(*) → agent เข้าใจ task ผิด รัน rm -rf node_modules / (เว้นวรรคผิด) → ลบทั้งเครื่อง ความเร็วที่ประหยัดไม่คุ้มเลย
  6. ทำไมเรียนก่อนไป Module 8 — ops คือที่ที่ทุก operation blast radius สูงสุด (แตะ prod จริง) ฝึกหลักคิดที่นี่ พอถึง M8 จะตั้งเป็นโดยไม่ต้องคิดใหม่
Step การลงมือจริง · 1.2
  1. ลิสต์ operation ที่ agent ทำบ่อย แยก read-only vs เปลี่ยนสถานะ
  2. read-only + idempotent (read/grep/test) → allow
  3. เขียน/ลบ/deploy → ask หรือ deny
  4. เขียนลง .claude/settings.json (project) แล้ว commit เข้า repo
  5. ทดสอบ: สั่งงานที่ควรถูก gate ดูว่ามัน ถามจริง ก่อนทำ

1.3

Agent Harness

  1. harness = ทุกอย่างที่ห่อ model — system prompt + tool definitions + context manager + loop controller + sub-agent orchestration · model เปล่าคุยได้อย่างเดียว harness ทำให้มัน "ลงมือทำงานได้"
    เปรียบเทียบmodel = เครื่องยนต์, harness = ตัวรถทั้งคัน (พวงมาลัย เกียร์ ล้อ เบรก) เครื่องยนต์ดีแต่ไม่มีรถ ก็ขับไปไหนไม่ได้
  2. คนละ harness = คนละพฤติกรรม บน model เดียวกัน
    ตัวอย่างClaude Code กับ Cursor เรียก model ตัวเดียวกันได้ แต่ Claude Code เด่น terminal/multi-file · Cursor เด่น inline edit ในเอดิเตอร์ — model เดียวกัน ประสบการณ์คนละแบบ
  3. สมการที่ต้องจำ: ผลลัพธ์ = model × harness × context (คูณ ไม่ใช่บวก) ตัวไหนห่วย พังหมด
    ตัวอย่างmodel ฉลาด 9/10 แต่ป้อน context ผิดไฟล์ (2/10) → ผลที่ได้ใกล้ 2 ไม่ใช่ 9 เพราะมันคูณกัน
  4. วินิจฉัยที่พลาดบ่อย — "model นี้โง่": ส่วนใหญ่คือ harness ป้อน context ผิด/ไม่พอ ก่อนโทษ model เช็ก context + tool ก่อน
    ตัวอย่าง"Claude เขียน component ผิด style ตลอด" → จริง ๆ คือคุณไม่ได้ป้อน design system ให้มันเห็น (context ขาด) ไม่ใช่ model โง่
  5. เทียบเครื่องมือ = เทียบ harness ไม่ใช่เทียบ model — ถามว่า "harness ไหนป้อน context + จัด tool ให้ งานของฉัน ได้ดีสุด"
  6. เชื่อมไป 5.2 MCP — MCP คือมาตรฐานการ "เพิ่ม tool เข้า harness" เพิ่มมั่ว = harness บวม ช้า แพง
Step การลงมือจริง · 1.3
  1. เวลาผลห่วย แยกตัวแปร: model / harness / context อันไหนพลาด
  2. เช็ก context ก่อน — ป้อนครบ/ถูกไฟล์ไหม
  3. เช็ก tool — harness มี tool ที่งานต้องการไหม
  4. ทั้งสองโอเคแล้วยังห่วย → ค่อยพิจารณาเปลี่ยน model/harness
  5. เทียบเครื่องมือจาก งานของฉัน ไม่ใช่ benchmark ทั่วไป

1.4

Cost / Token Management

  1. token มีต้นทุน 3 มิติพร้อมกัน — เงิน + เวลา + คุณภาพ (context ยาว → context rot ความแม่นตก) มิติที่ 3 มองไม่เห็นในบิลแต่สำคัญสุด
    ตัวอย่างคุยยาว 80 ข้อความในแชทเดียวแล้วถามใหม่ — คำตอบเริ่มมั่ว ไม่ใช่ model โง่ลง แต่ context ยาวจนของสำคัญถูกกลบ
  2. context window เต็มแล้วเป็นยังไง — harness ตัดของเก่าทิ้ง ของที่ตัดอาจสำคัญ → agent "ลืม" ตอนต้น
    ตัวอย่างต้นแชทบอก "ห้ามใช้ any ใน TS" คุยยาวไป 50 ข้อความ มันเริ่มใส่ any กลับมา เพราะ instruction นั้นถูกตัดออกจาก context แล้ว
  3. เครื่องมือจัดการ (Claude Code) — /clear (ล้างเริ่มใหม่ ใช้ทุกครั้งที่เปลี่ยนงาน), /compact (ย่อโดยเก็บสาระ), sub-agent (แตกงานหนักไปทำใน context แยก ส่งกลับแค่ผลสรุป)
    ตัวอย่าง sub-agentงาน "อ่าน 30 ไฟล์แล้วสรุป architecture" ถ้าทำใน context หลักจะบวม 30 ไฟล์ → โยนให้ sub-agent อ่าน ส่งกลับแค่ย่อหน้าสรุป context หลักสะอาดเหมือนเดิม
  4. prompt caching — ลดต้นทุนส่วนคงที่ — system prompt, CLAUDE.md, tool definitions ซ้ำทุก turn ถ้า cache ไว้ไม่ต้องจ่ายเต็มทุกรอบ
    ตัวอย่างCLAUDE.md ยาว 3,000 token ถ้าไม่ cache = จ่าย 3,000 ทุก turn · คุย 40 turn = 120,000 token เผาไปกับไฟล์เดิมซ้ำ ๆ
  5. วินัยหลัก: หนึ่งงาน = หนึ่ง context สะอาด — อย่าทำ 5 งานไม่เกี่ยวกันใน session เดียว /clear ระหว่างงานคือนิสัยที่แยก dev เก่ง/ไม่เก่ง
    ตัวอย่างแก้ bug login เสร็จ → จะไปทำ feature wallet ต่อ → /clear ก่อน ไม่งั้น context ของ login ค้างอยู่กวนงาน wallet
  6. เชื่อมไป 3.1 + Module 5 — นี่คือ "ทำไม" เบื้องหลังกฎข้างหน้า: ห้ามยัด CLAUDE.md และห้ามเปิด MCP เยอะ เพราะทั้งคู่กิน context ทุก turn แบบทบต้น
Step การลงมือจริง · 1.4

  5. จบงาน → /clear ก่อนงานถัดไป (อย่าปนงาน)

แผนห่วยที่ execute เป๊ะ ก็ยังคือผลงานห่วย — Planning คือจุดที่ได้ leverage สูงสุดต่อเวลาที่ลงไป เพราะมันกำหนดทิศของทุก turn ที่เหลือ การลงแรงคิดตรงนี้ถูกกว่าการแก้ทีหลังเสมอ

2.1

Research

  1. ให้ agent เป็นนักสำรวจ ไม่ใช่นักตัดสิน — สำรวจ codebase, อ่าน docs, เทียบ approach, รวบรวมข้อเท็จจริง แต่ อย่าเพิ่งให้มันสรุปทางเลือกสุดท้าย ในขั้นนี้
    ตัวอย่าง"หาว่าตอนนี้ xFish fetch data ยังไงบ้าง" → ให้มันไล่ useFetch/$fetch/Pinia action ทั้งโปรเจกต์แล้วทำตารางสรุป pattern ที่เจอ — ยังไม่ต้องบอกว่าควรใช้อันไหน
  2. ใช้ Plan Mode แยก phase วิจัยออกจาก phase ลงมือ — ใน Claude Code, Plan Mode บังคับให้มัน "อ่าน + เสนอแผน" โดยห้ามแก้ไฟล์ จนกว่าคุณจะอนุมัติ
    ตัวอย่างกด Shift+Tab เข้า Plan Mode → มันสำรวจแล้วเสนอแผนเป็นข้อ ๆ → คุณอ่าน เห็นว่ามันเข้าใจผิดตรงไหน แก้ก่อน → ค่อยให้ลงมือ (ประหยัดกว่าปล่อยให้มันแก้มั่วแล้ว revert)
  3. Verify ทุกข้อสรุป — อย่าเชื่อทันที agent มั่นใจเท่ากันทั้งตอนถูกและผิด (รากฐานจาก M1) ข้อสรุปจาก research ที่ไม่ตรวจ = ฐานทราย ที่ทุกอย่างข้างบนจะพังตาม
    ตัวอย่างมันบอก "โปรเจกต์ใช้ axios ทั้งหมด" → คุณ grep เองเร็ว ๆ พบว่าครึ่งหนึ่งใช้ $fetch ของ Nuxt อยู่ ถ้าเชื่อทันทีจะวางแผนผิดทั้งสาย
  4. research ที่ดีลด hallucination ของขั้นถัดไป — ยิ่งมันรู้ของจริงในโปรเจกต์มากเท่าไหร่ ตอน decide/implement มันยิ่งเดาน้อยลง

2.2

Decide — ทางเลือก + tradeoff

  1. บังคับรูปแบบ output: "ตัวเลือก + ข้อดี/ข้อเสีย + คำแนะนำ" ไม่ใช่ให้มันเลือกเสร็จสรรพแล้วลงมือเลย ขอเป็นตาราง 2–3 ทางเลือกพร้อม tradeoff
    ตัวอย่าง"data fetching ใน xFish ควรใช้อะไร" → ขอตาราง: useFetch (SSR-friendly, cache อัตโนมัติ / ยืดหยุ่นน้อย) vs useAsyncData + $fetch (คุมเอง / โค้ดเยอะ) vs Pinia action (state รวมศูนย์ / boilerplate) → แล้วคุณตัด
  2. คุณคือคนตัดสิน agent คือคนเสนอ — architecture decision ผูกพันยาวและแก้ทีหลังแพง มันต้องเป็นการตัดสินใจของมนุษย์ที่รับผิดชอบผลลัพธ์ ไม่ใช่ค่า default ของ model
    ตัวอย่างความเสี่ยงปล่อยให้มันเลือก state management เงียบ ๆ มันอาจหยิบ pattern ที่นิยมใน training data แต่ไม่เข้ากับ Nuxt 4 + Pinia ที่คุณวางไว้ → drift ทั้งโปรเจกต์
  3. อย่าปล่อยให้ "การตัดสินใจเงียบ" หลุดผ่าน — ระวัง decision ที่ฝังมาในโค้ดโดยไม่เคยถูกพูดถึง (เลือก library, ตั้งชื่อ pattern, โครง folder) ให้มัน ประกาศ decision ออกมาก่อนเขียน
  4. Lock decision ที่ตัดแล้วลงเป็นลายลักษณ์ — พอตัดสินใจแล้ว เขียนลง ticket หรือ CLAUDE.md เพื่อไม่ให้ session หน้าตัดสินใจใหม่ขัดของเดิม (เชื่อม 3.2 Memory)

2.3

Breakdown

  1. หน่วยมาตรฐาน: "หนึ่ง session = หนึ่ง PR ได้" — แตก feature ให้แต่ละ task จบได้ใน context เดียวและรีวิวเป็น PR เดียวที่ไม่ใหญ่เกินอ่านไหว
    ตัวอย่าง"ทำระบบ wallet" ใหญ่เกินไป → แตกเป็น: (1) Bruno spec + type ของ wallet API, (2) endpoint GET /wallet/balance + test, (3) component แสดง balance, (4) flow เติมเงิน · แต่ละอัน = 1 PR
  2. ใหญ่เกิน 1 PR = สัญญาณต้องแตกอีก — ถ้า task หนึ่งจะแตะ 15 ไฟล์หรือคุยยาวจน context ใกล้เต็ม นั่นคือ task ยังหยาบไป กลับไปซอย
    ตัวอย่างtask "refactor auth ทั้งระบบ" → ถ้าทำรวดเดียว diff มหาศาล รีวิวไม่ไหว → ซอยเป็น migrate ทีละ guard / ทีละ route group
  3. task เล็ก = verify ตรงจุด + context ไม่บวม (โยงกลับ 1.4) แต่ละ task มี surface เล็ก คุณตรวจได้ครบ และ agent ไม่ต้องแบก context ของทั้ง feature พร้อมกัน
  4. เรียงลำดับ dependency ระหว่าง task — อันไหนต้องเสร็จก่อน (เช่น contract/type ก่อน implementation) จัดคิวให้ถูก ไม่งั้น task หลังจะติดเพราะของหน้ายังไม่มี
Step การลงมือจริง · 2.3
  1. ดู feature ว่าใหญ่แค่ไหน
  2. ซอยเป็น task ขนาด 1 session = 1 PR
  3. ถ้า task แตะ >10 ไฟล์ หรือ context ใกล้เต็ม → ซอยอีก
  4. เรียง dependency (contract/type ก่อน implementation)
  5. ตรวจว่าแต่ละ task verify ได้จบในตัว

2.4

Ticket / Issue

  1. spec คือ prompt แบบใหม่ — ticket ที่คมคือ context ก้อนที่ดีที่สุดที่คุณป้อนให้ agent ได้ ยิ่งชัด agent ยิ่งหลุดน้อย คุณภาพ ticket = ขอบเขตที่มันจะไม่เพ้อ
  2. องค์ประกอบที่ ticket ที่ดีต้องมี — (ก) เป้าหมาย/บริบทสั้น ๆ, (ข) acceptance criteria ที่ทดสอบได้, (ค) ไฟล์/ขอบเขตที่เกี่ยวข้อง, (ง) ลิงก์ context (เช่น Bruno spec), (จ) สิ่งที่ ไม่ ต้องทำ (out of scope)
    ตัวอย่าง ticketเป้า: แสดง wallet balance ในหน้า profile · AC: เรียก GET /wallet/balance, แสดงเป็น THB, loading state, error toast เมื่อ fail · ไฟล์: pages/profile/, composables/useWallet.ts · contract: bruno/wallet/balance.bru · out of scope: การเติมเงิน
  3. acceptance criteria = เกณฑ์ verify ของคุณและ test ของ agent — AC ที่เขียนเป็น behavior ชัด ๆ จะกลายเป็น test case ตรง ๆ ในภายหลัง (เชื่อม 6.1)
  4. link context แทนการ copy — อ้างถึง Bruno spec / ไฟล์จริง ดีกว่าวางเนื้อยาว ๆ ลง ticket เพราะของจริงไม่ outdated และ agent ไปอ่านสด ๆ ได้
Step การลงมือจริง · 2.4
  1. เขียนเป้าหมายสั้น ๆ
  2. ใส่ acceptance criteria ที่ทดสอบได้
  3. ระบุไฟล์/ขอบเขตที่เกี่ยว
  4. link context (Bruno spec/ไฟล์จริง) แทน copy
  5. ระบุ out-of-scope กัน scope creep

3.1

  1. มันคือ context ที่ถูก inject ทุก turn อัตโนมัติ — agent รู้เสมอโดยไม่ต้องสั่ง แต่ก็จ่าย token เสมอ ทั้งจุดแข็งและจุดอ่อนอยู่ในข้อเดียวกัน
    ตัวอย่าง ทั่วไปเขียน "โปรเจกต์นี้ใช้ pnpm ไม่ใช่ npm" → ทุก session agent ใช้ pnpm install ถูก ไม่เผลอพิมพ์ npm xFishเขียน "~/ alias resolve ไป root" → ทุก session รู้ทันทีไม่ต้องบอกซ้ำ
  2. เกณฑ์คัดเข้า: เฉพาะของที่ "ต้องรู้ทุกครั้ง" — (ก) คำสั่งหลัก build/test/lint/dev, (ข) conventions ที่ ฝืน default, (ค) gotcha เฉพาะโปรเจกต์, (ง) โครงสร้าง/alias ที่ไม่ standard
    ตัวอย่าง ทั่วไป"รัน test ด้วย make test ไม่ใช่ npm test" / "ห้าม edit ไฟล์ใน generated/ เพราะ auto-gen ทับ" xFishflat root-level dir + page-local components ใต้ pages/<feature>/components/ ต้อง import explicit — ฝืน default ของ Nuxt
  3. เกณฑ์คัดออก: ของที่ใช้ "เป็นบางครั้ง" → ไปเป็น Skill — ถ้ายัดลง CLAUDE.md = จ่าย token ฟรีเกือบตลอดเวลา
    ตัวอย่าง ทั่วไปplaybook "วิธี rotate API key ตอน production" ใช้นาน ๆ ที → เป็น Skill xFish"วิธี migrate schema" ใช้เดือนละครั้ง → Skill โหลดตอนจะ migrate จริง
  4. ต้องตรงกับโปรเจกต์จริงเป๊ะ — mismatch อันตรายกว่าไม่มี agent เชื่อไฟล์แล้ววางของผิดที่อย่างมั่นใจ
    ตัวอย่าง ทั่วไปไฟล์บอก "test อยู่ใน __tests__/" แต่ทีมย้ายไป tests/ แล้ว → agent สร้าง test ในโฟลเดอร์ผี xFishอ้าง path เก่าที่ refactor ไปแล้ว → agent สร้างไฟล์ตาม path ผี ต้องไล่แก้ทีหลัง
  5. AGENTS.md = มาตรฐานกลางข้ามเครื่องมือ — CLAUDE.md เป็นของ Claude Code, AGENTS.md tool อื่นอ่านได้ (Codex, Cursor) วางของกลางไว้ที่เดียว
    ตัวอย่าง ทั่วไปทีมใช้ทั้ง Cursor + Claude Code → กฎหลักไว้ AGENTS.md ทั้งคู่อ่าน source เดียวกัน ไม่ maintain ซ้ำ xFishCLAUDE.md มีบรรทัด "ดู conventions หลักใน AGENTS.md" → Claude Code + Codex CLI อ่านที่เดียวกัน

  2. decision สำคัญต้อง "ตกตะกอน" ลง persistent เสมอ — อะไรที่ใช้ข้ามวัน/ข้ามคน ย้ายจาก session → project memory
    ตัวอย่าง ทั่วไปตัดสินใจ "error handling ใช้ Result type ไม่ throw" → จดลง CLAUDE.md 1 บรรทัด ทุก session ทำตาม xFishหลัง decide ใช้ useFetch เป็นมาตรฐาน → จดลง CLAUDE.md ไม่ปล่อยลอยในแชท

  4. memory มีลำดับชั้น — global (ทุกโปรเจกต์) → project (repo นี้) → local (เครื่องคุณ ไม่ commit)
    ตัวอย่าง ทั่วไป"อธิบายโค้ดสั้น ๆ" = global preference · "repo นี้ใช้ TS strict" = project · token ส่วนตัว = local (.gitignore) xFish"ตอบเป็นภาษาไทย" = global · "xFish ใช้ Bruno เป็น contract" = project

3.3

Design System

  1. ป้อนสิ่งที่มีอยู่ก่อนให้สร้างใหม่ — config (สี/spacing/breakpoint) + component ที่มี = "คำศัพท์" ที่ agent ต้องใช้พูด ไม่งั้นมันประดิษฐ์คำศัพท์ใหม่เอง
    ตัวอย่าง ทั่วไปมี <Button> กลางแต่ไม่บอก → agent สร้าง <MyButton> ใหม่ด้วย inline style คนละโทน หน้าเว็บมีปุ่ม 3 แบบ xFishมี <BaseButton> แต่ไม่ชี้ → component ซ้ำสไตล์ใหม่
  2. บอกทั้ง "มีอะไร" และ "ใช้ยังไง" — pattern การใช้: เมื่อไหร่ใช้ตัวไหน, prop convention, variant
    ตัวอย่าง ทั่วไป"หัวข้อใช้ <Heading level={1-6}> ห้ามใช้ <h1> ดิบ" → agent หยิบของกลางถูก xFish"ปุ่มหลัก <BaseButton variant='primary'> · ปุ่มรอง variant='ghost' · อย่าสร้างเอง"
  3. ถ้าไม่ป้อน → codebase แตกเป็นเสี่ยง (fragmentation) — แต่ละ session เดาสไตล์ใหม่ component ซ้ำซ้อน
    ตัวอย่างทั่วไป5 หน้าใช้ card 5 แบบ padding/radius/shadow ไม่ตรงกัน เพราะแต่ละครั้งเริ่มจากศูนย์
  4. design system ที่ดีลด decision ต่อ component — agent ไม่ต้อง "ตัดสินใจดีไซน์" ทุกครั้ง แค่ประกอบของที่มี → เร็วขึ้นและ consistent
    ตัวอย่างทั่วไป"ทุก spacing ใช้ scale (gap-4,p-6) ห้าม magic number เช่น padding:13px" → เป็นระเบียบทันที

3.4

API Design — contract-first

  1. contract คือ source of truth ที่ทั้งสองฝั่งยึด — FE กับ BE คุยผ่าน contract ตัวกลาง agent อ่าน spec ก่อนเขียนทั้งสองฝั่ง
    ตัวอย่าง ทั่วไปมี OpenAPI/Swagger spec → agent generate client type ฝั่ง FE และ validate ฝั่ง BE จาก schema เดียวกัน xFishbruno/wallet/balance.bru ระบุ response { balance, currency } → generate TS type FE + handler BE จาก type เดียว
  2. contract-first ลด FE/BE drift ที่ต้นเหตุ — ถ้าทั้งคู่ derive จาก contract เดียว มันแยกจากกันไม่ได้
    ตัวอย่างทั่วไปBE เปลี่ยน field created_at → createdAt แต่ FE ไม่รู้ → ถ้ายึด contract การเปลี่ยนต้องแก้ spec ก่อน ทั้งคู่เห็นพร้อมกัน
  3. อ่าน spec เป็น "สัญญา" ไม่ใช่ "ตัวอย่าง" — field name, type, optional/required ต้องตรงเป๊ะ
    ตัวอย่างทั่วไปspec บอก userId (camelCase) → agent ห้าม generate user_id เองตามนิสัย
  4. contract เป็นเสาค้ำหลายหัวข้อพร้อมกัน — ป้อน 4.1 (FE type), 4.2 (BE handler), 6.1 (test ตาม contract), 8.3 (auto-doc จาก spec)
    ตัวอย่างทั่วไปtest เขียน assert response ตาม schema → implementation หลุดจาก spec test แดงทันที = enforce contract อัตโนมัติ
  5. เปลี่ยน contract = เปลี่ยนที่ spec ก่อนเสมอ — แก้ spec → regenerate type → แก้ implementation สองฝั่ง · spec เน่าเมื่อไหร่ ทุกอย่างที่ยึดมันพังตาม

4.3

Error / Log / Context Handling

  1. feed stack trace / log กลับให้ agent อ่าน — แทนที่จะอธิบาย error เอง วาง trace จริงให้มันวิเคราะห์ มันอ่าน log ได้ดี
    ตัวอย่างทั่วไปcopy error เต็ม ๆ (ไม่ใช่แค่ "มัน error") วางให้ agent → มันชี้บรรทัดและสาเหตุได้ตรงกว่า
  2. ระวัง fix-the-symptom loop (เชื่อม 1.1) — agent แก้ให้ error หายทีละอันโดยไม่แตะ root cause สร้างบั๊กใหม่แทนของเก่า
    ตัวอย่าง ทั่วไปใส่ try/catch กลบ error ไปเรื่อย ๆ ทั้งที่ root cause คือ config ผิดตัวเดียว xFishใส่ ?. ไล่ทีละ field ทั้งที่ API ไม่ได้ return object นั้นตั้งแต่แรก
  3. กฎ 2–3 รอบ: วนแก้ไม่จบ → หยุด คุณเข้าไปอ่านเอง ถ้า agent วนแก้ error เดิม/ใกล้เคียงเกิน 2–3 รอบ นั่นคือสัญญาณว่ามันมองไม่เห็น context บางอย่าง

  4. บางทีปัญหาอยู่ที่ context ที่ agent มองไม่เห็น — error จริงอาจอยู่นอกไฟล์ที่มันอ่าน (config, env, service อื่น, state ภายนอก) งานของคุณคือเติม context ที่ขาด ไม่ใช่ให้มันเดาต่อ

5.2

MCP

  1. MCP = มาตรฐานต่อ "ระบบนอก" เข้า harness — ให้ agent คุยกับ DB, browser, external API ผ่าน protocol กลาง (เชื่อม 1.3: MCP คือวิธีเพิ่ม tool เข้า harness)
    ตัวอย่าง ทั่วไปต่อ MCP ของ GitHub → agent อ่าน issue/PR ได้ · ต่อ MCP database → query ข้อมูลจริงได้ xFishPlaywright MCP → agent ขับ browser ทดสอบ UI จริงได้ (ใช้ใน Module 7)
  2. ทุก tool definition กิน context "ทุก turn" ไม่ว่าจะใช้หรือไม่ — หัวใจของ Module นี้ MCP ที่เปิดไว้ inject tool schema เข้า context ตลอด แม้ turn นั้นไม่ได้เรียกใช้
    ตัวอย่างทั่วไปเปิด MCP 8 ตัว แต่ละตัวมี 10 tool → 80 tool schema กิน context ทุก turn ทั้งที่ใช้จริง 2-3 ตัว
  3. อย่าเปิดทุกอย่างพร้อมกัน — เลือก lean stack — tool ที่ทับ function กันและไม่ได้ใช้ = จ่าย token ฟรี + ทำให้ agent สับสนว่าจะใช้ตัวไหน
    ตัวอย่างxFishอย่าเปิด Playwright + Chrome DevTools + Perplexity + Firecrawl + Glif พร้อมกัน — Playwright/Chrome DevTools ทับกัน, search tool ซ้อนกัน → เลือก lean: Playwright (accessibility-tree) + search 1 ตัว + docs tooling พอ
  4. เลือก MCP จาก "งานที่ทำจริง" ไม่ใช่ "เผื่อไว้" — ถามว่า workflow นี้ต้องแตะระบบนอกอะไร แล้วเปิดเท่านั้น เปิด-ปิดตามช่วงงานได้
    ตัวอย่างทั่วไปช่วงทำ E2E → เปิด Playwright MCP · ช่วงเขียน logic ธรรมดา → ปิดไป ประหยัด context
  5. คุณภาพ MCP ต่างกันมาก — เลือกตัวที่ออกแบบ tool ดี — MCP ที่ดีมี granularity เหมาะ, description ชัด, ไม่คืน output ท่วม context ตัวห่วยจะคืน data มหาศาลกินที่
    ตัวอย่างxFishเลือก Playwright แบบ accessibility-tree เพราะคืน structure ที่ agent อ่านเข้าใจ ทนต่อ DOM เปลี่ยน มากกว่า pixel selector

Testing & Code Review

ต้องรู้ก่อน:Module 4 — คุณ test สิ่งที่ implement ขยายเต็ม
หลักคิดของ Module นี้

ของที่ agent เขียนต้อง review มากขึ้น ไม่ใช่น้อยลง — code จาก agent "ดูน่าเชื่อ" ซึ่ง อันตรายกว่า code ที่ดูพัง เพราะมันลื่นผ่านสายตา

6.1

Test Writing

  1. สั่งให้เขียน test จาก behavior/spec ไม่ใช่จาก code ที่เพิ่งเขียน — test ต้องสะท้อน "สิ่งที่ควรเป็น" (จาก acceptance criteria ใน 2.4 / contract ใน 3.4) ไม่ใช่ "สิ่งที่ code ทำตอนนี้"
    ตัวอย่าง ทั่วไป"เขียน test จาก AC: ถ้า input ว่างต้องคืน 400" → test มาจาก spec · ไม่ใช่ "อ่าน code แล้วเขียน test ให้ผ่าน" xFishtest assert ตาม Bruno spec ของ /wallet/balance → implementation หลุดจาก spec test แดงทันที
  2. กับดักใหญ่: tautological test (test ที่พิสูจน์ตัวเอง) — ถ้า agent เขียน test จาก implementation, implementation พัง test พังตาม → จับ bug ไม่ได้เลย เพราะมันแค่สะท้อนโค้ดกลับมา
    ตัวอย่างทั่วไปโค้ดบวกผิดเป็น a-b แล้ว agent เขียน test expect(sub(2,3)).toBe(-1) ตามโค้ด → test เขียวแต่ฟังก์ชัน "sum" ยังผิด
  3. ระบุ "จะ test อะไร" — agent เขียนกลไกได้ แต่ไม่รู้ว่าอะไรสำคัญ — คุณกำหนด case ที่ต้องครอบคลุม (happy path + edge + error) มันเขียนโครง test ให้
    ตัวอย่างทั่วไป"test: ค่าปกติ / ค่าติดลบ / 0 / ค่าเกิน limit / null" → agent gen ครบทุก case ที่คุณสั่ง
  4. ใช้ Skill ช่วยให้ test เข้า convention (เชื่อม 5.1) — Skill write-tests บรรจุ pattern/framework ของทีม → test ที่ออกมากลมกลืน ไม่ใช่สไตล์มั่ว
  5. test ที่ดี = ตาข่ายที่จับ regression ในอนาคต — ไม่ใช่แค่ผ่านวันนี้ แต่ต้องแดงเมื่อมีคน (หรือ agent) แก้อะไรพังในภายหลัง

6.2

Code Review

  1. ใช้ agent เป็น first-pass reviewer — ให้มันรีวิวรอบแรกจับของง่าย ๆ ก่อน (naming, unused, obvious bug, style) ด้วย Skill review-changes/review
    ตัวอย่าง ทั่วไปก่อนเปิด PR ให้คน → รัน agent review จับ console.log ที่ลืม, ตัวแปรไม่ได้ใช้, typo ก่อน xFishSkill review-changes เช็ค diff ตาม convention (double quotes, no semicolons, explicit imports) ก่อนถึงมือ reviewer
  2. มันคือ first pass ไม่ใช่ final — agent จับ low-hanging fruit ให้ เพื่อให้ คุณ เอาเวลาไป focus เรื่องยาก (logic, architecture, security จริง)
    ตัวอย่างทั่วไปagent ผ่านทุกอย่าง แต่คุณยังต้องดูเอง: "logic นี้ race condition ได้ไหม" — อันที่ agent มองไม่ออก
  3. agent review ของ agent มี blind spot ร่วมกัน — code ที่ agent เขียน + agent review อาจพลาดจุดเดียวกัน เพราะ reasoning คล้ายกัน อย่าวางใจ 100%
    ตัวอย่างทั่วไปagent เขียนโค้ดที่ "ดูสมเหตุสมผล" แล้ว agent review ก็เห็นว่า "สมเหตุสมผล" → ทั้งคู่พลาด assumption ผิดอันเดียวกัน
  4. ให้ review ทีละ diff เล็ก (เชื่อม 4.x) — PR เล็กที่ review บ่อย ได้ผลกว่ารีวิว diff มหึมาทีเดียว ทั้งคนและ agent

6.3

Security

  1. เช็คพื้นฐาน 3 อย่างก่อนเสมอ — (ก) secret หลุด (API key/token ใน code), (ข) injection (SQL/command/XSS), (ค) authz check (ตรวจสิทธิ์ก่อนทำ action)
    ตัวอย่าง ทั่วไปagent hardcode API key ลงไฟล์ตอน prototype → ต้องจับก่อน commit · query ต่อ string ดิบ → SQL injection xFishเช็คว่าทุก mutation มี DoAdmin/DoIp ตาม convention — authz ไม่หลุด
  2. first-pass อีกชั้น ไม่แทน security review จริง — agent กันพลาดโง่ ๆ ได้เยอะ แต่ไม่ใช่ penetration test การตัดสินเรื่อง security ขั้นจริงยังต้องมนุษย์/เครื่องมือเฉพาะ
  3. agent อาจสร้างช่องโหว่เองโดยไม่ตั้งใจ — เช่น เปิด CORS กว้างไป, log ข้อมูล sensitive, ปิด validation เพื่อให้ test ผ่าน ต้องตรวจสิ่งที่มัน "แก้ให้ง่าย"
    ตัวอย่างทั่วไปagent แก้ error CORS ด้วยการตั้ง origin: "*" → error หายแต่เปิดช่องโหว่ ต้องจับ
  4. ใส่ security check เข้า quality gate (ปูทาง 6.4) — secret scanning, dependency audit ควรเป็น automated ไม่ใช่พึ่งสายตา

6.4

Quality Gate

  1. นิยาม merge criteria ให้ชัด แล้ว automate — มาตรฐานขั้นต่ำที่ต้องผ่านก่อน merge: lint + typecheck + test (อย่างน้อย) ทำเป็น CI ไม่ใช่ความจำคน
    ตัวอย่าง ทั่วไปGitHub Actions: PR ต้องผ่าน lint + test + build สำเร็จ ถึง merge ได้ ไม่ผ่าน = block อัตโนมัติ xFishเพิ่ม typecheck (TypeScript strict mode ที่เปิดอยู่) เข้า gate → type ผิดหลุดไป prod ไม่ได้
  2. gate คือตาข่ายที่ "ไม่ขึ้นกับอารมณ์คนรีวิว" — วันที่ reviewer เหนื่อย/รีบ มาตรฐานคงเดิม เพราะเครื่องตรวจ ไม่ใช่คน
    ตัวอย่างทั่วไปcode ที่ "ดูน่าเชื่อ" ของ agent อาจลื่นผ่านสายตาคน แต่ถ้า typecheck แดง มันผ่าน gate ไม่ได้แน่นอน
  3. ทำไม gate สำคัญกับ AI เป็นพิเศษ — code จาก agent ดูเรียบร้อยน่าเชื่อ จึงอันตรายกว่า code ที่ดูพัง gate คือด่านที่ไม่หลงเชื่อ "ความดูดี"
  4. gate รวมผลของ 6.1-6.3 เป็นด่านเดียว — test (6.1) + lint/typecheck + security check (6.3) มารวมเป็นเงื่อนไข merge เดียว
  5. เป็น prerequisite ของ Deploy (ปูทาง Module 8) — ของที่จะ deploy ต้องผ่าน gate ก่อนเสมอ ไม่มีทางลัด

ต้องรู้ก่อน:Module 6Module 5 — จุดที่ agent โดดเด่นจริง ขยายเต็ม

นี่คือชั้น test ที่ เห็นเหมือน user เห็น — ต่างจาก unit test (6.1) ที่ทดสอบชิ้นส่วน E2E ทดสอบ "ทั้ง flow ทำงานจริงไหมในเบราว์เซอร์จริง" เป็นจุดที่ agent โดดเด่นมาก เพราะงาน tedious แต่ well-defined แต่มีหลุมพรางชื่อ flakiness ที่ต้องกันตั้งแต่แรก

7.1

Playwright E2E

  1. E2E = ทดสอบทั้ง flow จากมุม user ไม่ใช่ทีละฟังก์ชัน — เปิดหน้าจริง คลิกจริง กรอกจริง แล้ว assert ว่าผลที่ user เห็นถูกต้อง (ต่อยอดจาก unit test ใน 6.1 ขึ้นมาอีกชั้น)
    ตัวอย่าง ทั่วไปtest "login → ไป dashboard → เห็นชื่อตัวเอง" ครบทั้ง flow ไม่ใช่แค่ test validatePassword() xFishtest "เปิด profile → balance โหลด → กดเติมเงิน → เห็นยอดอัปเดต" เป็น flow เดียวต่อกัน
  2. accessibility-tree approach ทนกว่า pixel/selector — Playwright MCP ให้ agent "เห็น" หน้าเป็น structure (role, label, text) แทนพิกัด/CSS selector เปราะ ๆ → ทนต่อ DOM ที่เปลี่ยน
    ตัวอย่าง ทั่วไปlocate ด้วย getByRole("button",{name:"Submit"}) แทน .btn-3 > div:nth-child(2) ที่พังทันทีที่ layout เปลี่ยน xFishหาปุ่ม "เติมเงิน" ด้วย role+ชื่อ → refactor UI แล้ว test ยังทำงาน
  3. agent เหมาะกับงานนี้มาก — tedious แต่ well-defined — การไล่เขียน locator, รอ element, assert ทีละขั้น เป็นงานซ้ำที่มีกฎชัด agent ทำได้เร็วและครบกว่าคนเขียนมือ
  4. กับดักหลัก: implicit wait / race condition — สาเหตุ flaky อันดับหนึ่ง agent มักลืมรอ async ที่ถูกต้อง ใช้ auto-waiting / web-first assertion แทน sleep() ตายตัว
    ตัวอย่างทั่วไปawait expect(el).toBeVisible() (รอจนเห็นจริง) แทน await sleep(2000) ที่บางเครื่องเร็ว/ช้าไม่เท่ากัน
  5. ใช้ Playwright MCP ขับจริงได้ในขั้น dev (เชื่อม 5.2) — agent สั่ง browser ผ่าน MCP เพื่อ "ลองเอง" ก่อนเขียนเป็น test ถาวร — สำรวจหน้า เห็น accessibility tree แล้วค่อย gen test code

Visual Regression

  1. จับสิ่งที่ assertion จับไม่ได้: หน้าตาเพี้ยน — E2E (7.1) เช็ค "ทำงานถูกไหม" แต่ visual regression เช็ค "หน้าตาถูกไหม" — layout เลื่อน, สีเพี้ยน, element ซ้อน ที่ logic ยังผ่าน

  2. กลไก: snapshot baseline → เทียบภาพรอบใหม่ → diff — เก็บภาพ "ที่ถูกต้อง" ไว้เป็น baseline รอบต่อไปถ่ายใหม่แล้วเทียบ ต่างเกิน threshold = แดง
    ตัวอย่างทั่วไปawait expect(page).toHaveScreenshot() — Playwright เก็บ baseline ครั้งแรก รอบหลังเทียบอัตโนมัติ
  3. กับดัก: false positive จาก dynamic content — เวลา, avatar สุ่ม, animation, ฟอนต์โหลดช้า ทำให้ภาพต่างทั้งที่ไม่ได้พัง ต้อง mask/freeze ส่วนที่ dynamic

  5. เริ่มจากหน้า/component สำคัญก่อน — อย่าถ่ายทุกหน้าตั้งแต่แรก เลือกหน้าที่ regression แล้วเจ็บที่สุด (หน้าจ่ายเงิน, หน้าหลัก) ทำก่อน

  1. ให้ agent สรุปผลรันเป็น report ที่อ่านแล้วตัดสินใจได้ — ไม่ใช่ dump log ดิบ แต่สรุป: ผ่านกี่ / แดงกี่ / แดงเพราะอะไร / อันไหนน่าจะ flaky
    ตัวอย่าง ทั่วไป"20 เทสต์: ผ่าน 18, แดง 2 — ทั้งคู่ที่ checkout เพราะ timeout น่าจะ flaky" อ่านปุ๊บรู้ว่าต้องดูตรงไหน xFishreport flow wallet: ขั้นไหนพัง, screenshot ตอนแดง, น่าจะ env หรือ logic
  2. flaky test คือหลุมพรางที่ทำลายทั้ง suite — test ที่ fail แบบสุ่ม (เครื่องช้า/race) ทำให้ทีม "ชินกับสีแดง" แล้วเลิกเชื่อ suite ทั้งชุด สุดท้ายไม่มีใครดูผล test เลย
    ตัวอย่างทั่วไปtest แดงสุ่ม 1 ใน 10 รอบ → คนเริ่มกด re-run จนเขียวแล้ว merge → วันหนึ่ง bug จริงหลุดเพราะไม่มีใครเชื่อสีแดง
  3. ลงทุนทำให้ deterministic ตั้งแต่แรก คุ้มกว่าตามแก้ทีหลัง — ตามรากจาก 7.1 (race/wait) + 7.2 (dynamic content) flaky ส่วนใหญ่มาจากสองจุดนี้ แก้ที่ต้นเหตุ ไม่ใช่ retry กลบ
    ตัวอย่างทั่วไปแทนที่จะตั้ง retry: 3 กลบ flaky → ไปแก้ web-first assertion ให้รอถูกจังหวะ test เสถียรจริง
  4. report ควรชี้ "flaky" แยกจาก "พังจริง" — สองอย่างนี้ action ต่างกัน: พังจริง = แก้ code, flaky = แก้ test ถ้า report ปนกัน ทีมจะ prioritize ผิด
  5. ผูก report เข้า CI ให้เห็นใน PR — สรุปผลโผล่ใน PR เลย (เชื่อม quality gate 6.4) reviewer เห็นทันทีว่าควร merge ไหม

8.1

Deployment

  1. agent ช่วยร่าง config ได้ดี — แต่ค่ากระทบ cost/stability ตรง ๆ — Dockerfile, CI/CD pipeline, service config (memory, concurrency, min/max instances) agent ร่างโครงให้เร็ว แต่ตัวเลขพวกนี้คุณต้องตัด
    ตัวอย่าง ทั่วไปagent ตั้ง min instances: 5 ให้ "เผื่อ" → บิลพุ่งทั้งที่ traffic จริงต้องการ 1 xFish (Cloud Run)memory/concurrency/min instances กระทบทั้ง cold start และค่าใช้จ่าย → agent เสนอ คุณเคาะ
  2. config ผิด = บิลพุ่ง หรือ service ล่ม — นี่คือเหตุผลที่ deploy ต้องผ่าน Quality Gate (6.4) ก่อนเสมอ ของที่ยังไม่ผ่าน test/typecheck ห้ามถึงขั้นนี้
  3. permission เข้มสุดอยู่ที่นี่ (เชื่อม 1.2) — คำสั่ง deploy/ลบ resource จริงต้อง ask เสมอ ห้าม auto-approve เด็ดขาด เพราะ blast radius = ทั้งระบบ production
    ตัวอย่างทั่วไปdeny/ask สำหรับ deploy:prod, terraform apply, ลบ database — agent เตรียมคำสั่งให้ แต่คนกด enter
  4. แยก staging ก่อน production เสมอ — ให้ agent deploy ขึ้น staging ทดสอบก่อน ค่อยเลื่อนขึ้น prod ลด blast radius ด้วยการมีด่านกลาง
  5. เตรียม rollback ไว้ก่อน deploy — รู้วิธีถอยกลับก่อนกดไป ไม่ใช่ไปหาตอนพัง agent ช่วยร่าง rollback plan / เก็บ version เก่าไว้ได้

8.2

Routine / Scheduler

  1. งาน recurring กฎตายตัว = สวรรค์ของ agent — cleanup, sync, report ประจำ, backup เป็นงานซ้ำที่มีกฎชัด agent เขียน job + ตั้ง schedule ได้ดีมาก
    ตัวอย่าง ทั่วไปcron job ลบ log เก่าเกิน 30 วัน ทุกเที่ยงคืน / sync ข้อมูลจาก external API ทุกชั่วโมง xFishCloud Scheduler + job สำหรับ report ยอดประจำวัน / cleanup session หมดอายุ
  2. job ต้อง idempotent — รันซ้ำต้องปลอดภัย — scheduler อาจ trigger ซ้ำ (retry, overlap) job ที่รันซ้ำแล้วพังหรือทำงานซ้อน = หายนะเงียบ
    ตัวอย่างทั่วไปjob ส่งอีเมลแจ้งเตือน ถ้าไม่ idempotent → trigger ซ้ำ ส่งอีเมลซ้ำ 2 รอบ ผู้ใช้รำคาญ
  3. ต้องมี observability — job เงียบที่พังคืออันตราย — job ที่รันเบื้องหลังพังโดยไม่มีใครรู้ จะสะสมปัญหา ต้อง log + alert เมื่อ fail
    ตัวอย่างทั่วไปbackup job พังเงียบ 2 สัปดาห์ → วันที่ต้องใช้ backup ถึงรู้ว่าไม่มี → ต้อง alert ทันทีที่ fail
  4. ต้อง deploy ของก่อนถึงตั้ง schedule (ต่อ 8.1) — job ต้องมีที่รัน (service/container) ก่อน ลำดับคือ deploy → แล้วค่อยผูก scheduler
