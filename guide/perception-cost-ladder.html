pedpro::observe

วิศวกรรมการรับรู้ของ AI Agent

บันไดต้นทุนการรับรู้

Browser ไม่ได้แพงเพราะเป็น browser — มันแพงตอนที่ AI ใช้ตา. กุญแจคือให้โค้ดกลั่นข้อมูลก่อน แล้วค่อยปีนขึ้นใช้ภาพเฉพาะตอนจำเป็นจริง ๆ

หลักการ: parallelize ข้อมูล · serialize สายตา code > text > pixels

01 — วิธีที่ AI มองเว็บ

AI มองหน้าเว็บได้ 3 แบบ

แต่ละแบบ “เปลือง” ต่างกันมาก เพราะค่าใช้จ่ายคิดตามสิ่งที่เข้า context window — ไม่ใช่ตามการสั่ง browser

แบบที่ 1 · ภาพ

มองด้วยภาพ

ถ่าย screenshot ส่งให้ AI ดูเหมือนคนมองจอ เห็นทุกอย่าง แต่ภาพ 1 รูปกิน token มหาศาล และต้องถ่ายใหม่ทุก step

ต้นทุน: สูงสุด
แบบที่ 2 · โครงสร้าง

อ่าน accessibility tree

ดึงโครงสร้างหน้าเป็นข้อความ (ปุ่ม ลิงก์ ช่องกรอก) นี่คือ default ของ Playwright — เป็น text จึงถูกกว่าภาพมาก แต่หน้า Naive UI โครงสร้างบาน ยังเปลืองได้

ต้นทุน: ปานกลาง
แบบที่ 3 · โค้ด

เขียนโค้ดดึงเฉพาะจุด

สั่งโค้ดไปหยิบเฉพาะค่าที่ต้องการ คืนกลับมาแค่ผลลัพธ์สะอาด ๆ สิ่งที่เข้า context จึงเหลือแค่คำตอบที่กลั่นแล้ว — ถูกที่สุด

ต้นทุน: ต่ำสุด

02 — บันได 5 ขั้น

ปีนจากถูกไปแพง — คลิกแต่ละขั้น

กฎเหล็ก: หยุดที่ขั้นต่ำที่สุดที่ตอบคำถามได้ ยิ่งหยุดได้ต่ำ ยิ่งถูกและยิ่ง deterministic

01
ดึงข้อมูลด้วยโค้ด default · ตอบงานส่วนใหญ่ได้จบที่นี่
02
สัญญาณ console / network ราคาเกือบศูนย์ · มักเจอต้นเหตุก่อนต้องใช้ภาพ
03
เทียบ pixel กับ baseline deterministic · ไม่ต้องใช้ AI เลย
04
ถ่ายภาพเฉพาะจุด (clip) เฉพาะตอนสัญญาณเตือน หรือ diff ไม่ผ่าน
05
ถ่ายทั้งหน้า (full page) ทางเลือกสุดท้ายจริง ๆ เท่านั้น
◀ ถูก / deterministic แพง / ใช้สายตา ▶

04 — หน้า API ของ helper

    
ตัวที่ใช้ในโปรเจกต์จริง

    
คืนผลในทรง { Result, Error } เหมือน API ของ Pedpro เช็คในแพทเทิร์น res?.Result ได้เลย

    

      

        
observePage(page, url, opts)

        
ตัวสะดวก: รวม capture สัญญาณ → goto → wait → extract ให้ลำดับถูกเสมอ ใช้ตัวนี้เป็น default

      

      

        
observe(page, signals, opts)

        
เวอร์ชันคุมเอง: ต้อง captureSignals(page) ก่อน goto แล้วส่ง signals เข้ามาเอง ใช้ตอนต้องคุม flow ละเอียด

      

      

        
mapRows(page, sel, mapper)

        
วนอ่านแถวด้วย data-test แล้ว map เป็น object array สะอาด ๆ — ตัวกลั่นข้อมูลหลัก

      

      

        
clipShot(locator)

        
ถ่ายเฉพาะ element เดียว (ถูกสุดในกลุ่มภาพ) เรียกใช้ภายใน observe เมื่อ escalate เท่านั้น

      

    

  








  

    
05 — Bruno เป็น source of truth

    
cross-check หน้าเว็บกับ contract

    

       ดึงข้อมูลด้วยโค้ดอย่างเดียวบอกได้แค่ “หน้าแสดงอะไร” แต่ไม่รู้ว่ามัน “ถูกไหม” เอา Bruno spec มาเป็นตัวเทียบ — ถ้าหน้าเว็บไม่ตรงกับ contract ฝั่ง API คือมีบั๊ก
    

    

      

        STEP 01
        
โหลด contract

        
loadBru() อ่าน .bru ได้ method + url แล้ว fetchFromSpec() เรียก API จริง = ความจริงฝั่ง server

      

      

        STEP 02
        
ดึงหน้าด้วย observe

        
observePage() ดึงสิ่งที่ render จริงบนหน้า (code-first) = ความจริงฝั่ง client

      

      

        STEP 03
        
crossCheck สองทาง

        
เทียบทีละ field สองทิศ: หน้ามีเกิน API ไหม / API มีแต่หน้าไม่ render ไหม รายงาน mismatch ออกมา

07 — durable state และ metric

    
จำทุก gate ลง SQLite แล้ววัดผล

    

       gate ที่ยิงแล้วลืมไม่มีค่า ต้องจำไว้ถึงจะวัดได้ ใช้ node:sqlite (built-in ไม่ต้องลง native dep) เก็บทุก run + claim กัน race แล้วคำนวณ metric ที่สำคัญที่สุด
    

    

      

        DURABLE
        
trace ทุก run

        
recordGateRun() เก็บ decision, matched, mismatches ลงตาราง gate_runs ตามด้วย human_decision เมื่อคนกด

      

      

        RACE GUARD
        
optimistic claim

        
claimIssue() insert แบบ unique แล้วอ่านกลับมายืนยัน กันสอง agent หยิบ task เดียวกัน

      

      

        METRIC
        
blind_approve_rate

        
ในบรรดา gate ที่ระบบบอก fail แต่คนยัง approve = เซ็นผ่านโดยไม่ตรวจ ยิ่งต่ำยิ่งดี เป้า 0
