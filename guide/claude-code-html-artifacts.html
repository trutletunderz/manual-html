01ทำไมทีม Claude Code เลิกใช้ Markdown

บทความที่ว่า Anthropic "เลิกใช้ Markdown" จริงแต่เกินจริงนิดนึง มันคือ preference ของ Thariq Shihipar (engineering lead ทีม Claude Code) ที่หันมาใช้ HTML เป็น output format มากขึ้น ไม่ใช่ policy ทั้งบริษัท

เหตุผลเบื้องหลังสำคัญกว่า — workflow เปลี่ยน เมื่อก่อน AI เขียน → คนอ่าน → คนแก้มือ → commit (Markdown มีค่าเพราะ "แก้ง่าย") แต่ตอนนี้ AI เขียน spec → ส่งต่อให้ AI session ถัดไปลงมือ → คนแค่ตรวจ ข้อดีเดียวของ Markdown หายไป แต่ข้อจำกัด (อ่านยาวไม่จบ, ไม่มี visual, แชร์ยาก) ยังอยู่ครบ

เนื้อหาเดียวกัน สองรูปแบบ — ลองคลิกสลับ:

Spec: ตั้งระบบ Line Notification

ส่วนประกอบ:

  • ตารางผู้ใช้ — คอลัมน์ ชื่อ/รหัส/สถานะ/บริการ มีปุ่ม เกตเวย์/ตอบรับ/ส่งให้
  • ฟอร์มเพิ่มรหัส — field รหัส/ชื่อรหัส/คำอธิบาย

02HTML โผล่ตอนไหน — 3 จังหวะ

HTML ไม่ใช่ตัวโค้ดที่รันจริง มันคือ "เอกสารรอบตัวโค้ด" ที่ช่วยคิดและสื่อสาร ส่วน .vue/.ts คือสิ่งที่รันจริง และ HTML มีจังหวะเหมาะ 3 จุด:

1

ก่อนเขียนโค้ด — Implementation Plan

รวม context (bruno + mockup + docs) แล้วทำ spec ที่มี API contract, flow diagram, และจุดตัดสินใจที่เทียบทางเลือกข้างกัน ให้ review ก่อน lock

ตัวอย่าง: botline-implementation-plan.html
2

หลังเขียนโค้ด — Code Review Artifact

แนบ PR ด้วย HTML ที่เดินผ่านไฟล์ทีละไฟล์ พร้อม annotation ว่าตรง bruno contract ตรงไหน — เปิดดูได้โดยไม่ต้อง pull โค้ด

ตัวอย่าง: botline-pr-review.html
3

เมื่ออธิบายด้วยข้อความไม่พอ — Throwaway Playground

เช่น prototype animation ปุ่มที่มี slider ปรับค่า + ปุ่ม Copy CSS — ปรับจนถูกใจแล้ว export ไป implement ใช้ครั้งเดียวทิ้งได้

หลักการ: ทุก interactive artifact ต้องมีปุ่ม export

08นำไปใช้กับโปรเจกต์อื่น

workflow นี้ไม่ผูกกับ Vue/Naive UI — มันคือหลักการจัดระเบียบความรู้ ที่ย้ายไปโปรเจกต์ไหนก็ได้ แต่ต้อง "แปล" 3 ส่วนให้ตรงกับโปรเจกต์นั้นก่อน

A
แปล "source of truth"

ที่ Pedpro คือ bruno specs — โปรเจกต์อื่นอาจเป็น OpenAPI/Swagger, GraphQL schema, Protobuf, หรือ TypeScript types ที่ share จาก backend หา "เอกสารที่เป็นความจริงของ contract" ให้เจอก่อน แล้วให้ AI อ่านอันนั้นเป็นอันดับแรกเสมอ

B
แปล "pattern เดิม"

ที่ Pedpro คือ reference feature (GeneralSetting / MemberRegister) — โปรเจกต์อื่นชี้ไปที่ feature ที่เขียนดีที่สุด 1-2 อันเป็นตัวอย่าง ให้ AI match แทนที่จะคิด pattern ใหม่ หลักคือ "อย่าประดิษฐ์ ให้เลียนของที่มี"

C
แปล "ลำดับชั้น"

ที่ Pedpro คือ type → api → composable → component → view — โปรเจกต์อื่นอาจเป็น model → repository → service → controller หรือ schema → resolver → component หลักคือเขียนจากชั้นในออกชั้นนอก เพราะชั้นนอกพึ่งชั้นใน

checklist ตั้งระบบในโปรเจกต์ใหม่

1
เขียน CLAUDE.md (หรือเทียบเท่า) — ระบุ stack, golden rules, source of truth, ลำดับการเขียน, naming convention, สิ่งที่ห้ามแตะ ใช้ของ Pedpro (บท 06) เป็นโครง
2
ชี้ reference feature — เลือก feature ที่เขียนสะอาดที่สุดในโปรเจกต์ 1-2 อัน ระบุใน CLAUDE.md ว่า "ดูตัวอย่างที่นี่"
3
ทำ _template.html — โครงเปล่า HTML spec สำหรับหน้าที่มีจุดตัดสินใจ ปรับ section ให้ตรงโครงสร้างโปรเจกต์
4
ตั้งเกณฑ์ "เมื่อไหร่ทำ HTML spec" — ใช้ 3 คำถามจากบท 01 หรือปรับให้เข้ากับทีม
5
ลองกับ feature เล็กก่อน 1 อัน — อย่าเปลี่ยนทั้งทีมทันที พิสูจน์กับงานจริงชิ้นเดียวให้เห็นผลก่อน

09ข้อควรระวัง

workflow นี้มีกับดักที่เจอบ่อย — รู้ไว้ก่อนจะได้ไม่พลาดซ้ำรอยที่เราเจอในวันนี้

▲ อย่าเอา HTML ไปแทนโค้ดจริง
HTML artifact คือ "เอกสารรอบโค้ด" ไม่ใช่ production code ตัว feature ที่รันจริงต้องเป็น .vue/.ts เข้า git ตามปกติ — HTML mockup ที่ดูเหมือนเสร็จแล้วอาจหลอกให้คนคิดว่า implement แล้วทั้งที่ยังไม่ได้ทำ
▲ อย่าให้ AI เดา contract
บทเรียนตรงจากวันนี้ — ตอนแรกผมเดา stack ผิด (นึกว่า Nuxt) เขียนโค้ดที่ขัด CLAUDE.md ทั้งหมด ต้องทิ้งทั้งชุด ต้องให้ AI อ่าน bruno + CLAUDE.md จริงก่อนเสมอ field/type/required ห้ามเดา
◆ อย่าข้ามขั้น HTML spec ตอนมีจุดตัดสินใจ
อีกบทเรียนจากวันนี้ — ผมข้าม spec ไปเขียนโค้ดเลย แล้วมาเจอจุดที่ตัดสินใจแทนไม่ได้ตอนจบ (เชื่อม 2 หน้ายังไง) ถ้าทำ spec ก่อน จุดนั้นจะถูกถามตั้งแต่ต้น โค้ดออกมาถูกรอบเดียว
◆ git diff ของ HTML โหด
HTML artifact ที่ commit เข้า repo จะ review ยากเวลา diff (เต็มไปด้วย tag) — เก็บเฉพาะที่เป็น reference ถาวร (เช่น _template.html) ส่วน spec/review ที่ใช้ครั้งเดียวไม่ต้อง commit ก็ได้
● เพดานอยู่ที่ความสนใจของผู้รับ
interactive artifact ที่ดีไม่ได้ปลดล็อกให้ผลิตไม่จำกัด ทำ simulation สวย ๆ สิบอันคนก็ดูได้แค่จำนวนหนึ่ง — ถามทุกครั้งก่อนสร้างเพิ่มว่า "อันนี้มีคนรับสารจริงไหม" ไม่ใช่ "ทำสวยได้ไหม"
● อย่าทำ HTML spec กับทุกหน้า
หน้า single-form ธรรมดาที่ก๊อป pattern ได้เลย ไม่ต้องทำ spec — จะเปลืองเวลาเปล่า ใช้เกณฑ์ 3 ข้อ (บท 01) กรองก่อนเสมอ
10แก่นจริง — Productivity คืออัตราการส่งต่อ

วงการ AI ชอบวัด productivity จากฝั่งผลิต (จำนวนเอกสาร, บรรทัดโค้ด, นาทีที่ประหยัด) เพราะนับง่าย แต่มูลค่าจริงเกิดที่ฝั่งรับ — มีคนเข้าใจไหม เร็วแค่ไหน เอาไปตัดสินใจต่อได้ไหม

ถ้าฝั่งผลิตโต 100 เท่า แต่ฝั่งรับเท่าเดิม — ไม่ใช่แค่ "ไม่ progress" แต่เป็นลบ เพราะคอขวดย้ายจาก "เขียนไม่ทัน" เป็น "อ่านไม่ทัน" และกองเอกสารที่ไม่มีคนอ่านยังสร้าง noise กลบของสำคัญ

01HTML แทน Markdown เพราะ คนอ่านจบจริง คลิกสำรวจได้ — อัตราส่งต่อสูงขึ้น 02HTML เกิดตอน "คิด" และ "ตรวจ" — ไม่เพิ่ม artifact ที่ไม่จำเป็น 03BotLine spec มีจุดตัดสินใจ ที่กดเลือกได้ใน 30 วินาที — เทียบกับอ่าน Markdown 100 บรรทัดแล้วยังไม่ตัดสินใจ 04ย้ายไปโปรเจกต์อื่นได้ เพราะเป็นหลักการ ไม่ใช่ tool — ส่งต่อ "วิธีทำงาน" เองก็เพิ่มขึ้น

เพดานของหลักการนี้อยู่ที่ความสนใจของผู้รับเสมอ — interactive artifact ที่ดีไม่ได้ปลดล็อกให้ผลิตไม่จำกัด มันแค่ทำให้แต่ละชิ้นคุ้มความสนใจมากขึ้น คำถามก่อนสร้างอะไรเพิ่มจึงไม่ใช่ "ทำสวยได้ไหม" แต่คือ "มีคนรับสารจริงไหม"
