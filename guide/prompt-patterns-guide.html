40 Prompt Patterns
ที่ Thai Dev ควรรู้

โครง prompt เป็นอังกฤษเพราะ model เข้าใจ pattern พวกนี้แม่นกว่า — แต่เนื้อหาและคำตอบยังเป็นภาษาไทยได้ ไม่ต้องเขียนอังกฤษคล่อง

วิธีใช้: copy pattern → เติมเนื้อหาไทยของคุณ → ปิดท้ายด้วย "ตอบเป็นภาษาไทย" เสมอถ้า prompt ส่วนใหญ่เป็นอังกฤษ
01

กำหนดบทบาท · Role / Persona

ทำให้ model ปรับ tone และความลึกของคำตอบให้ตรงระดับคุณ

01
Act as a senior frontend engineer reviewing a PR.
อยากได้ review เข้มข้น
02
You are a TypeScript strict-mode linter. Flag every type issue.
หา type error
03
Respond as a skeptical staff engineer. Challenge my assumptions.
อยากโดนค้าน ไม่อยากได้ yes-man
04
Explain this like I already know Vue but am new to Nuxt 4.
ปรับ level ให้ตรงคุณ
ตัวอย่างใช้กับ xFish 
Act as a senior Vue 3.5 engineer. Review this Pinia store for
xFish conventions (no semicolons, double quotes, res?.Result pattern).

<store>
[paste code]
</store>

ชี้ปัญหาเป็นภาษาไทย เรียงตามความสำคัญ
02

ควบคุมการคิด · Reasoning Control

บังคับให้ model คิดก่อนตอบ → คุณภาพสูงขึ้นมากในงานซับซ้อน

05
Think step by step before answering.
บังคับ reasoning
06
First list the trade-offs, then recommend.
ได้การวิเคราะห์ก่อนสรุป
07
Before coding, outline your approach and wait for my confirmation.
กัน model เขียนผิดทิศ (plan-first)
08
Identify edge cases I might have missed.
หาช่องโหว่
09
What would break this? List failure modes.
stress-test idea
10
Rate your confidence (1-10) and explain why.
รู้ว่า model มั่นใจแค่ไหน
ตัวอย่าง plan-first 
Before refactoring this component, outline your approach in 3-5 bullets
and wait for my confirmation. Don't write code yet.

04

ให้ตัวอย่าง · Few-shot

วิธีที่ทรงพลังที่สุดในการสอน model ให้ทำตาม pattern คุณ — และเป็นตัวแก้ปัญหา "Sonnet ทำผิด convention"

17
Here's an example of the style I want: [example]. Now do the same for: [target].
อยากได้ style ตรง
18
Follow the same pattern as this existing page: [reference].
ให้เลียนแบบ code เดิม
19
Match the tone of this: [sample].
คุม tone งานเขียน
ตัวอย่างสำคัญ — แก้ปัญหา Sonnet ทำผิด convention 
Create a new product listing page following the EXACT same pattern as
this existing page. Match imports, store usage, error handling.

<reference_page>
[paste pages/existing-feature/index.vue]
</reference_page>

สร้างหน้าใหม่ชื่อ pages/products/index.vue

05

วิจารณ์และปรับปรุง · Critique / Iterate

บังคับให้ model หาจุดอ่อนแทนที่จะเห็นด้วยกับคุณ

20
Criticize this approach. What's wrong with it?
หาจุดอ่อน
21
Play devil's advocate against this decision.
ค้านเพื่อทดสอบ
22
What would a security reviewer flag here?
มอง security
23
Suggest 3 alternatives and rank them.
ได้ทางเลือก
24
Review your own answer and fix any mistakes.
self-correct
ตัวอย่าง — ทดสอบ design decision 
I'm planning to put the Plan Review Gate logic inside an LLM call.
Criticize this approach. What's wrong with it? Play devil's advocate.

06

ขอบเขตและข้อจำกัด · Constraints

กัน model ทำเกินขอบเขต — สำคัญมากกับ xFish ที่มี convention เยอะ

25
Do NOT add dependencies. Use only what's already imported.
กันเพิ่ม library มั่ว
26
Only change what I asked. Don't refactor anything else.
กัน scope creep
27
If you're unsure, ask instead of guessing.
กัน hallucination
28
Assume nothing. Ask clarifying questions first.
กันเข้าใจผิด
29
Use Tailwind core utilities only — no arbitrary values.
คุม xFish rule
30
If the answer requires info you don't have, say so explicitly.
กันมั่ว

Fix the type error in this component.
Constraints:
- Only change what's needed to fix the error
- Do NOT refactor other parts
- Do NOT add dependencies
- Keep no-semicolon, double-quote style
- If unsure about the intended type, ask instead of guessing

07

Agentic / Multi-step · สำหรับ Hermes & OpenClaw

pattern สำหรับงานที่ agent ต้องทำหลาย step ใช้ tool ต่อเนื่อง และมี human approval gate — ออกแบบมาเพื่อ workflow ที่คุณรันบน VPS

31
Break this into a numbered task list. Execute one step at a time, report after each.
บังคับ agent ทำทีละ step ไม่รวบ
32
After each tool call, summarize what you learned before the next action.
กัน agent หลงทางหลัง tool call เยอะ
33
Do NOT commit, push, or run destructive commands. Return the plan for my approval first.
human approval gate — หัวใจ OpenClaw
34
State your plan, confidence, and what could go wrong. Wait for "go" before executing.
Plan Review Gate pattern
35
If a step fails, stop and report. Do NOT retry silently or work around it.
fail-closed — กัน agent ซ่อน error
36
Use only the tools listed. If you need a tool you don't have, ask instead of improvising.
คุมขอบเขต tool ของ subagent
37
You are a read-only reviewer. Return findings to the parent agent. Make no changes.
subagent แบบ read-only (เช่น typescript-reviewer)
38
Persist state to YouTrack after each completed step so we can resume on restart.
scale-to-zero / state ownership
39
Run independent subtasks in parallel, but serialize anything needing my judgment.
"parallelize agents, serialize judgment"
40
Before finishing, verify your work against the original spec and list what you skipped.
self-verification ก่อนปิด task
ตัวอย่าง — Plan Review Gate (OpenClaw-style) 
You are implementing a xFish feature. Follow this protocol strictly:

1. Break the task into a numbered plan
2. State confidence (1-10) and what could go wrong
3. Do NOT write code, commit, or push yet
4. Wait for my "go" before executing
5. After each step, summarize and pause if anything fails

<task>
เพิ่มหน้า leaderboard ที่ดึงข้อมูลจาก endpoint /api/ranking
</task>

เสนอ plan เป็นภาษาไทยก่อน รอผมอนุมัติ
ตัวอย่าง — read-only subagent 
You are a read-only TypeScript reviewer subagent.
Review the diff below. Return findings to the parent agent.
Make NO changes, NO commits. Use only file-read tools.
If a step fails, stop and report — do not work around it.

[paste]
</diff>

สูตรลัด · ประกอบ prompt ใน 4 ส่วน

จำแค่ 4 ช่องนี้: ใคร → ทำอะไร → ห้ามอะไร → ออกมาแบบไหน

Role
ใคร — กำหนดบทบาท
Task
ทำอะไร — เนื้อหาไทยได้
Constraints
ห้ามอะไร
Format
ออกมาแบบไหน
โครงพร้อมใช้ 
[ROLE]        Act as a senior Vue engineer.
[TASK]        ช่วยรีวิว component นี้ โฟกัสที่ accessibility
[CONSTRAINTS] Only flag real issues. Use xFish conventions.
[FORMAT]      ตอบเป็นตารางภาษาไทย เรียงตามความรุนแรง

<component>
[code]
</component>
!

เกร็ดสำคัญสำหรับ Thai dev

  1. โครงเป็นอังกฤษ เนื้อเป็นไทยได้ — model เข้าใจ pattern อังกฤษแม่นกว่า แต่ไม่บังคับให้คุณเขียนเนื้อหาเป็นอังกฤษ
  2. ปิดท้ายด้วย ตอบเป็นภาษาไทย เสมอถ้า prompt ส่วนใหญ่เป็นอังกฤษ ไม่งั้น model จะตอบอังกฤษ
  3. XML tags เช่น <component>...</component> ช่วยมาก — แยก instruction ออกจาก data ชัดเจน ลดความสับสน
  4. ยิ่งให้ constraint ชัด ยิ่งได้ผลตรง โดยเฉพาะกับ xFish ที่มี convention เยอะ
  5. few-shot (หมวด 04) คือตัวแก้ปัญหา "Sonnet ทำผิด convention" — ชี้ไปหน้าเดิมที่ถูกต้อง ดีกว่าอธิบาย rule ยาว ๆ
