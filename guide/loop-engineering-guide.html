ส่วนที่ 1

3 ระดับของการทำงานกับ AI Agent

วิวัฒนาการจากการ prompt ทีละครั้ง ไปสู่ระบบอัตโนมัติที่ prompt agent แทนตัวเอง แต่ละระดับเพิ่ม layer ใหม่เข้าไป ไม่ได้แทนที่ระดับก่อน

ระดับ 1

Prompt Engineering — วิศวกรรม Prompt

Human-in-the-loop
Human → prompt → Agent → output → Human

คนถือ agent ตลอดเวลา ส่ง prompt ทีละ turn รอผล อ่าน และพิมพ์ต่อ ขีดจำกัดคือ bandwidth ของคน — agent ฉลาดแค่ไหนก็ช้าแค่คนพิมพ์

Leverage point: quality of the prompt you write · Bottleneck: human typing speed
เพิ่ม harness (สภาพแวดล้อม)
ระดับ 2

Harness Engineering — วิศวกรรม Harness

Human-triggers
Human → trigger → Harness + Agent → output → Human

ออกแบบ "สภาพแวดล้อม" ที่ agent อาศัยอยู่ — CLAUDE.md บอก conventions, Skills บอกวิธีทำงาน, Hooks เป็น guardrail อัตโนมัติ agent ฉลาดขึ้นมาก แต่คนยังต้อง trigger เอง ทำงานทีละ task

Harness = CLAUDE.md + Skills + Rules + Hooks + Subagents + MCP
💡 Addy: "A decent model with a great harness beats a great model with a bad harness." — model ธรรมดา + harness ดี ชนะ model เก่ง + harness แย่
เพิ่ม automation + memory (ความจำนอก session)
ระดับ 3

Loop Engineering — วิศวกรรม Loop

System-runs
Automation → หางาน → Agent → Sub-agent → ตรวจสอบ → loop / เสร็จ

ออกแบบ "ระบบ" ที่ส่ง prompt ให้ agent แทนตัวคุณ loop วนซ้ำจนกว่า condition ที่กำหนดจะเป็นจริง คนย้ายจาก "คนพิมพ์ prompt" มาเป็น "คนออกแบบ loop" แทน

Loop = Automation + Worktrees + Skills + Connectors + Sub-agents + Memory
✓ Peter Steinberger: "You shouldn't be prompting coding agents anymore. You should be designing loops that prompt your agents."
แปล: "คุณไม่ควร prompt coding agent อีกต่อไป คุณควรออกแบบ loop ที่ prompt agent แทนคุณ"

6 Building Blocks ของ Loop

ส่วนประกอบ 6 ชิ้นที่ทำให้ Loop Engineering ทำงานได้จริง แต่ละ block มีหน้าที่เฉพาะของตัวเอง คลิกเพื่อดู deep-dive ของแต่ละ block

Each block has a single responsibility. Missing any one of them creates a gap that breaks the loop.

💓
Block 01
Automations
heartbeat / จังหวะใจ
Mechanism that makes loop run without human
กลไกที่ทำให้ loop รันโดยไม่ต้องมี human กดปุ่ม
🌿
Block 02
Worktrees
parallel isolation / แยกพื้นที่ทำงาน
Multiple agents work simultaneously without conflicts
agent หลายตัวทำงานพร้อมกันบน codebase เดียวโดยไม่ชนกัน
📚
Block 03
Skills
reusable knowledge / ความรู้นำกลับมาใช้
Project knowledge agent reads every run
ความรู้ project ที่ agent อ่านทุก run ไม่ต้อง explain ซ้ำ
🔌
Block 04
Connectors
external integrations / เชื่อมต่อภายนอก
Connect agent to real tools via MCP and REST
เชื่อม agent กับ tools ภายนอกเช่น YouTrack, GitHub, Telegram
🤖
Block 05
Sub-agents
verify & specialize / ตรวจสอบและเชี่ยวชาญ
Creator ≠ Checker — cross-verify prevents hallucination
ผู้สร้างไม่ใช่ผู้ตรวจ — ป้องกัน hallucination ด้วยการตรวจข้ามตัว
💾
Block 06
Memory
external state / สถานะภายนอก
Agent forgets every run — but the repo doesn't
agent ลืมทุก run แต่ repo และ issue board ไม่ลืม
Block 01

💓 Automations — ระบบอัตโนมัติ

Automation คือกลไกที่ทำให้ loop "ตื่น" ขึ้นมาทำงานโดยไม่ต้องรอคนสั่ง เหมือนหัวใจที่เต้นตามจังหวะ ไม่ต้องบอกให้เต้นทุกครั้ง มี 3 แบบตามสิ่งที่ trigger ให้มันทำงาน

Automation is what separates "a loop you ran once" from "a loop that keeps running." Without automation, you still have to manually kick it off each time.
⏱
Schedule / กำหนดเวลา
cron / interval

ทำงานตามเวลาที่กำหนดไว้ล่วงหน้า เช่น "ทุก 5 นาที" หรือ "ทุกเช้า 8 โมง" ไม่รอ human สั่ง ระบบตื่นเองตามตาราง

Fires at fixed times regardless of what's happening. Good for polling and routine checks.
Pedpro: cron */5 * * * * → ดึง YouTrack sprint board ทุก 5 นาที
⚡
Event / เหตุการณ์
webhook / message

ทำงานเมื่อมีเหตุการณ์เข้ามา — ตอบสนองทันทีโดยไม่ต้อง poll เช่น เมื่อ user กด /approve ใน Telegram ระบบทำงานต่อทันที

Reactive, not polling. More efficient than schedule for infrequent events.
Pedpro: Telegram /approve → spec เปลี่ยน state เป็น ready_for_code ทันที
🔀
Condition / เงื่อนไข
watch / gate

ทำงานเมื่อ "เงื่อนไข" เป็นจริง เฝ้าดู state แล้วยิงเมื่อถึงเวลา เช่น "ถ้า ticket มี status = ReadyForAgent ให้ agent claim"

Polls a condition and acts when it becomes true. Good for state-based triggers.
Pedpro: YouTrack status = 'ReadyForAgent' → heartbeat claim ticket อัตโนมัติ

Block 02

🌿 Worktrees — พื้นที่ทำงานแยกกัน

Git Worktree คือการแยก "working directory" ออกจากกัน — เหมือนมีห้องทำงานหลายห้องในบ้านเดียวกัน แต่ละ agent ได้ห้องของตัวเอง ไม่เดินเข้าห้องคนอื่น ทำให้ agent หลายตัวทำงานบน codebase เดียวพร้อมกันได้โดยไม่ conflict

A git worktree is a separate working directory linked to the same repository. Each worktree has its own branch, its own files on disk, and its own uncommitted changes.
💥 File Conflict

ถ้าไม่แยก worktree — agent 2 ตัวแก้ไฟล์เดียวกันพร้อมกัน เมื่อ merge กลับจะเกิด conflict ที่แก้ยากมาก

🌀 Dirty Working Tree

uncommitted changes ของ task A ปนกับ task B ทำให้ไม่รู้ว่าโค้ดไหนเป็นของ task ไหน

⏳ Sequential Bottleneck

ถ้าใช้ repo เดียว agent ตัวที่ 2 ต้องรอตัวแรกเสร็จก่อน ทำให้ parallel ไม่ได้ผล

Convention สำหรับ Pedpro — โครงสร้าง directory
~/pedpro/              ← main repo  (human only — agents ห้ามแตะ!)
~/pedpro-wt/           ← worktree root
  feat/PEDPRO-142/     ← agent A ทำ BotLine UI (branch: feat/PEDPRO-142)
  feat/PEDPRO-143/     ← agent B ทำ BotLine API (branch: feat/PEDPRO-143)
  fix/PEDPRO-139/      ← agent C แก้ Auth bug (branch: fix/PEDPRO-139)

Block 03

📚 Skills — ความรู้นำกลับมาใช้

Skills คือ "ความรู้ project" ที่เขียนไว้ให้ agent อ่านทุกครั้งที่รัน แทนที่จะอธิบาย conventions ซ้ำทุก session ก็เขียนครั้งเดียวแล้วปล่อยให้ agent อ่านเอง แต่มี 3 ชั้นของ "ความรู้" ที่ต่างกัน — ต้องเลือกใช้ให้ถูก

📌
CLAUDE.md
Always injected · ฉีด context ทุก session

เหมือน "DNA" ของ agent — อ่านทุกครั้งก่อนทำอะไรทั้งนั้น เก็บ prohibitions (สิ่งที่ห้ามทำ), architecture rules, stack conventions

⚠ อย่าใส่ critical rules ใน Project Knowledge (RAG) — อาจไม่ถูก retrieve เลย ใส่ใน CLAUDE.md เท่านั้น
📚
Skills
On-demand · เรียกใช้ตามต้องการ

เหมือน "ผู้เชี่ยวชาญที่โทรเรียก" — มาเฉพาะตอนต้องการ triggered โดย description match หรือเรียกตรงๆ ด้วย @skill-name

💡 description field คือสิ่งสำคัญที่สุด — บอกว่าเมื่อไหร่ agent ควรเรียก skill นี้
🗄
Project Knowledge
RAG retrieval · ค้นหาตามความเกี่ยวข้อง

เหมือน "ห้องสมุดที่ต้องค้นหาก่อน" — semantic search ดึงเฉพาะส่วนที่เกี่ยวข้อง เหมาะสำหรับ docs ขนาดใหญ่ที่ไม่ได้ใช้ทุก session

⚠ reliability 80-90% — อาจพลาดถ้าคำถามไม่ตรง อย่าเก็บ critical rules ที่นี่

Hashimoto Pattern — ทำให้ Skills เติบโตเองได้

ทุกครั้งที่ agent ทำผิด แทนที่จะแค่แก้ผลลัพธ์ ให้ถามว่า "context อะไรขาดไป" แล้วเพิ่มเข้า CLAUDE.md หรือ skill ใหม่ ทำแบบนี้ทุก sprint — CLAUDE.md จะฉลาดขึ้นทุกรอบโดยอัตโนมัติ

1.Claude ทำผิด → อย่าแค่แก้ output แล้วเดินหน้าต่อ 2.หา root cause — "context อะไรที่ขาดทำให้ agent เดาผิด?" 3.เขียน rule ใหม่เพิ่มเข้า CLAUDE.md หรือสร้าง SKILL.md ใหม่ 4.เพิ่ม hook หรือ test ที่ agent รัน self-check ได้ทุกครั้ง
"ทุกบรรทัดใน CLAUDE.md คือ past agent failure ที่ป้องกันแล้ว" — Mitchell Hashimoto
แปล: ทุก rule ใน CLAUDE.md ควร trace กลับไปได้ถึง "สิ่งที่ agent เคยทำผิดจริงๆ" ไม่ใช่ rule สมมติ
Block 04

🔌 Connectors — การเชื่อมต่อภายนอก

Connector คือสะพานเชื่อม agent กับ tools ภายนอก — YouTrack, GitHub, Telegram, Cloud Run loop ที่เข้าถึงแค่ filesystem เป็น "loop เล็กๆ" แต่ loop ที่เชื่อมกับ tools จริงๆ คือ loop ที่ทำงานได้จริง มี 3 วิธีในการเชื่อม ต่างกันที่ระดับ intelligence และ predictability

🔌 MCP
Model Context Protocol

มาตรฐานกลางที่ Anthropic กำหนด — agent พูดคุยกับ service โดยตรงผ่าน tool calls schema-typed ทำให้ไม่ต้อง parse response เอง รองรับ bidirectional (service ส่ง event กลับได้)

Best when the service has an MCP server (YouTrack 2025.3+, GitHub, etc.)
Pedpro: YouTrack MCP — agent ดึง/อัปเดต ticket โดยตรงโดยไม่ต้อง paste ticket ID
🌐 REST / Webhook
HTTP API ทั่วไป

HTTP call ธรรมดา — agent เรียก fetch() หรือ curl ไปหา API ใช้ได้กับ service ทุกตัวที่มี HTTP API flexible แต่ต้อง parse response เอง

Use when no MCP server exists. More work but works with any HTTP API.
Pedpro: Telegram Bot API — curl ตรงไปหา Bot API ส่ง notification
⚙ Deterministic Layer
thin middleware — ไม่ใช่ LLM!

code ธรรมดา (if/switch) ทำ routing ระหว่าง connectors predictable 100% ไม่มีโอกาส hallucinate routing decision ทำงานเร็ว ไม่เสีย token

This layer MUST be deterministic code, never LLM. LLM routing = unpredictable.
Pedpro: heartbeat.mjs — if/switch routing ตาม ticket type ไม่ใช่ LLM ตัดสินใจ
Pedpro Connector Stack — หน้าที่ของแต่ละ connector
YouTrack State Owner เจ้าของ state ทั้งหมด — ทุก ticket state change เขียนที่นี่เท่านั้น เป็น single source of truth ของ loop
GitHub CI Gate จัดการ code review, CI/CD, branch protection — agent สร้าง PR ได้แต่ ไม่ merge เอง
Telegram Human Gate ช่องทาง approval เดียว — Human อยู่ที่นี่ รับ /approve /reject ไม่ต้องเปิด YouTrack ตลอดเวลา
Cloud Run Deploy รับ trigger จาก GitHub Actions → deploy ผ่าน cloudbuild.yaml อัตโนมัติ agent ไม่แตะ deploy โดยตรง
⚠ Anti-pattern: ห้ามให้ LLM ตัดสินใจ routing ระหว่าง connectors routing ต้อง predictable 100% ใช้ if/switch code เท่านั้น — LLM routing = โอกาส hallucinate ทุกรอบ
Block 05

🤖 Sub-agents — ผู้ช่วยเฉพาะทาง

Sub-agent คือ agent ลูกที่รับงานเฉพาะทาง — แทนที่จะมี agent ตัวเดียวทำทุกอย่าง แบ่งงานออกเป็นบทบาทชัดเจน แต่ละตัวมี context ที่ตรงกับงานตัวเอง หลักการสำคัญที่สุดคือ "ผู้สร้าง ≠ ผู้ตรวจ" — agent ที่เขียนโค้ดต้องไม่ตัดสินโค้ดตัวเอง

🨢
Separation of Concerns

ผู้สร้าง ≠ ผู้ตรวจ — agent ที่เขียนโค้ดมี bias ที่จะคิดว่าโค้ดตัวเองดี (conflict of interest) ทำให้ hallucination ผ่านไปได้

maker ≠ checker
🎯
Specialization

แต่ละ sub-agent มีขอบเขตงานแคบและชัดเจน — context ที่ inject เข้าไปตรงงาน ทำให้ hallucinate น้อยกว่า general agent

narrow scope = ลด hallucination
💰
Cost Routing

งานเบาใช้ model เบา (Haiku) งานหนักใช้ model หนัก (Opus) ไม่ต้องจ่ายค่า Opus กับงานที่ Haiku ทำได้เหมือนกัน

Haiku 60% · Sonnet 35% · Opus 5%

Block 06

💾 Memory — ความจำนอก Session

นี่คือ block ที่ทำให้ 5 blocks ก่อนหน้า "จำกัน" ข้ามรอบได้ ปัญหาพื้นฐานของ AI คือ ลืมทุก session — แต่ถ้าออกแบบให้เขียน state ลงไฟล์ก่อนจบ session ถัดไปก็อ่านขึ้นมาทำงานต่อได้ทันที กุญแจสำคัญ: "The agent forgets, the repo doesn't."

🧠
ปัญหา: Agent มี Amnesia

ทุก conversation เริ่มใหม่ตั้งแต่ต้น agent ไม่รู้ว่าเมื่อวานทำอะไรไป ถ้าไม่มี external state ข้อมูลหายทันทีที่ session จบ

Agent amnesia: every session starts blank. Without external state, nothing persists.
🗄
วิธีแก้: Repo ไม่ลืม

ไฟล์ที่อยู่นอก conversation window ไม่หายไปไหน ถ้า agent เขียน state ลงไฟล์ก่อน session จบ รอบถัดไปก็อ่านได้ทันที

The filesystem is the memory. Write state to disk before session ends. Read it on next start.
4 Memory Tiers — ชั้นของความจำ อายุขัยต่างกัน
💨
Tier 1 — Process Memoryตาย session จบ

context window ของ agent ใน session นี้ — ห้าม rely ข้ามรอบ ข้อมูลหายทันทีที่ session จบ

📁
Tier 2 — Project Memoryอยู่ตราบ repo อยู่

ไฟล์ใน repo — CLAUDE.md, docs/specs, BUGS.md, docs/adr · commit ก่อน session จบทุกครั้ง

📋
Tier 3 — Task Memoryอยู่ตราบ ticket มีชีวิต

YouTrack tickets — state, history, comments, PR links · single source of truth ของ loop ทั้งหมด

🧠
Tier 4 — Knowledge Memorycompound ตลอดชีวิต project

BUGS.md, ADRs, CLAUDE.md prohibitions, skill files — สะสมและฉลาดขึ้นทุก sprint

Spec State Machine — วงจรชีวิตของ spec

ทุก spec ผ่านสถานะต่างๆ ก่อนถึงมือ agent — สถานะเหล่านี้เป็น memory ที่บอกว่างานอยู่ที่ไหน

draft · ร่าง → revising · กำลังแก้ → approved · อนุมัติแล้ว → ready_for_code · พร้อม | rejected · ปฏิเสธ | stale · ค้าง >7d
Golden Rules of Loop Memory — กฎทองของ loop memory
ก่อน session จบcommit ทุก state change ลงไฟล์หรือ YouTrack ก่อนออก
หลัง agent failเขียน rule ใหม่เข้า CLAUDE.md ทันที — Hashimoto pattern ทำให้ failure ไม่ซ้ำ
ทุก architecture decisionเขียน ADR เก็บไว้ใน docs/adr/ — ป้องกัน agent ตั้งคำถามซ้ำหรือ reverse decisions
ทุก spec cycleเก็บ version ไว้ทุกครั้ง อย่า overwrite spec ที่ approved แล้ว

01 · /goal vs /loop — เงื่อนไขสิ้นสุดคือ design problem

สองคำสั่งที่คนมักสับสน: /loop รันตาม cadence แต่ /goal รันจนกว่า condition จริงๆ จะเป็นจริง และสำคัญกว่า — มี separate model ตรวจว่าเสร็จหรือยัง ไม่ให้ตัวเดียวกันตัดสินตัวเอง

"/goal keeps going until a condition you wrote is actually true, and after every turn a separate small model checks whether you are done, so the agent that wrote the code isn't the one grading it."
แปล: /goal ทำงานต่อจนกว่า condition ที่คุณเขียนจะเป็นจริง และหลังทุก turn มี model ตัวเล็กแยกต่างหากมาตรวจว่าเสร็จหรือยัง ทำให้ agent ที่เขียนโค้ดไม่ใช่คนตัดสินตัวเอง
/loop — รันตาม cadence (จังหวะ)

รันซ้ำตาม schedule ที่กำหนด เช่น ทุก 5 นาที Pedpro มีอยู่แล้วผ่าน heartbeat.mjs

/goal — รันจนกว่า condition จะจริง

รันต่อจนกว่าจะผ่าน condition ที่กำหนด เช่น "typecheck pass + lint clean + Bruno honored" Pedpro ยังขาดอยู่

⚠ Pedpro gap: task ทุกอย่างยังจบตาม human approve ผ่าน Telegram ไม่มี machine-verifiable done condition
## done_condition.md — เพิ่มใน .claude/agents/worker.md

- [ ] npm run lint → exit 0             (lint ผ่าน)
- [ ] bruno-spec-lookup: no endpoint drift  (API ไม่ drift)
- [ ] All existing tests pass               (test ทุกอันยังผ่าน)
# checker subagent ตรวจ condition เหล่านี้ทุก turn
# ถ้า fail → worker ทำงานต่อ ไม่ประกาศ "done" เอง

02 · Triage Inbox Pattern — inbox รับงาน

loop ที่ดีไม่ใช่แค่ "ทำงานเมื่อมีงาน" แต่ต้องรายงานด้วยว่า "scan แล้วไม่พบอะไร" ทำให้รู้ว่า loop ยัง healthy ไม่ใช่พัง และ surface signal ที่น่าสนใจให้ human เห็น

"The runs that find something go to a Triage inbox, and the runs that find nothing just archive themselves which is nice. OpenAI uses them for daily issue triage, summarizing CI failures, writing commit briefings."
แปล: รอบที่พบงานไปที่ Triage inbox ส่วนรอบที่ไม่พบอะไรก็ archive ตัวเองซึ่งดีมาก OpenAI ใช้ internally สำหรับงานน่าเบื่อเช่น triage issue รายวัน สรุป CI failure เขียน commit briefing
⚠ Pedpro gap: heartbeat.mjs ปัจจุบัน idle เงียบๆ เมื่อไม่มี ticket ไม่มี structured output ทำให้ไม่รู้ว่า loop ทำงานหรือพัง
// heartbeat.mjs — เพิ่ม triage output
const result = await youtrack.query({ status: 'ReadyForAgent' });

if (result.tickets.length === 0) {
  // archive run — ไม่ใช่แค่ return เฉยๆ
  await log.append({
    time: new Date().toISOString(),
    run: 'no-work',
    summary: 'scan complete — 0 tickets found'
  });
  return; // exit clean, ให้ cron ยิงรอบถัดไป
}

// มีงาน → triage + notify
const queue = await triageSkill.prioritize(result.tickets);
await telegram.notify(`📋 ${queue.length} tickets queued: ${queue.map(t=>t.id).join(', ')}`);

03 · Intent Debt — หนี้ของ intent ที่ไม่เขียนไว้

ทุก session ที่ไม่มี skill agent จะ "fill holes" ด้วยการเดา — สะสมเป็น "intent debt" Skills เป็นการชำระหนี้ทีละน้อย และยิ่งมี skills มากยิ่ง compound — agent ฉลาดขึ้นทุกรอบโดยไม่ต้องบอก

"An agent starts every session cold and it will fill any hole in your intent with a confident guess. Without skills the loop re-derives your whole project from zero every cycle, with skills it kind of compounds."
แปล: agent เริ่มทุก session แบบ cold และจะ fill ทุก hole ใน intent ด้วย confident guess ถ้าไม่มี skills loop จะ re-derive project ทั้งหมดจากศูนย์ทุกรอบ ถ้ามีมันจะ compound
✗ Without Skill

agent เดา: import path อาจผิด, API schema อาจผิด, convention อาจผิด — re-derive + เสีย token ทุก session

✓ With Skill

อ่าน convention จาก skill file — ทำถูกตั้งแต่ turn แรก ไม่ต้อง guess ไม่เสีย token กับ re-derivation

🔄 Compounding

skills + Hashimoto rules = agent ฉลาดขึ้นทุก sprint โดยอัตโนมัติ debt ลดทุกรอบ

💡 Pedpro gap: ยังขาด bruno-contract (enforce query Bruno spec ก่อน modify endpoint) และ pedpro-tdd (Nuxt-specific test patterns)

04 · Skill vs Plugin — รูปแบบ vs วิธีแจกจ่าย

Skill คือวิธีเขียน ใช้ได้ใน repo นี้ · Plugin คือวิธี ship มัน แชร์ข้าม repos ออกแบบ self-contained ตั้งแต่ตอนนี้เพื่อ plugin-ready ในอนาคตเมื่อ Pedpro โตขึ้น

"The skill is the authoring format and a plugin is how you ship it. When you want to share a skill across repos or bundle a few together you package them as a plugin."

05 · 3 ปัญหาที่แย่ขึ้นเมื่อ loop ดีขึ้น

นี่คือ counterintuitive truth ที่สำคัญมาก: loop ที่ smooth ขึ้น ไม่ได้แปลว่าปัญหาลดลง มี 3 ปัญหาที่ยิ่ง loop เร็วยิ่งแย่ขึ้น ถ้าไม่มี safeguard

"Three problems actually get sharper as the loop gets better, not easier: Verification is still on you. Your understanding still rots if you allow it. The comfortable posture is the dangerous one."
แปล: มีสาม problem ที่ยิ่ง loop ดีขึ้นยิ่งแหลมคม ไม่ใช่ง่ายขึ้น: Verification ยังอยู่ที่คุณ ความเข้าใจยัง rot ถ้าคุณยอมให้มันเป็นอย่างนั้น และท่าทีที่สบายใจคือท่าทีที่อันตรายที่สุด
Verification Rot

loop ที่รัน unattended = loop ที่ทำ mistake unattended ด้วย verifier subagent บอกว่า "done" แต่นั่นเป็นแค่ claim ไม่ใช่ proof

Comprehension Debt

ยิ่ง loop ship code เร็วขึ้น ยิ่ง gap กว้างขึ้นระหว่างสิ่งที่ exist กับสิ่งที่คุณเข้าใจ Addy เรียกว่า "comprehension debt"

Cognitive Surrender

เมื่อ loop รันได้เอง น่าดึงดูดมากที่จะหยุดมี opinion รับอะไรก็ได้ที่ loop ให้ Addy เรียกว่า "cognitive surrender" — อันตรายที่สุด

⚠ Pedpro fix: ส่ง diff summary ใน Telegram ก่อน /approve ทุกครั้ง (+/- lines, reviewer verdict, should-fix items) เพื่อบังคับ human อ่าน changes จริงๆ ก่อนกด approve

06 · Long-running Agents — agent ที่ทำงานข้าม session

task บางอย่างใหญ่เกิน 1 session จะจบ Addy แนะนำ 3 pattern: Ralph Loop ป้องกัน agent หยุดก่อนเวลา, Test Ratchet ป้องกัน agent ลบ test, progress file ทำให้ session ถัดไป pick up ได้ทันที

"CLAUDE.md as a living plan the agent edits as it learns, the Ralph loop — a for loop that kicks the agent back into context whenever it claims completion and asks if it's really done. The agent itself is amnesiac, but the filesystem isn't."
แปล: CLAUDE.md เป็นแผนที่มีชีวิตที่ agent แก้ไขขณะเรียนรู้ Ralph loop คือ for loop ที่เตะ agent กลับเข้า context เมื่อใดก็ตามที่มัน claim ว่าเสร็จ และถามว่าเสร็จจริงๆ ไหม ตัว agent เองมี amnesia แต่ filesystem ไม่มี
Ralph Loop

intercept "done" claim ของ agent → re-inject prompt ใน fresh context พร้อม state จาก disk → agent ทำงานต่อ

Test Ratchet

Anthropic rule: "it is unacceptable to remove or edit tests" ป้องกัน failure mode ที่พบบ่อยที่สุด: agent ลบ test ที่ fail เพื่อให้ CI pass

Progress File

claude-progress.txt เขียนก่อน session จบ บอก "ทำถึงไหนแล้ว ยังเหลืออะไร" session ถัดไปอ่านแล้วทำงานต่อได้ทันที

⚠ Pedpro gap: task ใหญ่เช่น BotLine ยังไม่มี progress file convention session หลุดปุ๊บ agent ถัดไปเริ่มใหม่จาก scratch
## เพิ่มใน CLAUDE.md — 2 rules สำคัญ:

# Test Ratchet — ป้องกัน agent ลบ test เพื่อให้ CI pass
NEVER comment out, skip, or delete tests to make CI pass.
If a test fails, fix the code — not the test.

# Progress File — สำหรับ task ที่ span หลาย session
Before ending any multi-session task, write claude-progress.txt:
  Task: PEDPRO-XXX / Completed: [...] / Remaining: [...] / Blockers: [...]

07 · 60% Failure = Harness Gap — ไม่ใช่ model problem

เมื่อ agent ทำผิด คนส่วนใหญ่ blame model แล้วรอ version ใหม่ แต่จริงๆ 60% ของปัญหาอยู่ที่ harness ไม่ใช่ model — แก้ได้ตอนนี้เลย

"LangChain 2026 report: around 60% of agent failures trace to the harness rather than the model. Viv's team moved a coding agent from Top 30 to Top 5 by changing only the harness."
แปล: รายงาน LangChain 2026: ประมาณ 60% ของ agent failures มาจาก harness ไม่ใช่ model ทีมของ Viv ขยับ coding agent จากอันดับ 30 ไป Top 5 ด้วยการเปลี่ยนแค่ harness
Harness Problem — แก้ได้ตอนนี้ (~60%)

agent ไม่รู้ convention → เพิ่มใน CLAUDE.md | ทำ destructive action → เพิ่ม Hook | หยุดก่อนเสร็จ → Ralph loop

Model Problem — รอได้ (~40%)

reasoning ผิดในงานซับซ้อนมาก, hallucinate API ที่ไม่มีจริง, context window หมดเร็วเกินไป

#!/bin/bash
# .claude/hooks/post-tool-use.sh — enforcement layer
# รันหลัง Write/Edit tool ทุกครั้ง เป็นกำแพงด่านสุดท้าย

# TypeScript check
if npx vue-tsc --noEmit 2>&1 | grep -q "error TS"; then
  echo "HOOK FAIL: TypeScript errors found" >&2; exit 1
fi

# Test Ratchet — ห้าม agent ลบ/comment test
if git diff --staged | grep -E "^-.*\.(test|spec)\." | grep -v "^---"; then
  echo "HOOK FAIL: Tests deleted/modified — fix the code, not the test" >&2; exit 2
fi

# Bruno contract warning
if git diff --staged | grep -q "server/api/"; then
  echo "HOOK WARN: API file changed — run bruno-contract skill" >&2
fi

📖 Glossary — คำศัพท์ที่ใช้ในคู่มือนี้

คำศัพท์เทคนิคทั้งหมดที่ปรากฏใน Loop Engineering — เรียงตามหมวดหมู่ แต่ละคำมีคำแปลไทย คำอธิบาย และตัวอย่างใน context ของ Pedpro

Core Concepts — แนวคิดหลัก
Loop Engineering
Core
วิศวกรรม Loop
การออกแบบระบบที่ส่ง prompt ให้ AI agent แทนตัวคุณ แทนที่จะ prompt เอง ระบบวนซ้ำ (loop) จนกว่า condition จะเป็นจริง
เปรียบเทียบ: เหมือน factory line ที่คุณออกแบบสายพาน แต่ไม่ต้องยืนทำงานเอง
Harness
Core
สภาพแวดล้อมของ agent
ทุกสิ่งที่ wrap รอบ model — CLAUDE.md, skills, hooks, tools, sandboxes Harness ที่ดีทำให้ model ธรรมดาทำงานได้ดีกว่า model เก่งที่ไม่มี harness
Pedpro: CLAUDE.md + .claude/agents/ + heartbeat.mjs รวมกัน = harness
Prompt Engineering
Level 1
วิศวกรรม Prompt
การออกแบบ prompt ที่ดีเพื่อให้ agent ตอบถูกต้อง ระดับพื้นฐานที่สุด — ยังต้องมีคนพิมพ์ prompt ทุกครั้ง
การพิมพ์ "implement BotLine UI ตาม spec" ใน Claude Code คือ prompt engineering
Harness Engineering
Level 2
วิศวกรรม Harness
การออกแบบ "สภาพแวดล้อม" ที่ agent อาศัยอยู่ agent ฉลาดขึ้นมาก แต่คนยังต้อง trigger เอง ทำงานทีละ task
การเขียน CLAUDE.md, skills, hooks สำหรับ Pedpro คือ harness engineering
Agent Amnesia
Problem
ความจำสูญหายของ agent
ปัญหาพื้นฐาน: ทุก conversation เริ่มใหม่ agent ไม่รู้ว่าเมื่อวานทำอะไรไป ต้องแก้ด้วย external state
agent ถามว่า "PEDPRO-142 คืออะไร" ทั้งที่เพิ่งทำ task นั้นเสร็จในรอบก่อน
Intent Debt
Concept
หนี้ของ intent ที่ไม่เขียนไว้
สะสมขึ้นทุกครั้งที่ agent ต้อง "เดา" convention แทนที่จะอ่านจาก skill — ยิ่งมาก ยิ่งเสีย token และ hallucinate มากขึ้น
agent เดาว่า import path คือ @/components แต่จริงๆ Pedpro ใช้ page-local explicit import
Building Blocks — ส่วนประกอบ
Automation
Block 01
ระบบอัตโนมัติ / heartbeat
กลไกที่ทำให้ loop ตื่นขึ้นมาทำงานโดยไม่ต้องรอคนสั่ง มี 3 แบบ: schedule, event, condition
Pedpro: cron ทุก 5 นาที ดึง YouTrack ticket เหมือนหัวใจที่เต้นตาม rhythm
Worktree
Block 02
พื้นที่ทำงานแยกกัน
Git feature ที่สร้าง working directory แยกบน branch ต่างกัน ทำให้ agent หลายตัวทำงานพร้อมกันโดยไม่ conflict ไฟล์
agent A อยู่ใน ~/pedpro-wt/feat/PEDPRO-142 agent B อยู่ใน ~/pedpro-wt/feat/PEDPRO-143
Skill
Block 03
ทักษะ / ความรู้นำกลับมาใช้
folder ที่มี SKILL.md อธิบาย workflow ที่ agent ทำซ้ำบ่อย triggered โดย description match หรือเรียกตรงๆ
/scrutinize skill: รัน code review ก่อน merge ทุกครั้ง
Connector
Block 04
ตัวเชื่อมต่อ
สะพานระหว่าง agent กับ service ภายนอก ผ่าน MCP, REST, หรือ deterministic middleware loop ที่ไม่มี connector ทำงานได้แค่กับ filesystem
YouTrack MCP ให้ agent อ่าน/เขียน ticket โดยตรง ไม่ต้อง paste ID
Sub-agent

Technical Terms — คำศัพท์เทคนิค
MCP
Protocol
Model Context Protocol
มาตรฐานกลางที่ Anthropic กำหนดสำหรับการเชื่อม AI model กับ external services schema-typed, bidirectional
YouTrack, GitHub, Google Drive ล้วนมี MCP server ที่ Claude Code เชื่อมได้
Race Condition
Bug Pattern
การแย่งทรัพยากรพร้อมกัน
เกิดเมื่อ agent 2 ตัวพยายาม claim/modify สิ่งเดียวกันพร้อมกัน ป้องกันด้วย Claim step ก่อน dispatch
agent A และ B ทั้งคู่ claim PEDPRO-142 → เกิด race condition
Hallucination
AI Problem
การสร้างข้อมูลเท็จ
เมื่อ AI model "เดา" แล้วเอ้าท์พุตข้อมูลที่ไม่มีอยู่จริงด้วยความมั่นใจสูง ลดได้ด้วย skills, narrow scope, และ reviewer sub-agent
agent เรียก API endpoint ที่ไม่มีใน Bruno spec เพราะเดาจากชื่อ
Context Window
AI Term
หน้าต่าง context / ความจำ session
ขนาดสูงสุดของ text ที่ model เห็นในครั้งเดียว Claude Sonnet มี 200k tokens ยิ่ง context เยอะ ยิ่ง cost สูง
CLAUDE.md ยาวมาก = กิน context ทุก session ไม่ว่าจะทำงานอะไร
Context Rot
Problem
การเสื่อมคุณภาพเมื่อ context เต็ม
เมื่อ context window เต็มมากขึ้น model reasoning แย่ลง แก้ด้วย compaction, tool offloading, และ context resets
session ยาวๆ มักทำ agent ตอบผิดมากขึ้นเรื่อยๆ ใน turns ท้ายๆ
Hook
Mechanism
ตะขอ / script ที่รันอัตโนมัติ
script ที่รันอัตโนมัติตาม lifecycle event เช่น PostToolUse, PreToolUse ใช้ enforce rules แบบ deterministic ไม่ใช่ trust LLM
PostToolUse hook รัน vue-tsc ทุกครั้งที่ agent แก้ไฟล์
ADR
Documentation
Architecture Decision Record
บันทึก "ทำไมถึงตัดสินใจแบบนี้" สำหรับ architecture decisions สำคัญ ป้องกัน agent หรือคนตั้งคำถามซ้ำหรือ reverse decisions
docs/adr/001-flat-directory.md อธิบายว่าทำไม Pedpro ใช้ flat directory ไม่ใช่ Nuxt default
Deterministic
Property
ทำนายผลได้แน่นอน
code หรือ logic ที่ให้ผลลัพธ์เดิมเสมอสำหรับ input เดิม ตรงข้ามกับ LLM ที่อาจให้ผลต่างกันแต่ละครั้ง routing logic ต้อง deterministic เสมอ
if (ticket.type === 'Bug') use fixer-agent ←ผลลัพธ์แน่นอน 100%
Addy's Named Concepts — concept ที่ Addy ตั้งชื่อเอง
Triage Inbox
Pattern
กล่องข้อความ triage
pattern ที่ automation runs ที่พบงานส่ง result ไปที่ inbox สำหรับ human รอ ส่วน runs ที่ไม่พบอะไรก็ archive ตัวเอง ทำให้ human เห็นแค่สิ่งที่ต้องทำ
Pedpro: heartbeat พบ 3 tickets → notify Telegram "3 tickets queued" | ไม่พบ → log "no-work"
Ralph Loop
Pattern
วงจร Ralph / loop ที่ไม่ยอมให้หยุด
pattern สำหรับ long-running agents: loop ที่ intercept "done" claim ของ agent แล้ว re-inject prompt ใน fresh context พร้อม state จาก disk บังคับให้ทำต่อ
agent บอก "PEDPRO-142 เสร็จแล้ว" → checker พบ typecheck fail → ralph loop เตะกลับ "ยังไม่เสร็จ แก้ด้วย"
Test Ratchet
Rule
กลไกล็อค test
rule ที่ห้าม agent ลบหรือ comment out tests แม้ CI จะ fail แก้ปัญหา failure mode ที่พบบ่อยที่สุด: agent ลบ test ที่ fail เพื่อให้ CI "ผ่าน"
agent เจอ failing test → hook block → "fix the code, not the test" → agent แก้ logic แทน
Cognitive Surrender
Anti-pattern
การยอมแพ้ทางความคิด
เมื่อ loop ทำงานได้ดี human หยุดมี opinion รับ output ทุกอย่างโดยไม่ตั้งคำถาม อันตรายมากเพราะ loop ยังทำผิดได้
approve PR โดยไม่อ่าน diff เพราะ "reviewer agent บอกว่าผ่านแล้ว"
Comprehension Debt
Problem
หนี้ความเข้าใจ
ยิ่ง loop ship code เร็ว ยิ่ง gap กว้างขึ้นระหว่างสิ่งที่ exist ใน codebase กับสิ่งที่คุณเข้าใจจริงๆ อันตรายเมื่อมี bug ที่ต้องแก้
loop เขียน 10 components ใน 1 วัน แต่คุณอ่านแค่ PR title ไม่รู้ logic จริงๆ
Harness Gap
Concept
ช่องว่างของ harness
ช่องว่างระหว่างสิ่งที่ model ทำได้กับสิ่งที่คุณเห็น LangChain 2026 พบว่า 60% ของ agent failures มาจากนี้ ไม่ใช่ model ตัวเอง
agent เดา import path ผิดซ้ำๆ → ไม่ใช่ model โง่ → ขาด rule ใน CLAUDE.md

Pedpro

API Drift
Risk

handler ส่ง response field ชื่อ "userId" แต่ Bruno spec บอกควรเป็น "user_id"
