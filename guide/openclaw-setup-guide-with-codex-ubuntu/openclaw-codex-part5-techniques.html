00 ทำไมระบบถึงพัง — กรอบความคิด

ปัญหาเกือบทุกอย่างเกิดที่ "ชั้นใดชั้นหนึ่ง" ใน 4 ชั้น · รู้ว่ามันอยู่ชั้นไหน = แก้ได้เร็ว

จำโครงสร้างจาก Part 1 ได้ไหม — Channel → Provider → Model → Runtime · เวลาเกิดปัญหา ให้ถามตัวเองว่า "อาการนี้อยู่ชั้นไหน" เพราะแต่ละชั้นมีสาเหตุและจุดแก้คนละที่:

💬
CHANNEL
เข้า/ออกไม่ได้

token ผิด, intent ไม่เปิด, allowlist, privacy mode → bot เงียบ ไม่เห็นข้อความ

🔑
PROVIDER
auth/โควต้า

login หมดอายุ, โควต้าหมด, key ผิด → ตอบ error หรือไม่ตอบ

🧠
MODEL
not allowed

โมเดลไม่อยู่ใน allowlist, ref เขียนผิด → ปฏิเสธคำสั่งเงียบ ๆ

⚙️
RUNTIME
harness fail

Codex/ACP ใช้ไม่ได้, sandbox, exec policy → run ค้างหรือ fail closed

🧭
กฎทอง: Gateway โหลด config ตอน start แล้วถือไว้ใน memory — ดังนั้นอาการ "แก้ config แล้วไม่เปลี่ยน" เกือบทั้งหมดเกิดจากลืม restart ไม่ใช่ config ผิด · จุดนี้คือสาเหตุอันดับ 1 ของทุกปัญหา

01 แยก Model กับ Runtime ให้ขาด

คนเลือก model ผิดเพราะสับสนสองคำนี้ — มันคนละหน้าที่

คำตอบคำถามว่าตัวอย่าง
Modelใช้ "สมอง" ตัวไหน คุณภาพ/ราคา/context เท่าไหร่openai/gpt-5.5, anthropic/claude-opus-4-8
Runtimeใคร "รันลูป agent" ให้codex harness, claude-cli, openclaw, acp

เหตุผลที่ต้องแยก: คุณเลือก model openai/gpt-5.5 ได้ แล้วยังเลือกได้อีกว่าจะให้ Codex harness เป็นคนรัน (ค่าเริ่มต้น) หรือให้ built-in openclaw runtime รัน — model เดียวกันแต่พฤติกรรมการรัน tool/agentic loop ต่างกัน

  • OpenAI model → Codex runtime อัตโนมัติ (ถ้าไม่ override) — นี่คือเหตุผลที่ Part 1 ตั้งแค่ model ก็พอ
  • ถ้าระบุ runtime "codex" ชัด ๆ แล้ว harness ใช้ไม่ได้ → fail closed (ไม่แอบสลับ) เพราะออกแบบให้เห็น error ชัดดีกว่าเงียบ ๆ
  • เช็กว่าใครรันอยู่จริง: /status → บรรทัด Runtime บอก OpenAI Codex / CLI / ACP / OpenClaw Default

02 เลือก model ตัวไหน เพราะอะไร

หลักคิด: จับคู่ "ความยากของงาน" กับ "คุณภาพ/ราคา/ความเร็ว" ของโมเดล

ลักษณะงานเลือกเหตุผล (ทำไม)
เขียน/แก้โค้ด, agentic, เรียก tool เยอะCodex openai/gpt-5.5Codex harness ออกแบบมาเพื่อ coding + เรียก tool ได้เสถียร (สำคัญมากบน Discord กลุ่ม ดูข้อ 09)
วางแผน/สถาปัตยกรรม/คิดซับซ้อนโมเดลแรง + /think highงานวางแผนได้ประโยชน์จาก reasoning depth มากกว่าความเร็ว
งานซ้ำ ๆ ปริมาณมาก (จัด tag, สรุปสั้น)โมเดลเล็ก/เร็ว (mini/haiku-class)คุณภาพพอ + ถูกกว่ามาก = คุมต้นทุนเวลาทำพร้อมกันหลายงาน
งานของ subagentโมเดลถูกกว่า mainsubagent มี context แยกของตัวเอง = นับ token เพิ่ม ถ้าใช้โมเดลแพงทุกตัวจะบานปลาย
💸
ทำไม subagent แพงกว่าที่คิด: subagent แต่ละตัวมี context window แยก ของมันเอง — ยิ่ง spawn เยอะ ยิ่งคูณ token · นี่คือเหตุผลที่ควรตั้ง agents.defaults.subagents.model เป็นโมเดลถูก แล้วเก็บโมเดลแพงไว้ที่ main
~/.openclaw/openclaw.json — main แพง / sub ถูก
{
  agents: {
    defaults: {
      model: "openai/gpt-5.5",          // main: คุณภาพสูง
      subagents: {
        model: "openai/gpt-5.4-mini"     // sub: ถูกกว่า คุมต้นทุน
      }
    }
  }
}
📊
ดูโควต้าคงเหลือก่อนสั่งงานหนัก: /status โชว์ "% left" ของ provider ปัจจุบัน · ตั้ง models.providers.*.models[].cost เพื่อให้ระบบประเมินค่าใช้จ่ายต่อ run ได้

03 Thinking levels + โหมดเร็ว

ปรับ "ความลึกการคิด" กับ "ความเร็ว" แยกกัน — เลือกตามว่างานต้องการอะไร

/think — ความลึกของ reasoning

ในแชต
/think off      # ตอบไว ไม่ต้องคิดมาก (งานง่าย)
/think medium   # สมดุล
/think high     # คิดลึก (debug ยาก, ออกแบบ)
/think xhigh    # ลึกสุด (แลกกับช้า + token เยอะ)

ตัวเลือกที่ใช้ได้ขึ้นกับ model/provider · ส่งเดี่ยว ๆ = ติดถาวรกับ session, แทรกในข้อความ = ใช้ครั้งเดียว

/fast — ความเร็วระดับ service

/fast on ทำงานต่างกันตาม provider เพราะ map ไป API คนละแบบ:

  • OpenAI / Codex → service_tier=priority (จองคิวลำดับสูงบน endpoint)
  • Anthropic ตรง → service_tier=auto หรือ standard_only
🎚️
ทำไม think กับ fast ไม่ใช่เรื่องเดียวกัน: /think คุมว่าโมเดล "คิดกี่ขั้น" (คุณภาพ) ส่วน /fast คุมว่า request "ถูกจัดคิวเร็วแค่ไหน" (latency) — งานคิดหนักอาจตั้ง /think high แต่ /fast off ก็ได้

04 Subagent ทำงานยังไง (กลไก)

เข้าใจกลไกก่อน จะใช้ถูกและ debug เป็น

subagent คือ "agent run เบื้องหลัง" ที่ถูก spawn จาก run หลัก · มันรันใน session แยกของตัวเอง (agent:<id>:subagent:<uuid>) ทำงานเสร็จแล้ว announce ผลกลับ มาหา session ที่สั่ง

ทำไมต้องมี subagent

  • ขนานงาน — งานวิจัย/งานยาว/tool ช้า ไม่ block run หลัก (run หลักตอบเรื่องอื่นต่อได้)
  • แยกออกจากกัน (isolation) — แต่ละตัว context แยก เพื่อไม่ปนเปื้อนกัน
  • กัน misuse — subagent ไม่ได้ tool ส่งข้อความ/tool ระบบโดยปริยาย
🔇
ทำไม subagent โพสต์ข้อความให้ user เองไม่ได้: มันไม่มี message tool โดยเจตนา — มันแค่ส่งผลลัพธ์ (เป็น "หลักฐาน") กลับให้ agent หลัก แล้วagent หลักเป็นคนตัดสินใจว่าจะตอบ user ยังไง · เหตุผล: ป้องกัน subagent หลายตัวยิงข้อความซ้อนกันมั่ว และกันไม่ให้ output ของ child กลายเป็นคำสั่งที่ override นโยบายได้

ทำไมห้าม "poll" รอผล

sessions_spawn เป็น non-blocking — คืน run id ทันที แล้ว completion เป็นแบบ push (ผลวิ่งกลับมาเอง) ดังนั้นถ้าจะรอผล ให้ใช้ sessions_yield (จบ turn แล้วรอ event) ไม่ใช่วน loop เรียก /subagents list ซ้ำ ๆ

🔁
ทำไม poll loop เป็นปัญหา: มันเปลือง turn + token ไปเปล่า ๆ (ถามสถานะวนไป) ทั้งที่ผลจะ push กลับมาเองอยู่แล้ว · ที่แย่กว่าคือบางทีทำให้ agent "ติดอยู่ในลูปรอ" — ใช้ sessions_yield คือคำตอบที่ถูก

05 isolated vs fork — เลือกผิด = เปลือง

subagent มี 2 โหมด context · default คือ isolated และมีเหตุผล

โหมดใช้เมื่อพฤติกรรม + ทำไม
isolated (default)งานวิจัยใหม่, งานอิสระ, งานที่บรีฟด้วยข้อความได้หมดสร้าง transcript เปล่าให้ child = ประหยัด token เพราะไม่ลากบทสนทนาเดิมมาด้วย
forkงานที่ต้องพึ่ง context/ผล tool/คำสั่งละเอียดที่อยู่ใน transcript เดิมแตกแขนง transcript ปัจจุบันเข้า child = แพงกว่า ใช้เท่าที่จำเป็น
🌱
ทำไม default เป็น isolated: เพราะการ fork ลาก context ทั้งก้อนเข้าไป = token เพิ่มทันที · เอกสารแนะนำให้ เขียน task prompt ให้ชัดแทนการ fork — fork มีไว้สำหรับงานที่อ้างอิงบทสนทนาก่อนหน้าจริง ๆ ไม่ใช่ทางลัดแทนการเขียนโจทย์ให้ดี
📄
subagent เห็นไฟล์ context แค่ไหน: มัน inject เฉพาะ AGENTS.md + TOOLS.md เท่านั้น — ไม่เอา persona/identity/USER.md/MEMORY.md มาด้วย · เหตุผล: ไม่ให้ child โคลนตัวตนและ memory ของ parent (กันความซับซ้อน + token)

06 Orchestrator pattern + คุมต้นทุน

เมื่อต้องการ "หัวหน้างาน" ที่กระจายงานให้ลูกน้องหลายตัว

ค่าเริ่มต้น subagent spawn ลูกของตัวเองไม่ได้ (maxSpawnDepth: 1) · ตั้งเป็น 2 เพื่อเปิด orchestrator pattern: main → orchestrator → worker

~/.openclaw/openclaw.json — orchestrator
{
  agents: { defaults: { subagents: {
    maxSpawnDepth: 2,        // ให้ sub spawn ลูกได้ 1 ชั้น (default 1)
    maxChildrenPerAgent: 5, // ลูกพร้อมกันสูงสุด/ตัว (กัน fan-out บาน)
    maxConcurrent: 8,       // เพดานขนานรวม (safety valve)
    runTimeoutSeconds: 900   // timeout ต่อ run (0 = ไม่จำกัด)
  } } }
}
Depthบทบาทspawn ลูกได้ไหม
0Main agentได้เสมอ
1Sub-agent (เป็น orchestrator ถ้า depth≥2)ได้เฉพาะเมื่อ maxSpawnDepth ≥ 2
2Worker (ใบ)ไม่ได้เด็ดขาด
  • Cascade stop: /stop ที่ main หยุด depth-1 ทั้งหมด และไล่หยุด depth-2 ให้ด้วย — เหตุผล: กัน worker ค้างทำงานต่อหลังหัวหน้าถูกสั่งหยุด
  • maxChildrenPerAgent กัน "runaway fan-out" — orchestrator ตัวเดียวสร้างลูก 50 ตัวพร้อมกันไม่ได้
  • depth สูงสุด 5 แต่ แนะนำแค่ 2 สำหรับเคสส่วนใหญ่ (ลึกกว่านั้น debug ยากและซับซ้อนเกินจำเป็น)

07 กับดัก subagent ที่ต้องระวัง

สาเหตุที่ subagent "หาย" หรือ "ไม่ตอบกลับ" — และทำไม

subagent เสร็จแล้วแต่ไม่มีผลกลับมาANNOUNCE LOST
สาเหตุGateway restart ระหว่างที่ผล "กำลังจะ announce กลับ"
ทำไมการ announce เป็น best-effort และ pending announce อยู่ใน memory ของ process — รีสตาร์ทเมื่อไหร่ก็หาย
แก้/เลี่ยงอย่ารีสตาร์ทระหว่างมี subagent run ค้าง · งานสำคัญให้ใช้ cleanup: "keep" แล้วตามดู transcript ด้วย /subagents info
agent ติดอยู่ในลูป "รอ subagent"POLL LOOP
สาเหตุโมเดลวนเรียก /subagents list / sleep เพื่อรอผล
ทำไมcompletion เป็นแบบ push อยู่แล้ว — การ poll จึงเปล่าประโยชน์และกิน turn/token
แก้ใช้ sessions_yield (จบ turn รอ event) แทน poll · ใส่คำสั่งนี้ใน AGENTS.md ให้ agent รู้ว่าห้าม poll
spawn ไม่ได้ / โดนปฏิเสธTOOL POLICY
สาเหตุsessions_spawn ไม่อยู่ใน tool profile ของ agent นั้น
ทำไมprofile messaging ไม่เปิด spawn (มีเฉพาะ coding/full) — เพื่อกัน agent ที่ไว้แชตเฉย ๆ ไป spawn งานหนัก
แก้ตั้ง tools.profile: "coding" หรือ tools.alsoAllow: ["sessions_spawn","sessions_yield","subagents"] · เช็กด้วย /tools
🗄️
subagent session หายไปเอง? ปกติ — มัน auto-archive หลัง archiveAfterMinutes (default 60 นาที) โดย rename transcript เป็น *.deleted.<timestamp> ไว้ (ไม่ได้ลบจริง) เพื่อไม่ให้ session เก่าค้างเต็มระบบ

08 เทคนิคประหยัด token + เร่งความเร็ว

เข้าใจว่า "อะไรทำให้ช้า/แพง" แล้วแก้ที่ต้นเหตุ

ทำไม session ยิ่งคุยยิ่งช้า + แพง

ทุกข้อความใน session ถูกส่งกลับเข้าโมเดลเป็น context — ยิ่งคุยยาว context ยิ่งบวม ทำให้ช้าลงและ token ต่อ turn สูงขึ้นเรื่อย ๆ

  • /new เมื่อขึ้นงานใหม่ที่ไม่เกี่ยวกับงานเดิม → ทำไม: ตัด context เก่าที่ไม่จำเป็นออก ไม่ลากไปคิดด้วยทุก turn
  • /compact เมื่องานเดียวยาวมาก → ทำไม: ย่อบทสนทนาให้สั้นลงแต่ยังเก็บสาระ
  • /context detail เพื่อดูว่า "อะไรกินที่" → ทำไม: เห็นขนาดต่อไฟล์/tool/skill ว่าตัวไหนบวม
  • ปิด tool/skill ที่ไม่ใช้ + ใช้ MCP เท่าที่จำเป็น → ทำไม: tool definition ก็กิน context เหมือนกัน
  • งานขนาน → ใช้ subagent isolated + โมเดลถูก → ทำไม: ไม่ลาก context หลักไปคูณ
📐
หลักวาง AGENTS.md: ใส่ convention/กฎที่ต้องใช้ทุก session ไว้ในไฟล์นี้ (lean เข้าไว้) — มัน inject ทุก session แต่ถ้ายาวเกินก็กิน context ทุกครั้ง · เก็บ note ยาว ๆ ไว้ใน MEMORY.md แล้วให้ agent ดึงเมื่อต้องใช้แทน

09 ตารางแก้ปัญหาเชิงสาเหตุ

อาการ → สาเหตุ → ทำไมถึงเกิด → วิธีแก้ · เรียงตามชั้นที่เกิด

แก้ config แล้วไม่มีอะไรเปลี่ยนอันดับ 1
สาเหตุลืม restart Gateway
ทำไมGateway อ่าน config เข้า memory ตอน start แล้วถือไว้ — ไม่ได้อ่านไฟล์ใหม่ทุกข้อความ
แก้openclaw gateway restart ทุกครั้งหลังแก้ openclaw.json
Discord: bot พิมพ์อยู่แต่ไม่มีข้อความโผล่CHANNEL · MODEL
สาเหตุโมเดลตอบแบบ private ไม่เรียก message tool (didSendViaMessagingTool: false)
ทำไมในห้องกลุ่ม Discord final reply เป็น private โดยปริยาย — agent ต้องเรียก tool เองถึงจะโพสต์ · ถ้าโมเดลเรียก tool ไม่เก่งก็จะตอบลอย ๆ
