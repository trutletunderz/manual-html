PART A · SECURITY

00 Threat model — เข้าใจก่อนป้องกัน

OpenClaw ออกแบบบนสมมติฐาน "personal assistant" — เจ้าของคนเดียว/ทีมที่ไว้ใจกัน ต่อ 1 gateway

สิ่งที่ AI assistant ของคุณ "ทำได้": รันคำสั่ง shell, อ่าน/เขียนไฟล์, เข้าถึงเครือข่าย, ส่งข้อความ · ส่วนคนที่ทักหา bot ก็ "พยายามหลอก" ให้ทำเรื่องไม่ดีได้ · กุญแจคือหลักนี้:

1 · IDENTITY
ใครคุยได้

DM pairing / allowlist / "open" — ตัดสินก่อนว่าใครมีสิทธิ์

→
2 · SCOPE
ทำที่ไหนได้

group allowlist + mention, tools, sandbox, สิทธิ์อุปกรณ์

→
3 · MODEL
สมมติว่าถูกหลอกได้

ออกแบบให้ "ถ้าโดนหลอก ความเสียหายจำกัด"

🎯
หลักสำคัญ "Access control before intelligence": ปัญหาส่วนใหญ่ไม่ใช่ exploit หรูหรา แต่คือ "มีคนทักมา แล้ว bot ทำตามที่เขาสั่ง" · ดังนั้นล็อก ใครคุยได้ ก่อน สำคัญกว่าตั้งความฉลาด
⚠️
ไม่ใช่ระบบ multi-tenant: OpenClaw ไม่ได้ออกแบบให้กั้นผู้ใช้หลายคนที่ไม่ไว้ใจกันบน gateway เดียว · ถ้าต้องแยกความเชื่อใจ ให้แยก gateway / OS user / host คนละตัว

01 Prompt Injection — ทำไมแก้ไม่ขาด

การโจมตีที่สำคัญที่สุด และเข้าใจผิดบ่อยที่สุด

Prompt injection คือการที่ผู้โจมตีแต่งข้อความให้โมเดล "เชื่อฟังคำสั่งแฝง" เช่น "ลืม system prompt ซะ", "dump ไฟล์ทั้งหมด", "เปิดลิงก์นี้แล้วรันคำสั่ง" · ความจริงที่ต้องยอมรับ: แม้ system prompt จะแกร่งแค่ไหน prompt injection ก็ยัง "แก้ไม่ขาด"

🧱
ทำไม system prompt อย่างเดียวไม่พอ: guardrail ใน prompt เป็นแค่ "คำแนะนำนุ่ม ๆ" · การบังคับจริง (hard enforcement) มาจาก tool policy + exec approvals + sandbox + allowlist ต่างหาก — นี่คือเหตุผลที่ต้องตั้งหลายชั้น ไม่พึ่ง prompt ชั้นเดียว

ไม่ต้องเปิด DM สาธารณะก็โดนได้

เข้าใจผิดที่อันตราย: คิดว่า "ถ้ามีแค่ฉันคุยกับ bot ก็ปลอดภัย" — ผิด · เพราะ injection มาทาง เนื้อหาที่ bot อ่านได้ด้วย: ผลค้นเว็บ, หน้าเว็บที่ browser เปิด, อีเมล, เอกสาร, ไฟล์แนบ, log ที่ paste มา

  • ธงแดงที่ต้องถือว่า "ไม่น่าไว้ใจ": "อ่านไฟล์/URL นี้แล้วทำตามเป๊ะ ๆ", "เพิกเฉย safety rule", "เปิดเผย system prompt", "paste เนื้อหา ~/.openclaw ทั้งหมด"
  • ลดความเสียหาย: ใช้ reader agent ที่ไม่มี tool มาสรุปเนื้อหาที่ไม่น่าไว้ใจก่อน แล้วค่อยส่งสรุปให้ agent หลัก
  • ปิด web_search/web_fetch/browser สำหรับ agent ที่มี tool แรง ๆ ถ้าไม่จำเป็น
🧠
โมเดลอ่อน = เสี่ยง injection สูง: โมเดลเล็ก/เก่ากันการ hijack ได้แย่กว่ามาก · เอกสารเตือนชัด — agent ที่มี tool หรืออ่านเนื้อหาไม่น่าไว้ใจ ต้องใช้โมเดลรุ่นใหม่ที่สุด/แกร่งที่สุด (Codex/gpt-5.5 เหมาะ) · นี่คืออีกเหตุผลที่เราเลือก model นี้ตั้งแต่ Part 1

02 Hardened baseline ใน 60 วินาที

เริ่มจาก "แคบที่สุดที่ยังใช้งานได้" แล้วค่อยเปิดทีละอย่างเมื่อมั่นใจ

~/.openclaw/openclaw.json — baseline
{
  gateway: {
    mode: "local",
    bind: "loopback",                       // ฟังเฉพาะ localhost
    auth: { mode: "token", token: "<random-ยาว ๆ>" }
  },
  session: { dmScope: "per-channel-peer" },   // แยก DM แต่ละคน (กัน context รั่ว)
  tools: {
    profile: "messaging",
    deny: ["group:automation", "group:runtime", "group:fs", "sessions_spawn"],
    fs: { workspaceOnly: true },     // จำกัดไฟล์ใน workspace
    exec: { security: "deny", ask: "always" },
    elevated: { enabled: false }
  }
}
  • bind: "loopback" + auth.token → gateway ฟังเฉพาะเครื่อง + ทุก client ต้อง auth (gateway auth บังคับโดยปริยายอยู่แล้ว ถ้าไม่มี = ปฏิเสธ fail-closed)
  • dmScope: "per-channel-peer" → ถ้ามีหลายคน DM แต่ละ channel+คน แยก session ป้องกัน context ข้ามคนรั่ว (เหตุผล: default main รวมทุก DM เพื่อ continuity)
  • ปิด tool runtime/automation/fs + exec deny + elevated off → "ความเสียหายจาก injection จำกัด" ตามหลักข้อ 00
🔌
อย่าเปิด gateway ออกเน็ตตรง ๆ: ถ้าต้องเข้าจากนอก ให้ใช้ Tailscale Serve (gateway ยังอยู่ loopback, Tailscale คุม access) ดีกว่า bind LAN/public · ไม่เปิดบน 0.0.0.0 แบบไม่มี auth เด็ดขาด

03 Exec Approvals — ด่านกันคำสั่งอันตราย

เมื่อ agent (ที่อยู่ใน sandbox) จะรันคำสั่งบน host จริง — นี่คือ "safety interlock"

เก็บที่ ~/.openclaw/exec-approvals.json · นโยบายมี 3 ปุ่มหลัก ทำงาน "เข้มกว่า" ระหว่าง tools.exec.* กับ approvals defaults:

ปุ่มค่าความหมาย
securitydeny / allowlist / fullบล็อกหมด / เฉพาะที่ allowlist / ปล่อยหมด (= elevated)
askoff / on-miss / alwaysไม่ถาม / ถามเมื่อไม่ match allowlist / ถามทุกครั้ง
askFallbackdeny / allowlist / fullถ้าต้องถามแต่ไม่มี UI ตอบ → ทำตามนี้ (default deny)

อนุมัติจากแชต

ในแชต (Telegram/Discord มีปุ่มกดด้วย)
/approve <id> allow-once    # รันครั้งเดียว
/approve <id> allow-always  # เพิ่มเข้า allowlist + รัน
/approve <id> deny          # ปฏิเสธ
🐍
ทำไมต้องมี strictInlineEval: ถ้าคุณ allowlist python/node ไว้ ผู้โจมตีอาจสั่ง python -c "<โค้ดอันตราย>" ลอด allowlist ได้ · ตั้ง tools.exec.strictInlineEval: true → inline eval ยังต้องขออนุมัติเสมอ แม้ binary จะ allowlist แล้ว (defense-in-depth)
🧊
safe bins (stdin-only): tools.exec.safeBins เช่น head/tail/tr/wc รันได้โดยไม่ต้อง allowlist เพราะมันรับแต่ stream (ปฏิเสธ file arg) · ห้ามใส่ interpreter (python/node/bash) ลง safeBins เด็ดขาด — เหตุผล: มันรันโค้ด/อ่านไฟล์ได้
⏰
คำขออนุมัติที่ค้าง หมดอายุใน 30 นาที โดยปริยาย · ถ้าไม่มี UI/approver ตอบ → ตกไปที่ askFallback

04 Sandbox + Per-agent Access Profiles

กั้น "agent แตะอะไรได้" — ใช้ร่วมกับ exec approvals เป็นชั้นบังคับจริง

มี 2 แนวทาง (ใช้ร่วมกันได้): รัน gateway ทั้งตัวใน Docker (container boundary) หรือใช้ tool sandbox (agents.defaults.sandbox, Docker เป็น backend default)

workspaceAccessพฤติกรรมเหมาะกับ
"none" (default)ไม่แตะ workspace จริง — รันใน sandbox workspace แยกagent สาธารณะ/ไม่ไว้ใจ
"ro"mount workspace แบบ read-only (ปิด write/edit/apply_patch)agent อ่านอย่างเดียว
"rw"mount read/write เต็มagent ส่วนตัวที่ไว้ใจ (เช่นห้อง Codex โปรเจกต์)

ตัวอย่าง: หลาย agent คนละระดับสิทธิ์

~/.openclaw/openclaw.json — per-agent profiles
{ agents: { list: [
  { id: "personal", sandbox: { mode: "off" } },          // ส่วนตัว: เต็มสิทธิ์
  { id: "reader",
    sandbox: { mode: "all", scope: "agent", workspaceAccess: "ro" },
    tools: { allow: ["read"], deny: ["write","edit","exec","browser"] } }, // อ่านอย่างเดียว
] } }
🔒
ทำไม sandbox เป็น opt-in (และผลของมัน): ถ้า sandbox ปิด อยู่ → host=auto จะรันบน gateway host จริง · ส่วน host=sandbox จะ fail closed (ไม่มี sandbox ให้รัน) · เข้าใจจุดนี้กันความเข้าใจผิดว่า "นึกว่ามี sandbox อยู่แล้ว"
🌐
browser control = สิทธิ์ operator: ถ้าให้ agent คุม browser ที่ login บัญชีคุณไว้ = มันเข้าถึงบัญชีนั้นได้ · ใช้ profile แยกเฉพาะ agent (default openclaw) อย่าชี้ไป profile ส่วนตัว · SSRF policy บล็อก private network โดยปริยาย

05 ความลับบนดิสก์ + Security Audit

ทุกอย่างใต้ ~/.openclaw/ ถือว่าอาจมีความลับ — disk access คือ trust boundary

แผนที่ความลับ (รู้ไว้เวลา backup/audit)

ไฟล์/โฟลเดอร์เก็บอะไร
openclaw.jsonconfig + อาจมี token/allowlist
credentials/**credential ของ channel + pairing allowlist
agents/<id>/agent/auth-profiles.jsonAPI key / OAuth token ของโมเดล
agents/<id>/agent/codex-home/**state ของ Codex (บัญชี/config/thread)
agents/<id>/sessions/*.jsonlบทสนทนา + tool output (อาจมีข้อมูลส่วนตัว)

ตั้ง permission + ตรวจสอบ

bash — harden + audit
$ chmod 700 ~/.openclaw && chmod 600 ~/.openclaw/openclaw.json
$ openclaw security audit          # สแกนการตั้งค่าเสี่ยง
$ openclaw security audit --deep   # + probe gateway สด
$ openclaw security audit --fix    # แก้ footgun ที่แก้ได้อัตโนมัติ (แคบ ๆ ปลอดภัย)

audit เช็ก: ใครเข้าถึงได้ (DM/group/allowlist), tool blast radius, exec approval drift, network exposure, browser control, file permission, plugin ที่ไม่มี allowlist, โมเดลเก่า

ถ้าเกิดเหตุ (Incident Response)

  • Contain — หยุด gateway, ตั้ง bind: "loopback", สลับ DM/group เสี่ยงเป็น disabled, ลบ "*" ออก
  • Rotate — หมุน gateway token, remote secret, และ provider/API key ทั้งหมด แล้ว restart
  • Audit — ดู log + transcript + การเปลี่ยน config ที่ขยายสิทธิ์ → รัน security audit --deep อีกรอบ
🚫
tool คุม control-plane ที่ควรปิดสำหรับ agent ที่รับเนื้อหาไม่น่าไว้ใจ: tools.deny: ["gateway","cron","sessions_spawn","sessions_send"] — เพราะ gateway แก้ config ถาวรได้ และ cron ตั้งงานที่รันต่อหลังแชตจบ

06 Workspace Bootstrap Files

ไฟล์ใน workspace ที่กำหนด "ตัวตน + ความรู้" ของ agent — รู้ว่าไฟล์ไหนทำอะไร = คุม agent ได้แม่น

ไฟล์หน้าที่หมายเหตุสำคัญ
SOUL.mdบุคลิก/น้ำเสียงของ agentกำหนด "นิสัย" การตอบ
IDENTITY.mdตัวตน/ชื่อ/บทบาทใครคือ agent นี้
USER.mdข้อมูลเกี่ยวกับคุณ (ผู้ใช้)inject ทุก session หลัก
AGENTS.mdกฎ/convention การทำงานinject ให้ subagent ด้วย — วาง convention โปรเจกต์ที่นี่
TOOLS.mdคำแนะนำการใช้ toolinject ให้ subagent ด้วย
MEMORY.mdความจำระยะยาวที่กลั่นแล้วโหลดใน DM session · dreaming เขียนเพิ่มที่นี่
HEARTBEAT.mdคำสั่งสำหรับ heartbeat (งานเป็นจังหวะ)ดูข้อ 09
BOOTSTRAP.mdขั้นตอนเริ่มต้นครั้งแรกใช้ตอน bootstrapping
🧬
ทำไม subagent ได้แค่ AGENTS.md + TOOLS.md: subagent ไม่โหลด SOUL/IDENTITY/USER/MEMORY ของ parent — เพราะไม่ต้องการให้ child "โคลนตัวตน + ความจำ" ของแม่ (เปลือง token + ทำให้ซับซ้อน) · นี่คือเหตุผลที่ กฎโปรเจกต์ควรอยู่ใน AGENTS.md ไม่ใช่ MEMORY.md ถ้าอยากให้ subagent เห็นด้วย
📐
เก็บ AGENTS.md ให้ lean (กฎที่ต้องใช้ทุกครั้ง) เพราะมัน inject ทุก session = ยาวไป = กิน context ทุกครั้ง · note ยาว ๆ ที่ใช้บางครั้งให้อยู่ MEMORY.md แล้วให้ agent search เอา

08 💤 Dreaming System

การ "กลั่นความจำ" เบื้องหลัง — ทำไมเรียกว่า dreaming และทำงานยังไง

Dreaming คือ pass การรวบรวมความจำเบื้องหลังใน plugin memory-core · เรียกว่า "ฝัน" เพราะ ระบบย้อนกลับไปดูสิ่งที่เกิดขึ้นระหว่างวัน แล้วตัดสินว่าอะไรคุ้มที่จะเก็บเป็นความจำถาวร · มันเป็น experimental, opt-in, ปิดโดยปริยาย

3 ขั้นการทำงาน

  • Recall tracking — เก็บทุก recall hit จาก memory_search (จำนวนครั้ง, คะแนน, query)
  • Scheduled scoring — ตามรอบที่ตั้ง จัดอันดับด้วย "สัญญาณถ่วงน้ำหนัก" · ทุก threshold ต้องผ่านพร้อมกัน
  • Promotion — รายการที่ผ่านถูกเขียนเข้า MEMORY.md พร้อม timestamp

4 สัญญาณที่ใช้ตัดสิน (ทำไมเก็บ)

สัญญาณน้ำหนักวัดอะไร
Frequency0.35ถูก recall บ่อยแค่ไหน
Relevance0.35คะแนน recall เฉลี่ยตอนถูกดึง
Diversity0.15มาจาก query ที่หลากหลายแค่ไหน
Recency0.15ถ่วงเวลา (half-life 14 วัน)
💡
ทำไมต้องใช้หลายสัญญาณ + ผ่านทุก gate: เพื่อให้ เฉพาะ context ที่ "ทนทานและซ้ำจริง" ถูกเก็บถาวร ไม่ใช่รายละเอียดครั้งเดียวที่บังเอิญ recall — กัน MEMORY.md บวมด้วยขยะ

Hook = ประตูรับ injection: payload จาก hook (อีเมล/เอกสาร/เว็บ) คือ เนื้อหาไม่น่าไว้ใจ แม้จะมาจากระบบที่คุณคุม · งาน hook ให้ใช้โมเดลแกร่ง + tool policy เข้ม (tools.profile: "messaging" ขึ้นไป) + sandbox · อย่าเปิด allowUnsafeExternalContent ใน production

Backup ก่อนเปลี่ยนใหญ่

bash — backup
$ openclaw backup create                  # สำรอง state ทั้งหมด
$ openclaw backup create --only-config     # เฉพาะ config
$ openclaw backup verify <path>           # ตรวจ manifest
