01 Directives — ปรับพฤติกรรม Codex

ตัวปรับการคิด/ความเร็ว/การแสดงผลของ agent · ส่งเดี่ยว ๆ = ติดถาวร, แทรก = ครั้งเดียว

/think off|low|medium|high|xhighระดับการคิด (reasoning) · alias /t
/fast on|off|statusโหมดเร็ว · กับ OpenAI/Codex = map เป็น service_tier=priority
/model <name>สลับโมเดลทันที เช่น /model openai/gpt-5.5 · บน Discord เปิด picker
/reasoning on|off|streamโชว์ reasoning แยกข้อความ · ระวังในกลุ่ม เผยความคิดภายใน
/verbose on|full|offเพิ่มรายละเอียด (debug) · ปกติปิดไว้ · alias /v
/elevated on|off|ask|fullโหมดสิทธิ์สูง · full = ข้าม exec approval (ระวัง!)
/queue <mode>จัดคิวข้อความ เช่น debounce:2s cap:25 drop:summarize
ในแชต — ตัวอย่าง
/think high                 # ส่งเดี่ยว = session คิดละเอียดขึ้นถาวร
/think high ช่วย refactor ฟังก์ชันนี้   # แทรก = ใช้แค่ครั้งนี้
⚠️
/reasoning และ /verbose ในห้องกลุ่มอาจเผย reasoning/tool output ที่ไม่ตั้งใจให้คนอื่นเห็น — ในกลุ่มควรปิดไว้

02 คำสั่งจัดการ session & บริบท

คุม context window, รีเซ็ต, ตรวจสถานะ

/new [model]เริ่ม session ใหม่ (ล้าง context) · ใส่ชื่อโมเดลพ่วงได้ · alias /reset
/compact [คำสั่ง]ย่อ context ที่ยาวเกินให้สั้นลง (text-only)
/context [list|detail|json]ดูว่าตอนนี้ context ประกอบด้วยอะไร + ขนาดต่อไฟล์/tool/skill
/statusสถานะ runtime + uptime + โควต้า provider (เช่น "Codex 80% left")
/usage off|tokens|full|costโชว์ token/ค่าใช้จ่ายต่อคำตอบ
/tools [compact|verbose]ดูว่า agent ใช้ tool อะไรได้บ้างตอนนี้
/btw <คำถาม>ถามแทรกแบบไม่กระทบ context หลัก (one-shot ไม่บันทึก)
/stopยกเลิก run ที่กำลังทำงานใน session นี้
/export-session [path]export session เป็น HTML พร้อม system prompt · alias /export
🧹
นิสัยที่ดี: งานใหม่ที่ไม่เกี่ยวกับงานเดิม → /new ก่อน เพื่อไม่ให้ context เก่าปนเปื้อนและเปลือง token · งานยาวที่ context บวม → /compact

03 คำสั่ง Codex โดยตรง

เมื่อ bundled Codex plugin เปิดอยู่ ใช้ surface คำสั่ง /codex ควบคุม Codex แบบ native

/codex bindผูก session ปัจจุบันเข้ากับ Codex thread
/codex threadsดูรายการ Codex thread ที่มี
/codex resumeกลับไปทำงานต่อใน thread เดิม
/codex steer <ข้อความ>แทรกคำแนะนำเข้าไประหว่าง Codex กำลังรันอยู่
/codex stopหยุด Codex run ปัจจุบัน

เช็กว่าใครกำลังรัน session อยู่ด้วย /status — บรรทัด Runtime จะบอกว่าเป็น OpenAI Codex, CLI backend, ACP backend หรือ OpenClaw Default

ในแชต — สลับโมเดล + ดู runtime
/model openai/gpt-5.5    # ใช้ Codex (GPT-5.5)
/status                  # ดู Runtime: OpenAI Codex + โควต้าคงเหลือ
/model status            # ดู provider/endpoint/auth
🧭
ใช้ ACP เฉพาะตอนต้องการจริง ๆ (เช่นผูกห้องในโปรเจกต์แบบ Part 3 #09) · งานทั่วไปให้คุม Codex ผ่าน surface /codex ... แบบ native

04 หลาย Agent + Subagents + Routing

กระจายงานออกเป็นหลายตัว แล้วคุมแต่ละตัว — หัวใจของ "multi-agent orchestration"

คำสั่งคุม subagent (ในแชต)

/subagents list|info|logดูรายการ/รายละเอียด/log ของ subagent ใน session นี้
/subagents spawnสร้าง subagent ใหม่ให้ทำงานย่อย
/steer <id|#> <ข้อความ>แทรกคำแนะนำให้ subagent ที่กำลังรัน · alias /tell
/kill <id|#|all>หยุด subagent ทันที (ไม่มีถามยืนยัน)
/agentsดู agent ที่ผูกกับ thread ใน session นี้

กำหนด agent หลายตัว + model routing

ตั้งหลาย agent ที่ใช้คนละโมเดล แล้ว route ตามห้อง/role ได้ — ตัวอย่างแนวทางจัดสรรโมเดล:

~/.openclaw/openclaw.json — multi-agent
{
  agents: {
    list: [
      { id: "coder",    model: "openai/gpt-5.5" },        // งานเขียนโค้ด (Codex)
      { id: "planner",  model: "anthropic/claude-opus-4-8" }, // งานวางแผน/สถาปัตยกรรม
      { id: "router",   model: "anthropic/claude-haiku-4-5" }   // งานเบา ปริมาณเยอะ
    ]
  }
}
  • กฎทั่วไป: โมเดลแรง (Opus/GPT-5.5) ไว้วางแผน/โค้ด · โมเดลเบา (Haiku/mini) ไว้งานซ้ำ ๆ ปริมาณมาก = คุมต้นทุน
  • maxConcurrent (ค่าเริ่มต้น 4) คุมจำนวน run ขนานข้าม session
  • route ตาม role บน Discord ได้ด้วย bindings[].match.roles (Part 3)
🎯
อย่ารีบทำ multi-agent: เอกสารทางการเตือนว่าให้ใช้ single agent ที่ตั้งดี ๆ ให้สุดก่อน — เพิ่ม agent เฉพาะเมื่อมีเหตุผลชัด เพราะแต่ละตัวเพิ่มความซับซ้อนในการ debug

05 Exec + Approvals (ความปลอดภัย)

เมื่อ agent ขอรันคำสั่งบนเครื่อง — คุมว่าจะให้รันได้แค่ไหน ต้องอนุมัติไหม

/exec host=<...> security=<...> ask=<...>ตั้งนโยบาย exec · ส่ง /exec เปล่าเพื่อดูค่าปัจจุบัน
/approve <id> <decision>อนุมัติ/ปฏิเสธคำขอ exec ที่ค้างอยู่
/elevated on|off|ask|fullระดับสิทธิ์ · full = ข้าม approval
พารามิเตอร์ /execค่าความหมาย
securitydeny / allowlist / fullระดับการอนุญาตคำสั่ง
askoff / on-miss / alwaysถามอนุมัติเมื่อไหร่
hostauto / sandbox / gateway / nodeรันที่ไหน
🚨
อย่าตั้ง allow-all กับ exec/diagnostics โดยเฉพาะคำสั่งที่ดึงข้อมูลระบบ · ให้ตั้ง ask=on-miss หรือ always ในงานที่เสี่ยง · บน Discord/Telegram จะมีปุ่มอนุมัติให้กดยืนยัน (หมดอายุใน 30 นาที)

06 AgentSkills + ClawHub

เพิ่มความสามารถใหม่ ๆ ให้ agent ด้วย plugin/skill จากชุมชน (มีกว่า 100 ตัว)

ติดตั้งจาก ClawHub

bash — ติดตั้ง plugin/skill
$ openclaw plugins install clawhub:<pkg>   # จาก ClawHub
$ openclaw plugins install ./local-path     # จาก path/archive/npm
$ openclaw plugins list

เรียกใช้ skill ในแชต

/skill <name> [input]รัน skill ตามชื่อ
/plugins list|enable|disableดู/เปิด/ปิด plugin (owner-only · ต้อง commands.plugins: true)

skill ที่ตั้ง user-invocable จะกลายเป็น slash command อัตโนมัติ (ชื่อ sanitize เป็น a-z0-9_) — บางตัวประกาศ command-dispatch: tool ให้รันตรงเข้า tool โดยไม่ผ่านโมเดล (เร็ว+แน่นอน)

🧩
เปิด /mcp set ... เพื่อต่อ MCP server (ต้อง commands.mcp: true) — สาย dev ต่อ MCP ของโปรเจกต์เข้ามาให้ agent เรียกได้ เช่นเดียวกับที่คุณคุ้นใน Claude Code

07 Cron + Automation

ให้ agent ทำงานเองตามเวลา หรือทำงานเบื้องหลัง

bash — cron
$ openclaw cron list
$ openclaw cron add      # สร้างงานตามเวลา (รองรับ --tools)
$ openclaw cron edit|rm|enable|disable|run
$ openclaw cron status
  • Scheduled tasks (cron) — งานตามเวลา เช่นสรุปข่าวทุกเช้า, เช็ก CI ทุกชั่วโมง แล้วส่งเข้า Telegram/Discord
  • Standing orders — คำสั่งประจำที่ agent ยึดถือตลอด
  • Background tasks — งานเบื้องหลัง ดูได้ด้วย /tasks ในแชต
  • Hooks — ผูก action กับ event ของระบบ
⏰
ตัวอย่าง: cron ตอน 9 โมงเช้า → agent ดึง issue ที่ค้างใน tracker มาสรุป แล้วส่งเข้าห้อง Discord #standup ให้คุณอ่านก่อนเริ่มงาน

09 ดูแลระบบระยะยาว

คำสั่ง CLI ที่ควรรู้ไว้ดูแลความสะอาดและความปลอดภัย

bash — operations
$ openclaw sessions                 # ดู session ทั้งหมด (--active 120 / --all-agents / --json)
$ openclaw sessions cleanup         # เคลียร์ตาม disk budget
$ openclaw memory status            # สถานะ memory index
$ openclaw memory search --query "deploy notes"
$ openclaw security audit            # ตรวจช่องโหว่การตั้งค่า (เช่น allowlist หละหลวม)
$ openclaw backup create             # สำรอง state (--only-config / --no-include-workspace)
$ openclaw backup verify <path>      # ตรวจ manifest
🛡️
รัน openclaw security audit เป็นระยะ โดยเฉพาะหลังเพิ่ม channel/skill ใหม่ — มันจะเตือนเรื่อง allowlist แบบชื่อ/tag (แทน ID) และการตั้งค่าเสี่ยงอื่น ๆ · และอย่าลืม openclaw update ให้ทันรุ่นล่าสุดเสมอ
