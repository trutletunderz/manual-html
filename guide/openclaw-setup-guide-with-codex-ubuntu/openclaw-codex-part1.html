01 สถาปัตยกรรม 4 ชั้น ที่ต้องแยกให้ออก

เอกสารทางการเตือนเองว่า "Provider, Model, Runtime, Channel เป็นคนละชั้นกัน" — คนตั้งค่าพังเพราะสับสน 4 คำนี้บ่อยที่สุด

CHANNEL คุณคุยผ่านอะไร — Telegram / Discord / CLI · เป็นแค่ "ปาก-หู" ไม่เกี่ยวกับสมอง
PROVIDER เจ้าของโมเดล — openai, anthropic, google · ใช้คุมเรื่อง auth/บัญชี
MODEL โมเดลตัวไหน — เขียนเป็น provider/model เช่น openai/gpt-5.5
RUNTIME รันลูป agent ด้วยอะไร — codex harness / claude-cli / built-in openclaw

ประเด็นสำคัญสำหรับคู่มือนี้: เราจะเลือก Model = openai/gpt-5.5 แล้วบอกให้ Runtime = Codex harness เป็นคนรันลูป งานนี้ OpenClaw ตั้งให้อัตโนมัติอยู่แล้ว — ถ้าใช้ OpenAI model มันจะเลือก Codex runtime ให้เองโดยไม่ต้องตั้งอะไรเพิ่ม (เดี๋ยวลงรายละเอียดในข้อ 08)

💡
คำว่า "harness" ที่จะเจอบ่อย = ตัว implement ของ runtime · "Codex harness" ก็คือโค้ดที่ทำให้ codex runtime ทำงานได้นั่นเอง

02 ข้อกำหนดระบบ

เช็กให้ครบก่อน จะได้ไม่ติดกลางทาง

รายการที่ต้องมีหมายเหตุ
OSUbuntu 22.04 / 24.04 (หรือ WSL2)คู่มือนี้อิง Ubuntu โดยตรง
RuntimeNode 24 (แนะนำ) หรือ 22.19+installer script ลงให้อัตโนมัติ
สิทธิ์ผู้ใช้ที่ sudo ได้ไม่จำเป็นต้องเป็น root
บัญชี CodexChatGPT/Codex subscription หรือ OpenAI API keyเลือกอย่างใดอย่างหนึ่ง (ข้อ 09)
เครือข่ายออกเน็ตได้ (outbound)ไม่ต้องเปิด public IP / webhook
🖥️
ถ้าจะให้ agent ออนไลน์ 24 ชม. แนะนำลงบน VPS เล็ก ๆ (เช่น Vultr/DigitalOcean) แทนเครื่องตั้งโต๊ะที่ปิด ๆ เปิด ๆ — ขั้นตอนติดตั้งเหมือนกันทุกประการ

03 เตรียม Ubuntu + ติดตั้ง Node 24

ถ้าใช้ installer script (ข้อ 04 วิธี A) ขั้นนี้ "ข้ามได้" เพราะ script ลง Node ให้เอง — แต่ถ้าอยากคุม Node เองหรือจะลงด้วย npm ให้ทำตามนี้

3.1 อัปเดตระบบ + ลงเครื่องมือพื้นฐาน

bash — เตรียมระบบ
$ sudo apt update && sudo apt upgrade -y
$ sudo apt install -y curl git build-essential
  • apt update ดึงรายการแพ็กเกจล่าสุด · apt upgrade -y อัปเกรดของเดิม (-y = ตอบ yes อัตโนมัติ)
  • curl ไว้ดึง installer · git ไว้กรณีลงจาก source · build-essential ให้ native module ของ Node คอมไพล์ได้

3.2 ติดตั้ง Node 24 ผ่าน NodeSource

วิธีนี้ได้ Node เวอร์ชันใหม่กว่าที่มากับ Ubuntu (ของ apt มักเก่าเกินไป)

bash — ติดตั้ง Node 24
$ curl -fsSL https://deb.nodesource.com/setup_24.x | sudo -E bash -
$ sudo apt install -y nodejs
  • -fsSL = fail เงียบ ๆ ถ้า error (f), เงียบ (s), โชว์ error จริง (S), ตาม redirect (L)
  • สคริปต์ setup_24.x เพิ่ม repo ของ Node 24 เข้า apt · sudo -E ส่งต่อ environment เดิมเข้าไปด้วย

3.3 ยืนยันเวอร์ชัน

bash — verify
$ node -v   # ควรขึ้น v24.x.x
$ npm -v    # npm มาพร้อม Node
⚠️
กับดักที่เจอบ่อย: ถ้าเคยลง Node เวอร์ชันเก่าจาก apt install nodejs มาก่อน อาจชนกัน ให้ sudo apt remove nodejs npm ก่อน แล้วค่อยลงใหม่ตาม 3.2

04 ติดตั้ง OpenClaw

มี 2 วิธีหลัก เลือกอย่างใดอย่างหนึ่ง — มือใหม่แนะนำวิธี A

วิธี A — Installer Script (แนะนำ)

วิธีเร็วสุด สคริปต์จะ detect OS, ลง Node ให้ถ้ายังไม่มี, ลง OpenClaw แล้วเปิด onboarding ต่อให้เลย

bash — one-liner
$ curl -fsSL https://openclaw.ai/install.sh | bash

ถ้าอยากลงเฉย ๆ ยังไม่ให้เปิด onboarding (จะตั้งค่าทีหลังเอง) ใส่ flag นี้:

bash — ลงแต่ยังไม่ onboard
$ curl -fsSL https://openclaw.ai/install.sh | bash -s -- --no-onboard

วิธี B — npm (เมื่อคุณคุม Node เอง)

ถ้าทำข้อ 3 ลง Node 24 ไว้แล้ว ใช้วิธีนี้ตรง ๆ ได้:

bash — npm global
$ npm install -g openclaw@latest
$ openclaw onboard --install-daemon
  • -g = ลงแบบ global (เรียก openclaw ได้จากทุกที่)
  • @latest = บังคับเอาเวอร์ชันล่าสุดเสมอ ✅ ตรงตามที่ต้องการ
  • --install-daemon = สั่งให้ติดตั้ง systemd user service เพื่อให้ Gateway รันค้างเอง (รายละเอียดข้อ 05)
🔧
เจอ "openclaw: command not found"? มักเป็นเพราะ path ของ npm global ไม่อยู่ใน $PATH — แก้ด้วยการเพิ่มบรรทัดนี้ลงท้าย ~/.bashrc แล้วเปิด terminal ใหม่:
~/.bashrc
export PATH="$(npm prefix -g)/bin:$PATH"

05 Onboarding + ตั้ง Daemon ให้รันค้าง

ขั้นนี้คือการ "ตั้งบ้าน" ให้ OpenClaw — สร้าง config และลงตัวรันเบื้องหลัง

คำสั่ง onboarding จะถามตอบเป็นขั้น ๆ (เลือก provider, สร้าง config ที่ ~/.openclaw/):

bash — onboarding
$ openclaw onboard --install-daemon

เกิดอะไรขึ้นบ้างหลังรันคำสั่งนี้

  • สร้างโฟลเดอร์สถานะทั้งหมดที่ ~/.openclaw/ — เก็บ config, credentials, workspace, sessions
  • ไฟล์ตั้งค่าหลักคือ ~/.openclaw/openclaw.json (เราจะแก้ไฟล์นี้บ่อยใน Part ถัด ๆ ไป)
  • ลง systemd user service ให้ Gateway รันค้างและ กลับมาเองหลัง reboot
🔌
สำคัญมากบน VPS / server: systemd "user service" ปกติจะหยุดเมื่อคุณ logout ออกจาก SSH ให้เปิด "linger" เพื่อให้มันรันต่อแม้ไม่มีใคร login:
bash — ให้รันต่อหลัง logout
$ loginctl enable-linger $USER

06 ตรวจสอบการติดตั้ง

3 คำสั่งทองที่ควรจำขึ้นใจ — ใช้เช็กสุขภาพระบบทุกครั้งที่สงสัย

bash — health check
$ openclaw --version        # ยืนยันว่า CLI ใช้ได้ + เวอร์ชันอะไร
$ openclaw doctor           # สแกนหาปัญหา config / สิ่งที่ต้องแก้
$ openclaw gateway status   # Gateway รันอยู่จริงไหม
คำสั่งดูอะไร
openclaw --versionเห็นเลขเวอร์ชัน = CLI ติดตั้งสำเร็จ ควรเป็นสาย 2026.6.x ขึ้นไป
openclaw doctorตัวช่วยวินิจฉัย ถ้าเจอ legacy config มันจะบอก · มี --fix ให้แก้อัตโนมัติ (ใช้บ่อยในข้อ 10)
openclaw gateway statusบอกว่า Gateway ทำงานหรือไม่ + service name ที่ระบบใช้
🩺
ติดอะไรเมื่อไหร่ ให้รัน openclaw doctor เป็นอย่างแรกเสมอ — มันมักบอกตรง ๆ ว่าพังตรงไหนและสั่งแก้ยังไง

07 จัดการ Gateway (หัวใจของระบบ)

Gateway = โปรเซสที่ต้องรันค้างไว้ ถ้ามันตาย = agent ไม่ตอบ ทุก channel เงียบหมด

คำสั่งจัดการผ่าน OpenClaw เอง (แนะนำ)

bash — gateway control
$ openclaw gateway           # รัน Gateway แบบ foreground (เห็น log สด ๆ — ดีตอน debug)
$ openclaw gateway status    # เช็กสถานะ
$ openclaw gateway restart   # รีสตาร์ท (ใช้ทุกครั้งหลังแก้ config)
$ openclaw gateway install   # ลง daemon ภายหลัง (ถ้าตอน onboard ไม่ได้ลง)

ดู log

bash — logs
$ openclaw logs --follow     # ตาม log แบบ real-time (Ctrl+C เพื่อออก)

หรือคุมผ่าน systemd โดยตรง

เพราะ daemon ลงเป็น systemd user service จึงต้องมี --user เสมอ (ชื่อ service ดูได้จาก openclaw gateway status):

bash — systemctl --user
$ systemctl --user status openclaw
$ systemctl --user restart openclaw
$ journalctl --user -u openclaw -f   # ดู log ผ่าน journald
🔁
กฎเหล็ก: แก้ openclaw.json เสร็จ ต้อง restart Gateway เสมอ ไม่งั้น config ใหม่ไม่ถูกโหลด — อาการคือ "ตั้งค่าแล้วแต่ไม่มีอะไรเปลี่ยน"

08 เชื่อม Codex: เข้าใจก่อนตั้งค่า

ตรงนี้คือหัวใจของคู่มือ — อ่านช้า ๆ จะตั้งค่าได้ถูกตั้งแต่ครั้งแรก

OpenClaw ใช้ provider id เดียวคือ openai สำหรับ "ทุกแบบของ OpenAI" ไม่ว่าจะ login แบบ ChatGPT/Codex subscription หรือใช้ API key ตรง ๆ แล้วแยกการตัดสินใจออกเป็นชั้น ๆ:

  • Model ref ที่ใช้ คือ openai/gpt-5.5 (เขียนแบบนี้เสมอ ทั้งกรณี subscription และ API key)
  • เมื่อ agent รันด้วย OpenAI model มันจะ เลือก Codex runtime ให้อัตโนมัติ (ถ้าไม่ตั้ง runtime override) — นี่คือ "native Codex app-server harness"
  • Codex harness ต้องมี bundled Codex plugin ซึ่ง OpenClaw เปิดให้เองเมื่อเจอ ref ที่ขึ้นต้นด้วย openai/*
🧭
เลือก auth ยังไง?
• มี ChatGPT/Codex subscription อยู่แล้ว → login แบบ OAuth (จ่ายเหมารายเดือน ไม่คิดตาม token) — ข้อ 09 วิธี 1
• อยากจ่ายตามการใช้งานจริง → ใช้ OpenAI API key — ข้อ 09 วิธี 2
🗂️
เลิกใช้ของเก่า: ถ้าเคยเห็น config เขียน codex/<model> แบบเก่า — มันคือ legacy แล้ว · รัน openclaw doctor --fix เพื่อแปลงเป็น openai/* ให้อัตโนมัติ

09 Login เข้า Codex

เลือกวิธีเดียวตามชนิดบัญชีของคุณ

วิธี 1 — ChatGPT / Codex subscription (OAuth)

ทำตอน onboard ได้เลย หรือสั่งทีหลังก็ได้:

bash — Codex OAuth login
# แบบทำตอน onboard:
$ openclaw onboard --auth-choice openai-codex

# หรือ login เพิ่มทีหลัง:
$ openclaw models auth login --provider openai-codex
  • จะมีลิงก์ให้เปิด browser แล้ว sign in ด้วยบัญชี ChatGPT — เหมือนตอน login Codex CLI ทั่วไป
  • ถ้าเครื่องนี้เคย codex login (Codex CLI) ไว้แล้ว OpenClaw จะ reuse บัญชีเดิมได้เลย

วิธี 2 — OpenAI API key (จ่ายตามใช้)

ตั้ง environment variable ให้ Gateway เห็น key (วางใน ~/.bashrc หรือไฟล์ env ของ service):

bash — API key
$ export OPENAI_API_KEY="sk-..."
🔐
อย่าวาง API key / token ลงในไฟล์ที่ agent อ่านได้ เช่น workspace file — เก็บไว้ใน env หรือ credential store เท่านั้น และถ้าหลุดให้ revoke + หมุน key ใหม่ทันที

10 ตั้งค่าโมเดลให้เป็น Codex (GPT-5.5)

บอก OpenClaw ว่า agent หลักให้ใช้ openai/gpt-5.5

วิธีเร็ว — สั่งผ่าน CLI

bash — set default model
$ openclaw config set agents.defaults.model '"openai/gpt-5.5"' --json
$ openclaw gateway restart

แค่นี้ก็พอ — เพราะ OpenAI model จะเลือก Codex runtime ให้เองโดยอัตโนมัติ

วิธีชัดเจน — เขียนใน openclaw.json เอง

เปิดไฟล์ด้วย editor (เช่น nano ~/.openclaw/openclaw.json) แล้วใส่โครงนี้ ถ้าอยาก "ปักหมุด" runtime ให้ชัด:

~/.openclaw/openclaw.json (JSON5)
{
  models: {
    providers: {
      openai: {
        agentRuntime: { id: "codex" }   // บังคับใช้ Codex harness
      }
    }
  },
  agents: {
    defaults: {
      model: "openai/gpt-5.5"            // สมองหลักของ agent
    }
  }
}
  • agents.defaults.model = โมเดลเริ่มต้นของทุก agent ที่ไม่ได้ระบุเอง
  • models.providers.openai.agentRuntime.id: "codex" = สั่งชัด ๆ ว่า turn ของ OpenAI ให้รันผ่าน Codex harness
  • ถ้าระบุ "codex" ตรง ๆ แล้ว harness ใช้ไม่ได้ มันจะ fail closed (ไม่แอบสลับไปตัวอื่น) — เห็น error ชัดดีกว่าเงียบ ๆ
🚫
เจอ "Model provider/model is not allowed"? แปลว่าโมเดลยังไม่อยู่ใน allowlist · เพิ่มด้วย:
bash
$ openclaw config set agents.defaults.models '{"openai/gpt-5.5":{}}' --strict-json --merge

11 ทดสอบว่า Codex ทำงาน

ก่อนต่อ channel จริง ลองคุยกับ agent ผ่าน CLI ให้ชัวร์ว่าสมองเชื่อมแล้ว

เช็กรายชื่อโมเดล + สถานะ

bash — models
$ openclaw models status     # ดูว่าตอนนี้ใช้ provider/model/runtime อะไร
$ openclaw models openai     # ลิสต์โมเดลของ openai ที่เลือกได้

คุยกับ agent ผ่าน CLI

ถ้าเห็นมันตอบกลับมาเป็นข้อความ = Gateway + Codex + agent ครบลูปแล้ว 🎉

$ openclaw chat
# แล้วพิมพ์: สวัสดี ช่วยบอกหน่อยว่าตอนนี้ใช้โมเดลอะไร

12 อัปเดต & ช่องทางเวอร์ชัน

เน้นใช้เวอร์ชันล่าสุด → จำคำสั่งชุดนี้ไว้

bash — update
$ openclaw update                  # วิธีเร็วสุด: detect ชนิดติดตั้ง, ดึงล่าสุด, รัน doctor, รีสตาร์ท Gateway
$ openclaw update --dry-run        # ดูว่ามันจะทำอะไรบ้าง โดยยังไม่ลงมือจริง
$ openclaw update status --json    # เช็ก channel + เวอร์ชันที่มี
Channelคำสั่งเหมาะกับ
stableopenclaw update --channel stableใช้งานจริง — นิ่งที่สุด (ค่าเริ่มต้น)
betaopenclaw update --channel betaอยากลองของใหม่ก่อน แต่ยังพอเสถียร
devopenclaw update --channel devสาย bleeding-edge: สลับเป็น git checkout ที่ตาม main
💾
การ update ไม่ยุ่งกับ config, credentials, workspace ของคุณใน ~/.openclaw — เปลี่ยนแค่ตัวโค้ด OpenClaw เท่านั้น
