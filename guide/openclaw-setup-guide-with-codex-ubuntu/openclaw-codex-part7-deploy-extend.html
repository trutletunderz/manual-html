00 Docker — ใช้เมื่อไหร่ดี (และไม่ดี)

Docker เป็น "ทางเลือก" ไม่ใช่ทางบังคับ · เข้าใจว่าเหมาะกับเคสไหน

✅
ใช้เมื่อ
ต้องการแยก/ใช้ซ้ำได้

อยากได้ gateway แยกโดด ทิ้งง่าย, รันบน host ที่ไม่อยากลงของในเครื่อง, deploy ซ้ำได้

❌
ไม่ต้อง
dev loop เร็ว ๆ

รันบนเครื่องตัวเองพัฒนาเร็ว ๆ → ใช้ install ปกติ (Part 1) ดีกว่า

📦
หมายเหตุ
sandbox ≠ ต้อง Docker ทั้งตัว

tool sandbox ใช้ Docker เป็น backend ได้โดยไม่ต้องเอา gateway ทั้งตัวลง container

💾
ต้องการ RAM ≥ 2 GB ตอน build image — host 1 GB อาจโดน OOM-kill (exit 137) ตอน pnpm install · ถ้า build บน VPS เล็ก ให้ใช้ pre-built image แทน

01 รัน Gateway ใน Docker

วิธีง่ายสุดคือใช้ setup script · ต้องมี Docker + Docker Compose v2

วิธีเร็ว — setup script + pre-built image

bash — จาก repo root
# ใช้ image สำเร็จรูป (ไม่ต้อง build เอง เหมาะกับ VPS เล็ก)
$ export OPENCLAW_IMAGE="ghcr.io/openclaw/openclaw:latest"
$ ./scripts/docker/setup.sh

script จะ: ถาม API key → สร้าง gateway token เขียนลง .env → สร้างโฟลเดอร์ key → start gateway ผ่าน Compose → ทำ onboarding ให้

เปิด Control UI + ต่อ channel

bash — หลัง gateway ขึ้น
# เปิด http://127.0.0.1:18789/ แล้ววาง token จาก .env ใน Settings
$ docker compose run --rm openclaw-cli dashboard --no-open   # ขอลิงก์ใหม่

# ต่อ channel ผ่าน CLI container
$ docker compose run --rm openclaw-cli channels add --channel telegram --token "<token>"
$ docker compose run --rm openclaw-cli channels add --channel discord --token "<token>"
🔗
ทำไมมี 2 container (gateway + cli): openclaw-cli ใช้ network_mode: "service:openclaw-gateway" เพื่อเรียก gateway ผ่าน 127.0.0.1 ได้ · มันเป็น post-start tool — คำสั่ง onboarding ก่อน gateway ขึ้น ต้องรันผ่าน openclaw-gateway ด้วย --no-deps --entrypoint node
🔑
Codex OAuth บน Docker headless: wizard จะเปิด URL ให้ — ใน Docker/headless ให้ copy redirect URL ที่ปลายทาง มาวางกลับใน wizard เพื่อจบ auth (เพราะไม่มี browser ใน container)

02 Persistence + เน็ตเวิร์ก (จุดพลาดบ่อย)

container ถูกแทนที่ได้ — ข้อมูลต้องอยู่นอก container

อะไรถูก mount ออกมา (รอด container replacement)

path ในเครื่องmap ไปที่ containerเก็บอะไร
OPENCLAW_CONFIG_DIR/home/node/.openclawconfig, auth-profiles, .env
OPENCLAW_WORKSPACE_DIR.../workspaceworkspace ของ agent
OPENCLAW_AUTH_PROFILE_SECRET_DIR/home/node/.config/openclawkey เข้ารหัส auth (เก็บแยกจาก config)
👤
EACCES / permission error: image รันเป็น user node (uid 1000) · ถ้าเจอ error สิทธิ์บน bind mount ให้ sudo chown -R 1000:1000 /path/to/config /path/to/workspace · plugin ที่ owner ไม่ตรงจะถูกบล็อก ("suspicious ownership")

เน็ตเวิร์ก 2 กับดักสำคัญ

  • bind LAN vs loopback: setup script ตั้ง OPENCLAW_GATEWAY_BIND=lan เพื่อให้ host เปิด 127.0.0.1:18789 ได้ผ่าน port publishing · ใช้ค่า bind mode (lan/loopback) ไม่ใช่ 0.0.0.0
  • provider บน host: ใน container 127.0.0.1 = ตัว container เอง! ถ้า Ollama/LM Studio รันบน host ต้องใช้ http://host.docker.internal:11434 และให้ service bind 0.0.0.0
🔥
VPS + Docker = ระวัง firewall: port ที่ publish ผ่าน Docker วิ่งผ่าน DOCKER-USER chain ไม่ใช่ INPUT ปกติ · ต้องตั้ง rule ใน DOCKER-USER (ดู Part 6 + เอกสาร security) ไม่งั้น UFW อาจไม่กันอย่างที่คิด

03 Health Check + Observability

รู้ว่า gateway สุขภาพดีไหม + ส่ง metric/trace ออกได้

bash — health probes (ไม่ต้อง auth)
$ curl -fsS http://127.0.0.1:18789/healthz   # liveness
$ curl -fsS http://127.0.0.1:18789/readyz    # readiness

image มี HEALTHCHECK ในตัว ping /healthz — ถ้า fail ซ้ำ Docker จะ mark unhealthy แล้ว orchestrator restart ให้

Observability (OpenTelemetry + Prometheus)

  • OTel: ติดตั้ง @openclaw/diagnostics-otel → ตั้ง OTEL_EXPORTER_OTLP_ENDPOINT → ส่ง trace/metric ออกแบบ outbound (ไม่ต้อง publish port)
  • Prometheus: ติดตั้ง diagnostics-prometheus → scrape ที่ /api/diagnostics/prometheus (มี gateway auth ป้องกัน — อย่าเปิด /metrics public)
🐚
อยากจัดการ Docker ง่ายขึ้น ลง ClawDock helper → ได้คำสั่ง clawdock-start, clawdock-dashboard ฯลฯ
PART B · MULTI-GATEWAY

04 แยกหลาย Gateway ตาม Trust Boundary

เมื่อ 1 gateway ไม่พอ — เพราะมีหลายระดับความเชื่อใจที่ต้องแยกขาด

จาก Part 6: OpenClaw ไม่ใช่ ระบบ multi-tenant — 1 gateway = 1 trust boundary · ดังนั้นถ้าคุณมีงานหลายระดับความเสี่ยง วิธีที่ถูกต้องคือ แยก gateway คนละตัว (ในอุดมคติคนละ OS user / host) ไม่ใช่ยัดทุกอย่างใน gateway เดียว

เครื่อง/VPS A — ส่วนตัว (เต็มสิทธิ์) └─ gateway #1 · agent: personal (sandbox off, exec full) └─ Telegram DM ส่วนตัว · ห้อง Codex โปรเจกต์ เครื่อง/VPS B — งานสาธารณะ/ทีม (จำกัด) └─ gateway #2 · agent: public (sandbox all, no fs/exec) └─ Discord server ทีม · tool policy เข้ม

เหตุผลที่ต้องแยก (ไม่ใช่แค่ "เผื่อ")

  • credential แยกกัน — gateway สาธารณะไม่มี API key/บัญชีส่วนตัว ถ้าโดน injection ก็เผาไม่ถึงของส่วนตัว
  • disk = trust boundary — ใครอ่าน ~/.openclaw ได้ = เห็น transcript/secret ทั้งหมด · แยก OS user กันการอ่านข้ามกัน
  • tool authority แยก — ถ้าหลายคนคุย agent ตัวเดียว = ทุกคนสั่ง tool ชุดเดียวกันได้ · แยก gateway = แยกอำนาจ
🔀
บนเครื่องเดียวก็แยกได้ด้วยการรันหลาย gateway คนละ port/config dir (ดู Multiple gateways) แต่ คนละ OS user/host แข็งแรงกว่า สำหรับความเชื่อใจที่ต่างกันจริง ๆ
📡
อยากคุมหลาย gateway จากที่เดียว → ใช้ remote gateway + Tailscale (gateway ยัง loopback, Tailscale คุม access) ตามแนวทาง Part 6
PART C · สร้าง SKILL / PLUGIN เอง

05 สร้าง Skill เอง (ไม่ต้องเขียนโค้ดหนัก)

Skill = ความสามารถที่เขียนเป็นไฟล์ SKILL.md + ทรัพยากรประกอบ — เบากว่า plugin

Skill เหมาะเมื่ออยากเพิ่ม "วิธีทำงานเฉพาะ" ให้ agent โดยไม่ต้องเขียน TypeScript · มันโหลดผ่าน skills watcher และ skill ที่ตั้ง user-invocable จะกลายเป็น slash command อัตโนมัติ

โครงสร้าง skill
my-skill/
├── SKILL.md          # หัวใจ: คำอธิบาย + วิธีใช้ + เมื่อไหร่ใช้
├── scripts/          # (ออปชัน) สคริปต์ที่ skill เรียก
└── resources/        # (ออปชัน) template/ข้อมูลประกอบ
SKILL.md — โครงตัวอย่าง
---
name: pedpro-spec-check
description: ตรวจสเปก Bruno ว่าตรงกับฟอร์มที่จะสร้าง
user-invocable: true
---

# Pedpro Spec Check

เมื่อผู้ใช้ขอสร้าง/แก้ฟอร์ม ให้:
1. อ่านไฟล์ .bru ที่เกี่ยวข้องใน bruno-api/
2. เทียบ field กับสเปก — ห้ามเดา field เอง
3. รายงานส่วนที่ไม่ตรงเป็น checklist
bash — ติดตั้ง + เรียกใช้
$ openclaw plugins install ./my-skill   # หรือ clawhub:<pkg> ถ้าเผยแพร่แล้ว
$ openclaw gateway restart
# ในแชต: /pedpro-spec-check
⚡
ทำไม skill บาง command ทำงานเร็ว+แน่นอน: ถ้า skill ประกาศ command-dispatch: tool มันจะรันตรงเข้า tool โดยไม่ผ่านโมเดล — deterministic เหมือน slash command · skill ที่ไม่ประกาศจะถูกตีความโดย agent
🔐
skill = trusted code: การแก้ SKILL.md อัปเดต snapshot ใน turn ถัดไป (skills watcher) · จำกัดคนที่แก้โฟลเดอร์ skill ได้ — มันคือโค้ดที่ agent จะทำตาม

06 สร้าง Plugin (tool) ด้วย SDK

เมื่ออยากให้ agent มี tool ใหม่ที่เขียน logic เอง (TypeScript/ESM) · ไม่ต้อง fork repo OpenClaw

ต้องการ: Node ≥ 22, รู้ TypeScript (ESM) · เผยแพร่ผ่าน ClawHub/npm แล้วผู้ใช้ openclaw plugins install ได้เลย

2 ไฟล์ที่ต้องมี

package.json
{
  "name": "@myorg/openclaw-my-plugin",
  "version": "1.0.0",
  "type": "module",
  "openclaw": { "extensions": ["./index.ts"] }
}
index.ts — register tool
import { definePluginEntry } from "openclaw/plugin-sdk/plugin-entry";
import { Type } from "@sinclair/typebox";

export default definePluginEntry({
  id: "my-plugin",
  name: "My Plugin",
  description: "Adds a custom tool",
  register(api) {
    api.registerTool({
      name: "my_tool",
      description: "Do a thing",
      parameters: Type.Object({ input: Type.String() }),
      async execute(_id, params) {
        return { content: [{ type: "text", text: `Got: ${params.input}` }] };
      },
    });
  },
});

ทดสอบ + ตรวจว่าโหลดจริง

bash
$ openclaw plugins install --link ./my-plugin   # dev: link local
$ openclaw gateway restart
$ openclaw plugins inspect my-plugin --runtime --json  # พิสูจน์ว่า runtime register tool จริง
อยากเพิ่มเรียก API
tool ให้ agent เรียกapi.registerTool(...)
model provider ใหม่api.registerProvider(...)
channel ใหม่api.registerChannel(...) (defineChannelPluginEntry)
slash commandapi.registerCommand(...)
event hookapi.registerHook(...)
HTTP route / CLIapi.registerHttpRoute / registerCli
🎚️
required vs optional tool: tool ปกติ "ใช้ได้เสมอ" · ตั้ง { optional: true } สำหรับ tool ที่มี side effect/ต้องลง binary เพิ่ม — ผู้ใช้ต้อง opt-in ผ่าน tools.allow: ["my_tool"] · เหตุผล: กัน tool อันตรายเปิดเองโดยไม่ตั้งใจ
📦
import ให้ถูก: import จาก subpath เจาะจง openclaw/plugin-sdk/<subpath> เสมอ · อย่า import จาก root openclaw/plugin-sdk (deprecated) · และอย่า import plugin ตัวเองผ่าน SDK path (ใช้ local barrel)

07 Plugin Policy + Slots

คุมว่า plugin ตัวไหนโหลด/ไม่โหลด — และใครคุมหมวดที่มีได้ตัวเดียว

~/.openclaw/openclaw.json — plugin policy
{ plugins: {
  enabled: true,
  allow: ["voice-call", "my-plugin"],   // allowlist เด็ดขาด (นอกลิสต์ไม่โหลด)
  deny: ["untrusted-plugin"],            // deny ชนะ allow เสมอ
  slots: { memory: "memory-core" },     // เลือก 1 ตัวสำหรับหมวดผูกขาด
  entries: {
    "my-plugin": { enabled: true, config: { /* ... */ } }
  }
} }
  • plugins.deny ชนะ allow และ per-plugin enable เสมอ — ใช้ปิดเด็ดขาด
  • plugins.allow เป็น allowlist เด็ดขาด — tool ของ plugin นอกลิสต์ใช้ไม่ได้ แม้ tools.allow: ["*"]
  • plugins.slots.<slot> เลือก 1 ตัวสำหรับหมวดผูกขาด (เช่น memory, context engine) — slot จะ force-enable ตัวที่เลือก
  • plugin จาก workspace ปิดโดยปริยาย — ต้อง allow/enable ก่อน (เหตุผลความปลอดภัย: กันโค้ดใน workspace รันเอง)
🔄
install/update/uninstall plugin ต้อง restart gateway · gateway ที่ managed จะ restart เองเมื่อ install record เปลี่ยน · ตรวจด้วย openclaw plugins inspect <id> --runtime --json (plugins list อ่านแค่ config เย็น ไม่พิสูจน์ว่า runtime โหลดจริง)
⚠️
plugin รัน in-process กับ gateway = trusted code: ลงเฉพาะที่ไว้ใจ · pin เวอร์ชันเป๊ะ (@scope/pkg@1.2.3) · ตรวจโค้ดบนดิสก์ก่อนเปิด · ใช้ security.installPolicy ถ้าต้องคุมการติดตั้งระดับ host
